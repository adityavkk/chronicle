<Callout icon="alert" label="A design, not shipped code">
  Everything else on this site documents code that runs. This page documents a
  *plan*. The single-slot control plane, the recovery sweep, and the generation
  fence below are built and tested; **slot-homing, the due-set, and leased slot
  ownership are proposed** — Option A from the [horizontal-scale
  research](https://github.com/adityavkk/chronicle/blob/main/docs/specs/horizontal-scale/research/05-proposed-architecture.md).
  The load-bearing numbers are not yet measured; §10 says which, and why they
  gate the build.
</Callout>

<Tldr>
  <ul>
    <li><strong>The control plane is pinned to one Redis slot.</strong> Every subscription key shares the fixed <code>{"{__ds}"}</code> hash tag, so all of it lands on one cluster slot. State capacity stops scaling there, no matter how many replicas serve traffic.</li>
    <li><strong>Delivery is event-driven; the sweep only backstops crashes.</strong> A webhook fires the instant a matching append lands (in-process, at append latency) — not on a timer. The ~2s recovery sweep delivers nothing in steady state; it exists to re-derive a wake a crash dropped. But it runs on every replica over every subscription, so its redundant <code>O(N·K)</code> "ask" — not the wakes — is the cost. Adding replicas multiplies that ask; it does not divide it.</li>
    <li><strong>Option A keeps one binary and shards two things.</strong> Slot-home each subscription's keys under <code>{"{__ds:h}"}</code> with <code>h = hash(id) % S</code> (state), and shard the background work by a leased per-slot owner (work). Capacity scales with S; total sweep work drops to <code>O(owed)</code>.</li>
    <li><strong>The fence is safety, not liveness.</strong> The generation fence makes a duplicate wake harmless; it does nothing for a <em>coverage gap</em> when a slot is briefly owned by nobody. So the full sweep stays — demoted to a backstop that covers every slot, never deleted.</li>
    <li><strong>Don't split the binary yet.</strong> Splitting serving from the control plane is Option B, deferred until measurements force it. The one binary held at K=10k; the fork is a later question, not a starting one.</li>
  </ul>
</Tldr>

<section class="block" id="ceiling">
  <SectionHead num="01" label="The ceiling" title="Where one binary <em>stops scaling</em>" />

  Chronicle is one Go binary. It serves the durable-streams API — append, read,
  long-poll, SSE — and it runs the subscription control plane: the recovery
  sweep, the lease worker, and the retry worker that together keep a durable
  cursor delivered.[^subs] Put more replicas behind the load balancer and the
  serving path scales: each replica handles its share of appends and reads
  against the per-stream log, which is already sharded across the cluster by each
  stream's own hash tag.

  The control plane does not scale the same way, for two separate reasons.

  First, **all of it lives in one slot.** Every subscription key — the config
  hash, the cursor hash, the lease and retry schedules, the fan-out index —
  shares the fixed `{__ds}` hash tag,[^slot] so Redis Cluster routes every one to
  the same slot on the same node. That co-location is deliberate: it lets the
  multi-key Lua scripts (`ack.lua` touches 4 keys, `delete_sub.lua` 5) run as
  single-slot atomic scripts. But it also means subscription *state capacity* —
  memory, and the write throughput of one primary — is fixed at one node's worth,
  and no number of serving replicas moves it.

  Second, **every replica sweeps everything.** The recovery sweep is the
  durability backstop: every ~2s it walks all K subscriptions, re-derives pending
  work from each durable cursor, and re-fires any wake a crash dropped. Correct,
  and necessary. But every replica runs the same sweep over the same K
  subscriptions against the same slot, so the cost is `O(N·K)` for N replicas —
  redundant work that grows with the cluster instead of being divided by it.

  <Diagram
    name="seq-hs-ceiling"
    label="The O(N·K) sweep"
    caption="Three replicas, each sweeping all K subscriptions every ~2s against the single {__ds} slot. The reads are redundant — every replica re-derives the same owed work — and they all converge on one node. Scaling out the fleet scales the load on that one slot up, not down."
  />

  Batching the sweep's reads pushed this cliff out — pipelining the per-tick
  reads moved it from a few thousand subscriptions to past 10k on one
  replica[^batch] — but it moved the cliff, it did not remove it. The two limits
  above are structural: one slot for state, `O(N·K)` for work. Option A is the
  design that addresses both while keeping the single binary.
</section>

<section class="block" id="eventflow">
  <SectionHead num="02" label="Event flow" title="The happy path is <em>already event-driven</em>" />

  Before sharding anything, be precise about what the sweep is *for* — because it
  is easy to read §01 and conclude that wakes are delivered by the sweep, in
  ~2-second bursts. They are not. On the happy path a webhook fires the instant a
  matching append lands; the sweep delivers nothing.

  An append is atomic with its wake signal. `append.lua` writes the durable frame,
  advances the tail, and `PUBLISH`es the stream's notify channel in one
  single-slot script, so the byte and the signal commit together or not at all.

  ```lua title="store/redis/scripts/append.lua — the frame and its wake signal commit together"
  redis.call('ZADD', KEYS[2], '0', frame)        -- the message, in the stream's ZSET
  redis.call('HSET', KEYS[1], 'tail', new_tail)  -- the new tail offset
  -- … then, in the same atomic script, signal waiters:
  redis.call('PUBLISH', channel, closing and 'c' or 'a')  -- ds:notify:<path>
  ```

  That `PUBLISH` wakes long-poll and SSE readers. Subscriptions wake through a
  parallel, synchronous hook: after a successful append the HTTP handler calls
  `OnStreamAppend`, which reads the stream's subscribers from the fan-out index and
  arms-and-delivers each idle one immediately. No poll, no timer — the wake rides
  the same request that committed the append.

  ```go title="webhook/manager.go:237 — the append-time hook, in-process on the appending origin"
  func (m *Manager) OnStreamAppend(path string) {
      ids, _ := m.store.StreamSubscribers(path) // the fan-out index for this stream
      for _, id := range ids {
          m.maybeWake(id, path) // idle with pending work? arm the wake, deliver now
      }
  }
  ```

  **A durable outbox already ships — for pull-wake.** A webhook is delivered
  in-process and forgotten; a pull-wake is delivered through a durable record. The
  server appends a `{"type":"wake"}` event to the subscription's `wake_stream` (an
  ordinary durable stream), and a pool of workers tails it and claims. That wake
  event *is* a transactional outbox: a durable entry that says "this subscription
  owes a wake," written on the append, drained by whichever worker reads next, and
  marked sent so it is not re-emitted.

  ```go title="webhook/manager.go:302 — pull-wake writes the durable wake event, then marks it sent"
  if err := m.streams.AppendWakeEvent(sub.Config.WakeStream, data); err != nil {
      // Leave wake_event_sent_ns at 0 so the recovery sweep re-emits it.
      return
  }
  // Record the durable emit, fenced on (generation, wake_id), so the sweep
  // does not re-emit a wake that was already delivered.
  m.store.RecordWakeEventSent(sub.ID, generation, wakeID, time.Now())
  ```

  <Diagram
    name="seq-hs-eventflow"
    label="The event-driven happy path (pull-wake)"
    caption="The append commits the frame and the wake signal in one atomic script. The append-time hook writes a durable wake event to the wake_stream — the outbox — and a worker drains it. The recovery sweep fires nothing on this path; it exists only to re-emit a wake a crash left unsent."
  />

  So where does the ~2s sweep come in? **Only on failure.** The webhook hook runs
  on the appending origin alone, in process. If that origin crashes between the
  durable append and the delivery, the wake is lost and nothing else knows it was
  owed, so the sweep re-derives it from the durable cursor. In steady state, with
  no crashes, the sweep walks the subscriptions, finds every cursor already at its
  tail, and fires nothing — but it still pays `O(K)` every tick to *ask*, on every
  replica. That redundant question, not the wakes, is the `O(N·K)` cost §01 named,
  and removing it is what Move 2's due-set is for.

  <Callout icon="info" label="Sweep, not outbox — for the webhook path, for now">
    The webhook path has no durable outbox today: its recovery is the sweep, a
    deliberate trade of one re-derivation pass for a dedicated delivery queue and
    worker — the cost is recovery at sweep latency for a post-crash event. The
    pull-wake `wake_stream` above is the durable-outbox shape already in the
    codebase, and the [due-set](#move2) (§05) is what carries that shape onto the
    work-sharded webhook path, so the periodic sweep can stop being the only thing
    that finds owed work and shrink toward a pure crash backstop.
  </Callout>
</section>

<section class="block" id="shape">
  <SectionHead num="03" label="The shape" title="Partition, own, <em>no coordinator</em>" />

  Redis Cluster, Kafka, NATS JetStream, Pulsar, and Orleans all reach horizontal
  scale the same way: **partition the keyspace, give each partition exactly one
  owner, and keep no central coordinator in the request path.**[^priorart] The
  two limits from §01 map cleanly onto the two halves of that move, because they
  are two different problems wearing one symptom.

  <Table>
    | Limit | What it caps | The move |
    | --- | --- | --- |
    | One slot holds all state | memory + one primary's write throughput | **partition the state** across S slots |
    | Every replica sweeps all K | redundant `O(N·K)` background work | **partition the work** by a per-slot owner |
  </Table>

  Option A makes three moves on this shape, in order. **Move 1** slot-homes each
  whole subscription into one of `S` virtual slots, so state capacity scales with
  S. **Move 2** shards the schedules with the subscriptions and adds a
  per-subscription *due-set*, so a worker fires owed subscriptions directly
  instead of scanning all K. **Move 3** leases each slot to one replica, so the
  background work for a slot runs on exactly one node. A short glossary, used
  throughout:

  <Table>
    | Symbol | Is |
    | --- | --- |
    | `N` | serving replicas (Chronicle processes) |
    | `K` | subscriptions |
    | `S` | virtual slots — fixed at **256**, independent of N |
    | `h` | one slot index, `h = hash(id) % S` |
    | `F` | subscriptions a single stream wakes |
  </Table>

  The thread that runs through all three moves, and the reason the design is safe
  to build incrementally: **none of the moves is a correctness dependency.** Each
  is an optimization layered over the existing, still-correct full sweep. §07 is
  where that claim is paid for; the three moves come first.
</section>

<section class="block" id="move1">
  <SectionHead num="04" label="Move 1" title="Slot-home <em>whole</em> subscriptions" />

  The single `{__ds}` tag becomes `S` tags of the form `{__ds:h}`, with `h =
  fnv32a(id) % S`. Every key one subscription touches — its config hash, its
  cursor hash, its lease and retry schedules, its due-set, its fan-out membership
  — takes the *same* `h`. The whole subscription homes in one slot; no
  subscription is ever split across two.

  That "whole subscription, one slot" rule is the load-bearing constraint, and it
  exists to keep the atomic scripts atomic. `ack.lua` and `delete_sub.lua` are
  multi-key Lua. Redis refuses a multi-key script whose keys span slots with a
  `CROSSSLOT` error, so the moment one subscription's keys land in different
  slots, those scripts stop running. Co-tagging every key for a subscription
  keeps them byte-for-byte single-slot: sharding becomes "compute the tag," not
  "rewrite the atomicity contract."

  <Callout icon="alert" label="Use your own hash, not Redis CRC16">
    `h` MUST come from an application hash — `hash/fnv` FNV-1a — and MUST NOT be
    Redis's CRC16. The client already CRC16-hashes the `{…}` tag to pick a
    cluster slot; if `h` were also CRC16, two subscriptions you meant for
    different `h` could collide onto one cluster slot, and the tag would stop
    being the stable, application-chosen value the whole scheme depends on.
  </Callout>

  `S` is fixed at **256** — a compile-time constant, not a function of the replica
  count. It is chosen once and never changed in place: because `h` is `hash(id) %
  S`, a different `S` re-tags every key, so changing it is a dual-write migration,
  not a config edit. 256 comfortably exceeds any realistic replica count and is
  the upper bound the fan-out experiment in §10 measures.

  <Table>
    | Key | Type | Holds |
    | --- | --- | --- |
    | `ds:{__ds:h}:sub:<id>` | hash | config + runtime: `generation`, `wake_id`, `phase`, lease state |
    | `ds:{__ds:h}:sub:<id>:links` | hash | the durable cursors — `path → "<type>:<acked_offset>"` |
    | `ds:{__ds:h}:subs` | set | the ids homed in slot h, for the sweep |
    | `ds:{__ds:h}:stream:<path>` | set | which subscriptions a stream wakes, in slot h |
    | `ds:{__ds:h}:sched:lease` | zset | `sub → lease_expiry_ns` — the liveness lease |
    | `ds:{__ds:h}:sched:retry` | zset | `sub → next_attempt_ns` — the webhook backoff |
    | `ds:{__ds:h}:due` | zset | `sub → now_ns` — the outbox of owed wakes (Move 2) |
  </Table>

  State capacity now scales with S across the cluster's nodes, because the
  schedules shard *with* the subscriptions rather than staying one global ZSET.
  The cost lands on one operation. `OnStreamAppend` could read the subscribers of
  a stream with a single `SMEMBERS`; once subscriptions scatter across slots, the
  fan-out membership scatters too, so an append must read up to `S` per-slot sets.
  Issued as one pipeline, that is `S` reads grouped by the nodes they land on —
  about one round trip per node, not `S` serial trips, but a real and measurable
  regression. The mitigation is a per-stream "occupied-slots" bitmap so an append
  probes only the slots that actually hold a subscriber; §10 measures the cost
  with and without it.

    why="Every per-subscription script stays single-slot O(1) hash work plus an O(log L) schedule touch — unchanged, because the keys co-tag. </section>

<section class="block" id="move2">
  <SectionHead num="05" label="Move 2" title="A due-set <em>instead of a scan</em>" />

  Move 1 shards the schedules but does not change how owed work is found: the
  sweep still walks all K. Move 2 removes the scan from the hot path by adding a
  per-slot *due-set* — `ds:{__ds:h}:due`, a sorted set that names exactly the
  subscriptions that owe a wake.

  The due-set is an outbox. When an append arms a wake, the same atomic
  `arm_wake.lua` that bumps the generation also `ZADD`s the subscription into its
  slot's due-set, scored at `now_ns`. When the work is acked, the same atomic
  `ack.lua` that advances the cursor also `ZREM`s it. A per-slot *due-worker*
  then reads what is due and fires it — `O(owed)` work, not `O(K)`. The score is
  `now_ns` and nothing cleverer: the due-set answers "which subscriptions need a
  wake," and the lease schedule already answers "which in-flight wakes have timed
  out," so the due-set is a pure owed-list, not a second deadline queue.

  ```lua title="Proposed: the due ZADD/ZREM fold into scripts that are already atomic"
  -- arm_wake.lua, ARMED branch — add one line (now takes the due ZSET as a key):
  redis.call('ZADD', KEYS[3], tonumber(ARGV[2]), ARGV[1])   -- :due ← now_ns

  -- ack.lua, done branch — add one line beside the existing lease/retry ZREMs:
  redis.call('ZREM', KEYS[n], ARGV[1])                      -- clear the owed mark
  ```

  Because the `ZADD` rides inside the script that already flips the phase, the
  owed-mark and the phase transition land together or not at all. There is no new
  window between them for a crash to fall into.

  <RedisTrace
    script="claim_due.lua"

The same machinery already drains the lease and retry schedules today."
    steps={[

      { cmd: "ZADD", key: "ds:{__ds:h}:due now+visibility", does: "Re-score each claimed member forward, never <code>ZREM</code>.     ]}
  />

  The honesty here is that Move 2 *relocates* a dual write rather than removing
  one. The existing arm-then-emit-then-record sequence already writes twice; the
  due-set adds a `ZADD` on arm and a `ZREM` on ack to the hot path. That is write
  amplification on append and ack, paid for a recovery-latency win: an owed wake
  fires at the next due-worker tick instead of waiting for the next full sweep.
  §10 measures the amplification. And the full sweep does not go away — it demotes
  to a backstop, for the reason §07 makes precise.

  <Complexity

<section class="block" id="move3">
  <SectionHead num="06" label="Move 3" title="Lease <em>one owner</em> per slot" />

  Moves 1 and 2 shard the state and give each slot its own outbox, but every
  replica would still run a due-worker over every slot — the `O(N·K)` redundancy,
  relocated. Move 3 removes it: each slot is *leased* to exactly one replica, and
  a replica runs the due, lease, and retry workers only for the slots it owns.

  Ownership is a Redis-backed membership protocol, separate from Redis Cluster's
  own membership. It has four parts.

  **Membership.** Each replica has a stable `replica_id` for its process lifetime
  — the pod name plus a per-start nonce, so a restarted pod reusing a name is a
  distinct incarnation. Every replica heartbeats by `ZADD`-ing its `replica_id`
  into a `ds:{ownership}:members` ZSET scored at `now + memberLeaseTTL`, and evicts
  expired members with `ZREMRANGEBYSCORE`. This ZSET is the live set: Kubernetes
  decides who gets HTTP traffic, this decides who is eligible to own slots.

  **Assignment.** From the live set, every replica computes the same owner for
  each slot by rendezvous (highest-random-weight) hashing: `score(r, h) =
  fnv64a(r + ":" + h)`, and the owner of `h` is the replica with the top score,
  ties broken by the greatest `replica_id`. Adding or removing one replica
  reassigns only ~`1/N` of slots, so a join or leave is not a full reshuffle.

  **The lease, as a real compare-and-set.** Computing the same owner is not enough
  — two replicas reading the member set a moment apart could disagree. So the
  target owner *claims* the slot with `claim_shard.lua`, a CAS that grants the
  lease only when the slot is unowned, expired, or already the caller's, and
  bumps an `owner_epoch` on every transfer:

  ```lua title="Proposed: claim_shard.lua — the slot-ownership CAS"
  -- KEYS: 1=slot (ds:{ownership}:slot:<h>)   ARGV: 1=replica_id 2=now_ns 3=lease_ttl_ms
  -- Reply: {status, owner_id, owner_epoch, lease_expiry_ns} ; CLAIMED | RENEWED | BUSY
  local slot, me = KEYS[1], ARGV[1]
  local now = tonumber(ARGV[2])
  local owner = redis.call('HGET', slot, 'owner_id')
  local exp   = tonumber(redis.call('HGET', slot, 'lease_expiry_ns')) or 0
  if owner ~= false and owner ~= me and exp > now then
    return { 'BUSY', owner, redis.call('HGET', slot, 'owner_epoch'), tostring(exp) }
  end
  local epoch
  if owner == me then                                  -- renew: keep the epoch
    epoch = redis.call('HGET', slot, 'owner_epoch')
  else                                                 -- transfer: bump it (1 on first claim)
    epoch = tostring(redis.call('HINCRBY', slot, 'owner_epoch', 1))
  end
  local until_ns = now + tonumber(ARGV[3]) * 1000000
  redis.call('HSET', slot, 'owner_id', me, 'lease_expiry_ns', tostring(until_ns))
  return { (owner == me) and 'RENEWED' or 'CLAIMED', me, epoch, tostring(until_ns) }
  ```

  The single-threaded Redis serializes the script, so exactly one of two racing
  replicas wins the slot. The loser reads `BUSY`. This is the same shape as the
  per-subscription lease CAS in `claim.lua`, lifted to slot granularity.[^kleppmann]

  **The owner-epoch fence.** A replica that was deposed during a GC pause and then
  resumes still believes it owns the slot. `check_owner.lua` fences its writes: a
  background worker checks `(owner_id, owner_epoch)` before an external side
  effect, and a stale epoch returns `FENCED`.

  ```lua title="Proposed: check_owner.lua — fence a deposed-but-resumed owner"
  -- KEYS: 1=slot   ARGV: 1=replica_id 2=expected_epoch   Reply: OWNER | FENCED | UNOWNED
  local owner = redis.call('HGET', KEYS[1], 'owner_id')
  if owner == false then return { 'UNOWNED' } end
  if owner ~= ARGV[1] or redis.call('HGET', KEYS[1], 'owner_epoch') ~= ARGV[2] then
    return { 'FENCED' }
  end
  return { 'OWNER' }
  ```

  <Callout icon="info" label="Two fences, one boundary">
    The owner-epoch fence sits *above* the per-subscription `(generation,
    wake_id)` fence that subscriptions already use. They do different jobs: the
    owner-epoch fence suppresses a deposed owner's *wasted work*, while the
    `(generation, wake_id)` fence is the actual *safety boundary* — it makes any
    duplicate that slips through harmless. Schedule and due-set writes inline the
    owner check inside their atomic script rather than calling `check_owner`
    first, because a separate check-then-write leaves a GC pause's worth of gap
    between the two; the standalone `check_owner` is only for the external webhook
    POST, where an atomic inline check is impossible anyway.
  </Callout>

  <Complexity
    label="Complexity — Move 3"
    time="O(S/N) slots owned per replica · O(1) claim_shard CAS"
    space="O(S) slot records + O(N) member entries"
    why="Each replica owns ~S/N slots and runs the due/lease/retry workers only for those, so total background work across the fleet is O(owed), not O(N·K). A claim is one O(1) hash CAS; a rebalance on join/leave moves ~1/N of slots, each one a single <code>claim_shard</code>."
  />
</section>

<section class="block" id="liveness">
  <SectionHead num="07" label="Liveness" title="Why the sweep <em>can't be deleted</em>" />

  Move 3 introduces a window the earlier moves did not have: during a rebalance,
  a slot can briefly be owned by *nobody* — the old owner has stopped, the new
  owner has not yet claimed. The natural instinct is to lean on the generation
  fence to cover it. That instinct is wrong, and the reason is the safety/liveness
  distinction.

  The generation fence is a **safety** mechanism: it guarantees nothing bad
  happens — a duplicate or stale wake loses the comparison and is rejected. It
  says nothing about **liveness** — that something good eventually happens. An
  unowned slot is precisely a liveness gap: a wake that is owed but that no
  worker is currently responsible for firing. The fence makes a *duplicate* wake
  harmless; it does nothing for a *missing* one. Work-sharding must therefore
  supply liveness separately, and it does so by leaning on a mechanism that was
  never sharded: the full sweep.

  **The full sweep stays, covering every slot.** It keeps its ~2s cadence and
  walks all `S` slots, owned or not, re-deriving owed work from the durable
  cursors and re-firing anything behind. It carries no owner guard, because it
  issues only fence-safe wakes — if it races a real owner's wake, both bump the
  generation, one supersedes, and the stale one is fenced. So an unowned slot is
  covered by the sweep within one sweep interval (~2s), and a split-brain slot
  with two owners is safe because the double wake coalesces. The work-sharded
  due-worker is the fast path; the sweep is the floor under it.

  <Callout icon="info" label="One sweep interval, not one reconcile interval">
    The coverage gap is bounded by the ~2s sweep, not the ~30s reconcile loop.
    The sweep — `O(owed)` per tick after its reads are batched — covers all S
    slots cheaply; only the `O(streams)` fan-out-index reconcile stays at 30s. So
    a wake stranded by a rebalance waits at most ~2s, the bound the fault-injection
    tests assert against.[^bound]
  </Callout>

  There is a second reason the sweep cannot be deleted, and it is about
  durability, not rebalancing. **Only the sweep re-derives owed work from the
  durable cursor.** Suppose a failover loses a subscription's lease-ZSET tail but
  keeps its config hash: the lease worker cannot recover it, because its schedule
  entry is gone, but the sweep can, because it computes pending work from the
  cursor versus the stream tail rather than from any schedule. So the sweep is
  *demoted* — from the hot path to a rare backstop — but not *retired*. "Remove
  the `O(K)` scan" is true for the steady state and false for the recovery path.

  This is the whole argument that makes Option A safe to build one slice at a
  time: **every shard-and-own optimization sits over a still-correct full sweep.**
  Turn off slot ownership and the sweep still delivers everything, slower. That is
  what lets the migration land leased ownership, then the due-set, then the slot
  shard, each behind the same intact baseline.
</section>

<section class="block" id="e2e">
  <SectionHead num="08" label="End to end" title="One append, <em>all the way through</em>" />

  Put the three moves together and follow a single append across a three-replica
  cluster. A webhook subscription `sub-1` matches `events/*`; its keys home in
  slot `h`; Origin B currently holds slot `h`'s lease.

  <Diagram
    name="seq-hs-e2e"
    label="Steady state — append to delivery"
    caption="The appending origin (A) computes the slot, arms the wake into slot h's due-set, and delivers on the fast path — one signed POST, unchanged from today except for the slot tag. The owner (B) finds the due-set already cleared by the ack, so it does no redundant work. Ownership matters for the recovery path, not the happy one."
  />

  Walking it step by step:

  1. **The append commits.** A client `POST`s to `events/abc` on Origin A. A runs
     `append.lua`, the byte is durable, and the client gets its `204` — earned at
     the durable write, on its own clock. Everything below rides a different clock.
  2. **The fan-out finds the subscriber.** A's `OnStreamAppend` reads the
     subscribers of `events/abc`. Under slot-homing that is `S` pipelined
     `SMEMBERS`, narrowed by the occupied-slots bitmap to the slots that hold a
     subscriber; it finds `sub-1` in slot `h`.
  3. **The wake arms into the slot.** A runs `arm_wake.lua` on `{__ds:h}`: bump
     the generation, mint a `wake_id`, and — Move 2 — `ZADD sub-1` into
     `{__ds:h}:due` at `now_ns`, all in one atomic script.
  4. **The fast path delivers.** Because A handled the append, A signs and POSTs
     the wake immediately. One origin handled this append, so this delivery is not
     redundant across the fleet — ownership is not needed for a single, fence-safe
     POST.
  5. **The ack clears the owed mark.** The receiver returns `{done:true}`. A runs
     `ack.lua`: advance the cursor forward-only, release the lease, and `ZREM
     sub-1` from `{__ds:h}:due`. The subscription is idle and owes nothing.
  6. **The owner does nothing.** Origin B's due-worker, ticking over the slots it
     owns including `h`, reads `{__ds:h}:due` and finds `sub-1` already gone. The
     happy path cost the owner nothing.

  The owner earns its keep only when the fast path does not finish. Say Origin A
  crashes between step 3 and step 4 — armed the wake, never delivered. The owed
  mark is still in `{__ds:h}:due`. Origin B's due-worker claims it, checks it
  still owns `h`, and delivers the wake A dropped. Exactly one replica does this,
  because exactly one owns slot `h`.

  <Diagram
    name="seq-hs-takeover"
    label="Recovery — owner death and the coverage floor"
    caption="Origin A dies mid-delivery and Origin B's due-worker covers it. Then B dies too: its slot lease expires, the full sweep re-derives owed work for slot h from the durable cursors within ~2s, and Origin C claims the slot — owner_epoch bumps, fencing a resumed zombie B. Two layers cover the gap: the sweep for liveness, the epoch fence for safety."
  />

  Now kill Origin B too. Its member lease and its `slot:h` lease expire after
  their TTL (~9s). Two things converge to recover slot `h`. The **full sweep**,
  running on Origins A-replacement and C every ~2s over all slots, re-derives
  `sub-1`'s owed wake from its durable cursor and re-fires it — so delivery does
  not wait for ownership to settle. Meanwhile rendezvous hashing now points slot
  `h` at Origin C, which `claim_shard`s it; the CAS bumps `owner_epoch`, so if the
  old owner B ever resumes from a pause and tries to write, `check_owner` returns
  `FENCED`. Liveness is covered by the sweep; safety is covered by the epoch
  fence. Neither depends on the other.
</section>

<section class="block" id="code">
  <SectionHead num="09" label="In the code" title="What actually <em>changes</em>" />

  The design maps onto specific changes in `webhook/`. None of the per-request
  scripts changes shape; the background loops gain a slot dimension, and two new
  scripts join the dozen already in `webhook/scripts/`.

  <Table wide>
    | Loop · function | Today | Under Option A |
    | --- | --- | --- |
    | `leaseWorker` | one global lease ZSET | for each owned slot: `DueLeases(h)` → `ExpireLease`, which inlines the owner+epoch check |
    | `retryWorker` | one global retry ZSET | for each owned slot: `DueRetries(h)`; deliver only after `check_owner` returns `OWNER` |
    | `dueWorker` *(new)* | — | for each owned slot: `claim_due` over `{__ds:h}:due` → `maybeWake` |
    | `recoverySweeper` · `sweepOnce` | `List()` over one slot | iterate **all** `h ∈ [0,S)`, owned or not — the unowned-slot backstop, no owner guard |
    | `OnStreamAppend` | one `SMEMBERS` | `S` pipelined `SMEMBERS` over the occupied slots |
    | `issueWake` · `maybeWake` | as-is | unchanged but for deriving keys from `slotTag(id)` |
  </Table>

  `ownedSlots()` is recomputed each reconcile tick from the rendezvous result
  intersected with the slot leases the replica actually holds. The new tunables —
  the member-lease TTL, the heartbeat interval, the slot-lease TTL, and the
  slot-reconcile interval — join `ManagerOptions` beside the existing worker-tick,
  sweep, and reconcile intervals:

  <Table>
    | Tunable | Default | Holds the invariant |
    | --- | --- | --- |
    | `memberLeaseTTL` | 9s | a missed heartbeat past this drops the replica |
    | `heartbeatInterval` | 3s | `heartbeat < memberLeaseTTL / 2` — renew with headroom |
    | `slotLeaseTTL` | 9s | `claim_shard` lease on a slot record |
    | `slotReconcileInterval` | 3s | `slotReconcile ≤ heartbeat` — reclaim before a peer's lease lapses |
  </Table>

  These are the ownership leases, and they are unrelated to the per-subscription
  webhook `lease_ttl_ms` already in a subscription's config. The full set —
  keyspace, both Lua contracts, the rendezvous spec, and the metrics each gating
  experiment reads — is the build spec in [research doc
  05](https://github.com/adityavkk/chronicle/blob/main/docs/specs/horizontal-scale/research/05-proposed-architecture.md#option-a--build-spec-everything-an-implementer-needs).
</section>

<section class="block" id="measure">
  <SectionHead num="10" label="Measure first" title="The numbers that <em>gate the build</em>" />

  This design rests on numbers that are not yet measured. The honest order is to
  measure them before building, because two of them can fail the premise. The
  load-test rig runs the first today; the rest need the foundation in place or new
  tooling, so they are sequenced against the build, not all run up front.[^gke]

  <Table wide>
    | # | Experiment | Gates | Runnable |
    | --- | --- | --- | --- |
    | 1 | Ramp replicas 1→4 at K=10k, read managed-Redis CPU | the premise — does the sweep multiply? | now, plus a CPU reader |
    | 2 | append→wake p99 at 1 vs S parallel `SMEMBERS`, S=2/4/8/256 | **whether slot-homing is viable at all** | after the slot shard |
    | 3 | due-set `ZADD`/`ZREM` QPS on append/ack | the cost of Move 2 | after the due-set |
    | 4 | kill a slot owner; coverage-gap latency, lost/double wakes | the liveness claim (≤ ~2s) | after leased ownership |
    | 5 | drop a lease-ZSET tail mid-lease; only the reconciler recovers it | the durability claim | after the fence, on HA Redis |
  </Table>

  Experiment 2 is the one that can sink the design. Slot-homing trades a single
  `SMEMBERS` per append for up to `S` of them; if that fan-out regresses
  append→wake p99 past budget on a real cluster, even with the occupied-slots
  bitmap, the state shard is not worth its cost and the work-sharding (Moves 2-3)
  has to stand on the single slot. That is why the order is measure, then build —
  not build, then measure.

  <Callout icon="alert" label="The standing cost">
    Slot-homing makes the common path — one append fanning out — *more* expensive
    in exchange for raising the capacity ceiling. The current single-slot design
    is cheaper per append and adequate up to the point where one slot's memory or
    one primary's write throughput is the bottleneck. Option A is the right trade
    only past that point, and only if experiment 2 says the fan-out cost is
    affordable. A design that raises a ceiling you have not hit is premature.
  </Callout>
</section>

<section class="block" id="optionb">
  <SectionHead num="11" label="The fork" title="Why not <em>split the binary</em>" />

  Option A keeps one binary. The alternative, Option B, splits serving from the
  control plane into two tiers and makes each subscription an actor owned by a
  coordinator — and it is deliberately **not** chosen, only deferred. It buys
  three things Option A cannot: a cross-instance fast wake (today the append-time
  hook is in-process, so only the appending replica fires it), CPU isolation
  between serving and the control plane, and independent scaling of the two.

  The reason to defer it is that none of those three is known to be needed. The
  single binary held at K=10k, the fast path is already adequate for an
  append-then-deliver on the same origin, and a split adds a deployable, a
  pub/sub wake channel, and a second failure surface. Option A's substrate —
  slot-homing, the due-set, the `claim_shard` CAS — is exactly what Option B would
  build on, so nothing done now is wasted if the fork is later taken. The trigger
  is a measurement, not an intuition: build Option B only when experiments show
  Option A cannot deliver the cross-instance wake, the isolation, or the
  independent scaling.[^optionb]
</section>
