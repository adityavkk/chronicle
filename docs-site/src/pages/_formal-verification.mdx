<section class="block" id="why">
  <SectionHead num="01" label="Motivation" title="Why prove, when the <em>tests</em> are already good?" />

  Chronicle is already tested well, and the most dangerous logic is pinned from
  more than one side[^t-diff][^t-porcupine][^t-fault]. Formal verification adds
  three things a test cannot give, no matter how many cases you write.

  <Table>
    | A test gives you | Formal methods give you |
    | --- | --- |
    | The cases you thought to write | A statement that holds for **every** input or interleaving |
    | A green check | A **machine-checked argument** a reviewer can re-run |
    | A failure you have to reproduce | The **smallest counterexample**, already reduced for you |
  </Table>

  It pays the most where Chronicle already invested, e.g. the producer fence, the
  offset order, the generation fence, and the cursor. These are small, total, and
  central. That is the profile that rewards a proof. It is small enough to model
  faithfully and important enough that a single missed case is a correctness bug.
</section>

<section class="block" id="ladder">
  <SectionHead num="02" label="The spectrum" title="Seven methods, <em>cheapest first</em>" />

  The seven methods trade rigor against cost, cheapest first.

  <Table wide={true}>
    | # | Candidate | What it establishes | Coverage | Effort | Status |
    | --- | --- | --- | --- | --- | --- |
    | 1 | **Model-based property testing** (`rapid`) | `MemoryStore` oracle ≡ Redis backend over random op sequences | reachable states (sampled, shrunk) | low | built[^t-diff] |
    | 2 | **Differential testing**, generative | Lua mirror ≡ pure-Go validators for every generated input | input space (sampled, shrunk) | low | built[^t-diff] |
    | 3 | **Metamorphic testing** | relations: `read∘append`, fork-prefix, batch-split equivalence | sampled | low | built |
    | 4 | **Linearizability** (`porcupine`) | per-stream append/read/close has a valid serialization | recorded histories | low to med | built[^fn-porc] |
    | 5 | **Machine-checked proof** (Lean 4) | the pure cores correct for all inputs | exhaustive (proof) | medium | built |
    | 6 | **Model checking** (TLA+ / TLC / Apalache) | subscription fence safety + liveness, all interleavings | exhaustive (bounded) | med to high | built |
    | 7 | **Bounded relational** (Alloy) | fork lineage + refcount GC has no bad configuration | bounded exhaustive | medium | built |
  </Table>
</section>

<section class="block" id="pbt">
  <SectionHead num="03" label="Methods 1 to 3" title="The cheap layer: <em>oracle</em>, differential, metamorphic" />

  The method with the most coverage costs the least. `MemoryStore` is pure Go and is the oracle for the whole `Store`
  contract[^t-store]. A model-based property test treats it as a specification. It
  generates a random sequence of `Create`, `Append`, `Close`, `Delete`, `Read`,
  and `Fork` operations, runs each one against both `MemoryStore` and the Redis
  backend, and checks that the two agree after every step. One property replaces
  dozens of example tests and reaches sequences no one would write by hand.
  `rapid` drives the state machine and reduces any failure to a small reproducer.

  **Generative differential testing** extends the existing oracle table. Today one
  hand-built §5.2.1 producer table is checked byte-for-byte between pure-Go
  `ValidateProducer` and the Redis Lua path[^t-diff]. Replace the fixed rows with
  a generator over `(state, epoch, seq)` and the same check now covers the whole
  input space. The Lua mirror cannot disagree with the Go oracle on any input
  without producing a small witness.

  **Metamorphic testing** pins relations that hold without a full oracle. Reading
  from the start returns exactly what was appended. A fork read equals the source
  prefix followed by the fork's own data[^fn-fork]. For byte streams, one batch of
  N bytes and the same N bytes split into several batches read back the same.

  <Aside label="Why the oracle is free" summary="The expensive part of property testing is <em>writing the model</em>; Chronicle already paid it">
    <code>MemoryStore</code> is a complete, pure, in-process implementation of the
    same <code>Store</code> contract the Redis backend serves. Pointing a generator
    at both and checking they agree turns the model Chronicle already maintains
    into a conformance check over a large input space.
  </Aside>
</section>

<section class="block" id="linearizability">
  <SectionHead num="04" label="Candidate 4" title="Linearizability of the <em>log</em>" />

  Chronicle already uses `porcupine`, the Go linearizability checker, to model one
  subscription's lease fence as a `(generation, wake)` register. An `OK` ack under
  a stale token is the two-holder violation the checker reports[^t-porcupine]. The
  checker, the histories, and the dependency are all in the tree[^fn-porc].

  The gap was the **log itself**. Append, read, and close had never been
  linearizability-checked, even though Chronicle makes a strong claim about them.
  Every mutation is a single Lua script over keys in one hash-tag slot, so writes
  to a stream are totally ordered and serialized on the Redis primary[^fn-lua].
  That claim is what a linearizability checker tests. Record a concurrent history
  of appends and reads against a live stream, define the sequential model from the
  `MemoryStore` semantics, and let `porcupine` decide whether a valid serialization
  exists. A torn read, a lost append, or a reordering has no serialization, and the
  checker returns the witness.

  Linearizability is the correctness condition for a concurrent object.[^fn-hw] It
  pins the property the single-script design rests on, under real concurrency.
</section>

<section class="block" id="lean">
  <SectionHead num="05" label="Candidate 5" title="Machine-checked proof of the <em>pure cores</em>" />

  Three functions carry most of Chronicle's correctness. All three are pure,
  total, and deterministic, which is the shape a proof assistant handles best.

  <Table>
    | Core | File | Theorem to prove |
    | --- | --- | --- |
    | Idempotent producer | `ValidateProducer`[^fn-prod] | within an epoch the accepted sequences are a contiguous prefix; a stale epoch is always fenced; a replay is always a duplicate, never a second write |
    | Offset ordering | `Offset.Compare`[^fn-off] | `Compare` is a strict total order; appended offsets strictly increase; parse∘format is identity; no append mints a reserved sentinel |
    | Cursor progression | `GenerateResponseCursor`[^fn-cur] | the returned cursor is strictly greater than any client cursor at or ahead of the current interval, and it never goes backwards |
  </Table>

  In **Lean 4**, with `mathlib`'s order and induction libraries, these are ordinary
  theorems. The result holds for all `2^53` epochs and every sequence, not the ten rows in the table. The
  theorems below shipped on `main`.

  <Switch name="lean-cores" label="The proofs, as they shipped" tabs={["producer", "offset"]}>
    <div data-panel="0">

```lean title="lean/Chronicle/Producer/Proofs.lean"
-- ValidateProducer is total and deterministic: every input lands in exactly
-- one of two disjoint result/error shapes.
theorem validateProducer_total_det
    (state : Option ProducerState) (epoch seq nowUnix : Int) : ...

-- An accept stamps the request epoch and now, and advances lastSeq by the rules.
theorem accept_advances
    (state : Option ProducerState) (epoch seq nowUnix : Int)
    (st' : ProducerState)
    (h : (validateProducer state epoch seq nowUnix).2 = some st') :
    st'.epoch = epoch ∧ st'.lastUpdated = nowUnix
      ∧ (st'.lastSeq = 0 ∨ st'.lastSeq = seq) := ...
```

    </div>
    <div data-panel="1">

```lean title="lean/Chronicle/Offset/Proofs.lean"
-- The lexicographic offset key is injective, so the LinearOrder transports.
theorem key_injective : Function.Injective key := ...

-- compare agrees with the natural order on the (readSeq, byteOffset) pair.
theorem compare_toNat (a b : Offset) :
    compare a b =
      (if a.readSeq.toNat < b.readSeq.toNat then -1
       else if b.readSeq.toNat < a.readSeq.toNat then 1
       else if a.byteOffset.toNat < b.byteOffset.toNat then -1
       else if b.byteOffset.toNat < a.byteOffset.toNat then 1
       else 0) := ...
```

    </div>
  </Switch>

  The catch is the **proof-to-code gap**. A Lean proof proves the Lean *model*, not
  the Go. Chronicle already has the bridge. Generative differential testing
  (method 2) ties the Go to the same statement the proof closes, so the proof and
  the running code are pinned to one specification from two sides. This is the
  practical alternative to verified extraction or a Dafny rewrite. Prove the
  model, differential-test the implementation, and use the oracle the codebase
  already trusts as the contract between them.
</section>

<section class="block" id="tla">
  <SectionHead num="06" label="Candidate 6" title="A <em>TLA+</em> model of the subscription protocol" />

  The subscription engine is the part no sequential proof reaches. It is a wake,
  claim, ack, and release protocol with a monotonic `generation` fence, leases with
  a TTL, and a recovery sweep hardened against four origin-restart windows[^fn-sub].

  A **TLA+** specification states the protocol once. `TLC`, or `Apalache` working
  symbolically, explores every interleaving of concurrent workers, crashes, and
  lease expiries. The merged spec checks the safety invariant and the liveness
  properties below.

  <Switch name="tla-props" label="What TLC and Apalache check" tabs={["safety", "liveness"]}>
    <div data-panel="0">

```tla title="formal/tla/SubscriptionFence.tla"
\* SingleHolder: no two workers hold an ack-acceptable token for one sub.
\* GenMonotone (checked as an action property): sub.gen never decreases.
\* StaleInert: a token below the current gen is never ack-acceptable.
Inv ==
    /\ TypeOK
    /\ SingleHolder
    /\ AtMostOneInflightWake
    /\ CursorBounded
    /\ StaleInert
    /\ NoStrandedLease
```

    </div>
    <div data-panel="1">

```tla title="formal/tla/Liveness.tla"
\* Pending work eventually issues a wake, and is eventually delivered
\* (the cursor reaches the tail). A stranded wake is re-emitted.
PendingWorkLeadsToWake == [](( phase = "idle" /\ HasPendingWork ) ~> WakeIssued)
PendingWorkEventuallyDelivered == [](HasPendingWork ~> (cursor = tail))
```

    </div>
  </Switch>

  It completes the porcupine model, which is time-free and safety-only by design,
  with the lease clock and the liveness properties[^t-porcupine].

  No, it is not too late to model a built system.

  <Aside label="Why not too late" summary="<code>TLC</code> still finds design bugs, and <strong>trace validation</strong> binds the spec to the running Go">
    Trace validation instruments Chronicle (or the jepsen harness) to emit
    operation traces and checks each is a behaviour the spec permits. MongoDB and
    Microsoft's CCF apply this to shipping systems. The spec doubles as precise,
    executable documentation of the fence the porcupine model only half-captures.
  </Aside>
</section>

<section class="block" id="alloy">
  <SectionHead num="07" label="Candidate 7" title="Alloy for the <em>fork graph</em> (optional)" />

  Forking forms a relational structure. A stream points at a source and a fork
  offset. A reference count tracks live forks. Deleting a referenced stream
  soft-deletes it until a garbage collector reclaims it in a cascade. The
  invariants are relational. The reference count equals the number of live forks.
  The GC cascade terminates. No live fork loses inherited data. No soft-deleted
  stream is left behind after its last fork is gone.

  **Alloy** fits this case. Declare the relations, assert the invariants, and the
  analyzer finds a minimal counterexample configuration within a bounded scope or
  reports that none exists. It suits the lineage graph in a way that a value proof
  and a linear-history checker do not, and gives a cleaner exhaustive argument over
  the graph shape than model-based PBT (method 1) does.
</section>

<section class="block" id="verdicts">
  <SectionHead num="09" label="Decisions" title="The strategic <em>verdicts</em>" />

  <Table>
    | Question | Verdict |
    | --- | --- |
    | **Is it too late for TLA+?** | No. `TLC` and `Apalache` still find protocol design bugs, and **trace validation** binds the spec to the running Go. The spec doubles as precise documentation. |
    | **Lean vs Dafny vs TLA+ for the cores?** | **Lean** for the pure, sequential, total functions, a value-level proof over all inputs. Dafny is strong when the proof must discharge against imperative code, but Chronicle's cores are already pure and the Go is the oracle, so a Lean model plus differential PBT has less friction. TLA+ is the wrong tool for value functions and the right one for the concurrent protocol. |
    | **Which Go property library?** | **`rapid`**, for its built-in stateful state-machine testing and integrated shrinking. `gopter` is heavier and `testing/quick` has no shrinking. Native Go fuzzing complements it for the byte-level parsers. |
    | **What is provable vs only chaos-testable?** | Everything inside one Redis primary and all the pure logic is provable or exhaustively checkable, e.g. per-stream serialization, the fences, and the orders. **Cross-failover durability is not.** Async replication can lose an acked write. That is documented honestly and exercised by the pod and Redis kill nemesis, never proven. |
  </Table>
</section>

<section class="block" id="lands">
  <SectionHead num="10" label="Plan" title="What proof does <em>not</em> buy, and where this lands" />

  A proof is only as good as its model, and bounded checkers like `TLC` and Alloy
  are exhaustive only up to a scope. None of this replaces the jepsen durability
  runs. It surrounds them. The work split across two worktrees, cheapest and
  highest-coverage first.

  <Table>
    | Phase | Work | Worktree |
    | --- | --- | --- |
    | **P1** | Model-based equivalence + generative differential + metamorphic, on `rapid`; native fuzz on the parsers | `property-based-testing` |
    | **P2** | `porcupine` linearizability of the log; Lean proofs of the three cores | both |
    | **P3** | TLA+ spec of the subscription fence + trace-validation hooks | `formal-verification` |
    | **P4** | Alloy fork-lineage model; wire the green checks into CI | `formal-verification` |
  </Table>
