<Tldr>
  <ul>
    <li><strong>A write crosses five files and is decided once.</strong> <code>ServeHTTP</code> routes <code>POST</code> to <code>handleAppend</code>, the backend snapshots the tail and sends one <code>EVALSHA</code>, and <code>append.lua</code> commits or returns <code>RETRY</code> if the live tail moved.</li>
    <li><strong>A long-poll reads, then subscribes, then re-reads.</strong> The fast-path read finds nothing, and the waiter <code>SUBSCRIBE</code>s before re-reading. That order is the whole missed-wakeup argument. An append in the gap is caught by the re-read or the <code>PUBLISH</code>.</li>
    <li><strong>A durable append wakes a subscriber, fenced.</strong> <code>onStreamAppend</code> hands off to the webhook manager, which signs a webhook or arms a pull-wake. A <code>generation</code> counter fences every ack, so a crash costs latency and not correctness.</li>
  </ul>
</Tldr>

<section class="block" id="append">
  <SectionHead num="01" label="POST" title="A write <em>lands</em>" />

  A `POST` carries a JSON batch to a stream that already exists. The path crosses
  five files. The HTTP handler parses and delegates, the backend snapshots the
  tail and sends one Lua call, the script validates and writes, and the handler
  answers. The diagram below shows the call order. The steps after it show the
  source lines.

  <Diagram name="seq-trace-append" label="Append · call stack" caption="The optimistic-concurrency loop is the part to watch. The backend snapshots the tail before framing, and <code>append.lua</code> rejects the write with <code>RETRY</code> if the live tail has moved. The Go caller then re-snapshots and retries, up to 64 times." />

  <Step n="1" fn="Handler.ServeHTTP" loc="handler.go:130" role="The method switch routes POST to handleAppend. Every verb takes this same switch.">

```go title="handler.go" showLineNumbers startLineNumber=130
	case http.MethodPost:
		err = h.handleAppend(w, r, streamPath)
```

Before control reaches the switch, `ServeHTTP` has set CORS and security
headers, answered any `OPTIONS` preflight, and ruled out the reserved `__ds`
namespace. This switch is the whole router.

  </Step>

  <Step n="2" fn="Handler.handleAppend" loc="handler.go:584" role="Confirm the stream exists, parse the producer and closure headers, read the body, build options, call the store.">

```go title="handler.go" showLineNumbers startLineNumber=584
meta, err := h.Store.Get(path)
if err != nil {
	if errors.Is(err, store.ErrStreamNotFound) {
		return newHTTPError(http.StatusNotFound, "stream not found")
```

After the existence check, `handleAppend` parses `Producer-Id`, `Producer-Epoch`,
`Producer-Seq`, and `Stream-Closed`, reads the request body, verifies the
`Content-Type` matches the stream, and packs it all into an `AppendOptions`
before calling `h.Store.Append`. It decides nothing about ordering or
idempotency. The store and the script handle that.

  </Step>

  <Step n="3" fn="redis.Store.Append" loc="store/redis/store.go:574" role="The backend has already snapshotted the tail with HMGET and pre-framed the batch. Now it sends everything to one EVALSHA.">

```go title="store/redis/store.go" showLineNumbers startLineNumber=574
args := append([]any{
	s.nowNsArg(), notifyChannel(path), reqCT, opts.Seq, closeArg,
	hasProd, pid, epochArg, seqArg,
	base.String(), newTail.String(), valOnly,
}, frames...)
raw, err := appendScript.Run(ctx, s.client, keysFor(path), args...).Result()
```

`base.String()` is the snapshot tail. The script checks that the live tail still
matches this value before it commits. `newTail.String()` is where the tail moves
if this write wins. `frames` are the `endOffset|bytes` members to add. `Run` is
`EVALSHA`, and it reloads the script on a `NOSCRIPT` miss. A `RETRY` reply sends
the caller back to re-snapshot and re-run.

  </Step>

  <Step n="4" fn="append.lua commit" loc="store/redis/scripts/append.lua:95" role="Every check has passed and the snapshot tail still matches. The script writes atomically, because the whole script runs as one Redis step.">

```lua title="store/redis/scripts/append.lua" showLineNumbers startLineNumber=95
-- 11. Write frames (chunked: unpack is C-stack bounded) and commit metadata.
if #ARGV >= 13 then
  local i = 13
  while i <= #ARGV do
    local stop = math.min(i + 999, #ARGV)
    local zargs = { 'ZADD', KEYS[2] }
    for j = i, stop do
      zargs[#zargs + 1] = '0'
      zargs[#zargs + 1] = ARGV[j]
    end
    redis.call(unpack(zargs))
    i = stop + 1
  end
  redis.call('HSET', KEYS[1], 'tail', new_tail)
end
```

Every frame goes into the `msg` sorted set at score `0`. The ordering comes from
the offset prefix of each member, so `ZRANGEBYLEX` reads them back in sequence.
Then the tail advances. Just below this, the script records producer state into
the `prod` hash and runs `PUBLISH channel 'a'`. That publish is the wakeup the
[read walkthrough](#read) is waiting on. The script then returns `OK` with the
new tail.

  </Step>

  <Complexity
    time="O(k log k) for a k-frame batch · plus O(r) on contention"
    space="O(b) for a b-byte batch"
    why="Each frame is one <code>ZADD</code> into the <code>msg</code> sorted set, chunked at 1,000 members because Lua's <code>unpack</code> is C-stack bounded. The <code>r</code> term is the optimistic retry: a concurrent append on the same stream re-snapshots the tail and re-runs, up to 64 times."
  />

  <Aside label="Edge case" summary="Why the <code>ZADD</code> is chunked and the retry caps at 64">
    The `ZADD` writes at most 1,000 members per call because Lua's `unpack` is
    bounded by the C stack, so a larger batch is split across calls inside the one
    script. On the Go side, a `RETRY` reply means the live tail moved between the
    snapshot and the commit, so the caller re-snapshots and re-runs, up to
    `maxAppendRetries` (64) times before giving up.
  </Aside>

  <Step n="5" fn="Handler.handleAppend respond" loc="handler.go:775" role="Back in Go. Fire the subscription hook, set the next offset, then choose the status.">

```go title="handler.go" showLineNumbers startLineNumber=775
h.onStreamAppend(path)
w.Header().Set(protocol.HeaderStreamNextOffset, result.Offset.String())
if opts.ProducerId != "" {
	w.WriteHeader(http.StatusOK)
} else {
	w.WriteHeader(http.StatusNoContent)
}
return nil
```

`onStreamAppend` is the best-effort hand-off into the subscription plane. The
[wake walkthrough](#wake) starts exactly here. A producer append answers `200 OK`.
A plain append answers `204 No Content`. Both carry `Stream-Next-Offset`, the one
string a client stores and passes back as `?offset=` to resume. There are no
consumer groups and no server-side cursor.

  </Step>

  Five files, one durable write. The core makes the decisions and the script
  enforces them. The one weak point on this path is concurrency. Several in-flight
  appends on the <em>same</em> stream collide on the `RETRY` loop, and past
  roughly 1,000/s in aggregate the retries run out and useful throughput drops to
  near zero.[^perf] One in-flight append per stream sustains far more.
</section>

<section class="block" id="read">
  <SectionHead num="02" label="GET live=long-poll" title="A reader <em>tails</em>" />

  A `GET` with `offset=now&live=long-poll` means the caller is at the tail and
  wants the <em>next</em> write. The fast path reads and finds nothing. The slow
  path subscribes to a pub/sub channel and blocks. Correctness depends on doing
  those two steps in the right order. The same tail-follow request has two live
  modes, and only the `live=` value changes.

  <Switch name="trace-live" label="Live read modes" tabs={["long-poll", "SSE"]}>
    <div data-panel="0">

```bash title="Long-poll: block for the next write"
curl -i 'http://localhost:4437/v1/stream/demo?offset=now&live=long-poll'
```

    </div>
    <div data-panel="1">

```bash title="SSE: stream writes as they land"
curl -i 'http://localhost:4437/v1/stream/demo?offset=now&live=sse'
```

    </div>
  </Switch>

  <Diagram name="seq-trace-read" label="Long-poll · call stack" caption="The race this design closes. An append that lands between the fast-path read and the <code>SUBSCRIBE</code> would be missed by a waiter that subscribed too late. Subscribing first and then re-reading means the waiter sees data that arrived in the gap, or the <code>PUBLISH</code> that follows." />

  <Step n="1" fn="Handler.handleRead parse" loc="handler.go:369" role="Resolve the offset and the live mode. offset=now means the tail, and live=long-poll means block.">

```go title="handler.go" showLineNumbers startLineNumber=369
	// Parse offset
	offset, err := store.ParseOffset(offsetStr)
	if err != nil {
		return newHTTPError(http.StatusBadRequest, "invalid offset")
	}

	// Check for live mode
	liveMode := query.Get("live")
	cursor := query.Get("cursor")
	// Validate long-poll requires offset
	if liveMode == "long-poll" && !offsetProvided {
		return newHTTPError(http.StatusBadRequest, "offset required for long-poll mode")
	}
```

`ParseOffset` maps `now` to a value that means the current tail, and `-1` to the
start. Any other value parses as `readseq_byteoffset`. `cursor` is the CDN
cache key. The handler echoes it back and advances it on each request, so a cache
cannot serve a stale tail in a loop.

  </Step>

  <Step n="2" fn="redis.Store.Read" loc="store/redis/store.go:644" role="One Lua call with a lexicographic lower bound built from the caller's offset.">

```go title="store/redis/store.go" showLineNumbers startLineNumber=644
func (s *Store) Read(path string, offset store.Offset) ([]store.Message, bool, error) {
	ctx := context.Background()
	raw, err := readScript.Run(ctx, s.client, keysFor(path), s.nowNsArg(), lexLowerBound(offset)).Result()
```

`lexLowerBound(offset)` is the exclusive bound just past the caller's offset. It
selects everything strictly after the caller's position. The work happens inside
`read.lua`.

  </Step>

  <Step n="3" fn="read.lua range the set" loc="store/redis/scripts/read.lua:22" role="A read counts as access, so the script slides the TTL forward, then ranges the sorted set.">

```lua title="store/redis/scripts/read.lua" showLineNumbers startLineNumber=22
-- Sliding-TTL touch: Read counts as access (Get does not).
m.accessedAtNs = string.format('%.0f', now)
redis.call('HSET', KEYS[1], 'accessedAtNs', m.accessedAtNs)
refresh_backstop(m, now)

local members = redis.call('ZRANGEBYLEX', KEYS[2], ARGV[2], '+')
return { 'OK', redis.call('HGETALL', KEYS[1]), members }
```

`ZRANGEBYLEX` returns every frame from the caller's bound to the end. Go decodes
the `offset|bytes` members into messages. When the caller is already at the tail,
`members` is empty, and an empty result is what sends this request onto the
waiting path.

  </Step>

  <Complexity
    time="O(log N + M) for M frames returned"
    space="O(M) for the returned frames"
    why="<code>ZRANGEBYLEX</code> seeks the lexicographic lower bound in <code>O(log N)</code> over the <code>msg</code> set, then streams the <code>M</code> frames from the caller's offset to the tail. At the tail, <code>M</code> is zero and the empty result is what sends the request onto the waiting path."
  />

  <Step n="4" fn="Handler.handleRead decide to wait" loc="handler.go:457" role="Empty result, caller at the tail, long-poll asked for. Only now does the request block.">


If the stream is already closed at the tail, the handler returns `204`
immediately rather than waiting forever. Otherwise it calls
`h.Store.WaitForMessages` with the configured `LongPollTimeout`, which is 30
seconds by default.

  </Step>

  <Step n="5" fn="redis.Store.WaitForMessages" loc="store/redis/notify.go:48" role="Subscribe first, then re-read. The order is the entire correctness argument.">

```go title="store/redis/notify.go" showLineNumbers startLineNumber=48
	pubsub := s.client.Subscribe(ctx, notifyChannel(path))
	defer func() { _ = pubsub.Close() }()
	if _, err := pubsub.Receive(ctx); err != nil { // confirm subscription
		return nil, false, false, err
	}
	wake := pubsub.Channel()

	// Re-read before waiting: closes the missed-wakeup race window between
	// the fast-path read and the subscribe.
	if msgs, closed, done, err := s.recheck(path, offset); done {
		return msgs, false, closed, err
	}
```

An append that lands between the fast-path read and the `SUBSCRIBE` would never
reach a listener that was not there yet. Subscribing <em>before</em> the re-read
closes that window. If the write arrived in the gap, the re-read finds it. If it
arrives later, the subscription catches the `PUBLISH 'a'`. From here a `select`
loop waits on the wake channel, a one-second backup poll, or the timeout.

  </Step>

  When [walkthrough one](#append) runs `PUBLISH channel 'a'` on this stream, the
  wake channel fires, `recheck` re-runs the read, and the new frames come back.
  The handler formats them, sets a fresh `Stream-Cursor`, and answers `200` with
  the frames and the next offset. If nothing arrives before the timeout, it
  answers `204 No Content`, and the client long-polls again.

  <Callout icon="info" label="One known cost">
    Each `WaitForMessages` opens its own pub/sub subscription, plus a one-second
    backup poll (`defensivePollInterval`) that re-reads while blocked. The wake
    arrives on the `PUBLISH`, so the poll is a floor, not the common case. The fix
    sketched in the benchmark report is one pub/sub hub per stream that stays
    open.[^perf]
  </Callout>

<section class="block" id="wake">
  <SectionHead num="03" label="OnStreamAppend" title="A subscription <em>wakes</em>" />

  The append walkthrough ended by firing `onStreamAppend`. This walkthrough picks
  it up. A durable write on a matched stream wakes a registered subscriber, in one
  of two ways. A signed webhook, or a pull-wake event that a worker claims. A
  generation counter fences every message, so a crash anywhere costs latency and
  not correctness.

  <Diagram name="seq-trace-wake" label="Subscription wake · call stack" caption="Two delivery modes share one fencing model. <strong>Webhook</strong>: sign and POST, auto-ack on <code>{done:true}</code>. <strong>Pull-wake</strong>: append a wake event, let workers race to <code>claim</code> a lease. A periodic <code>sweepOnce</code> floor re-arms anything left stranded by a crash." />

  <Step n="1" fn="Handler.onStreamAppend" loc="handler.go:63" role="The exact call that closed the append trace. With no subscription layer wired, it is a no-op.">

```go title="handler.go" showLineNumbers startLineNumber=63
func (h *Handler) onStreamAppend(path string) {
	if h.SubHooks != nil {
		h.SubHooks.OnStreamAppend(subStreamPath(path))
	}
}
```

`subStreamPath` strips the leading slash to match the protocol's wire paths.
This runs inline on the append's goroutine, right after the durable write. It is
the best-effort, low-latency path into the subscription plane.

  </Step>

  <Step n="2" fn="Manager.OnStreamAppend" loc="webhook/manager.go:359" role="Look up who is listening on this stream from the fan-out index, and wake each.">

```go title="webhook/manager.go" showLineNumbers startLineNumber=359
func (m *Manager) OnStreamAppend(path string) {
	start := time.Now()
	ids, slotsProbed, err := m.store.StreamSubscribers(path)
	if err != nil {
```

`StreamSubscribers` reads a Redis set kept per stream. That set is the index of
who is listening. For each subscription, `maybeWake` checks that it is idle and
that its cursor is behind the new tail, which means it has pending work, before
arming anything. Arming a wake runs `arm_wake.lua`, which bumps the
subscription's `generation` and stamps a fresh `wake_id`. The [modules
page](./modules#subs) shows that script.

  </Step>

  <Step n="3" fn="claim.lua the pull-wake fence" loc="webhook/scripts/claim.lua:36" role="Two workers race to claim the wake. The lease compare-and-set picks one, and the fence rotates.">

```lua title="webhook/scripts/claim.lua" showLineNumbers startLineNumber=36
local gen = redis.call('HGET', sub, 'generation')
local wake = redis.call('HGET', sub, 'wake_id')
if not (phase == 'waking' and wake ~= '') then
  gen = tostring(redis.call('HINCRBY', sub, 'generation', 1))
  wake = ARGV[5]
  redis.call('HSET', sub, 'wake_id', wake)
end
local until_ns = now + tonumber(ARGV[4]) * 1000000
redis.call('HSET', sub, 'phase', 'live', 'holder', '1', 'holder_worker', ARGV[2], 'lease_until_ns', tostring(until_ns))
```

The winner takes the lease and the current `generation`. The loser finds a held,
unexpired lease and gets the `409 BUSY` reply. Every ack the holder later sends
carries its generation. Any ack under a stale generation, from a former holder
whose lease expired and was reclaimed, is fenced out. The webhook path skips the
race. `deliverWebhook` signs the body with Ed25519, runs the URL past an SSRF
guard, POSTs it, and auto-acks on a `2xx {done:true}`. That is the same
generation fence applied to a push instead of a pull.

  </Step>

  A backstop keeps the whole plane working through crashes. The append-time hook
  is the fast path, and a periodic floor guarantees the work still happens.

  <Aside label="Recovery" summary="How the sweep rebuilds state after a crash">
    The recovery floor runs in `recoveryLoop`, which calls `sweepOnce` on a ticker
    at `defaultSweepInterval`, 30 seconds by default. It re-emits a pull-wake event
    that was armed but never written (`wake_event_sent_ns == 0`), expires a lease
    whose holder vanished, and re-wakes an idle subscription that has pending work.
    The latency-sensitive cases are event-triggered (boot, reconnect, append
    error), so the periodic floor only bounds the one eventless case. A crash
    between any two steps above is recoverable for this reason. The
    [Subscriptions](./../subscriptions) reference walks the fencing and recovery in
    full.
  </Aside>

  [^perf]: Both the append throughput drop and the wakeup-latency floor are measured and explained in the [benchmark report](https://github.com/adityavkk/chronicle/blob/loadgen/docs/benchmarks/2026-06-10/REPORT.md). The [Overview](./../#performance) summarizes them.
