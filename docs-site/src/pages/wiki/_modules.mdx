<Tldr>
  <ul>
    <li><strong>The shell is the only networked code.</strong> <code>Handler.ServeHTTP</code> sets headers, guards the <code>__ds</code> namespace, and dispatches one <code>switch</code> over five verbs to five handlers. No router, no route table.</li>
    <li><strong>The core is pure functions.</strong> TTL and offset grammars, cursor math, and the producer state machine (<code>ValidateProducer</code>) decide everything possible without I/O, so they test without mocks.</li>
    <li><strong>The handler never names Redis.</strong> It depends on <code>store.Store</code>, an eleven-method interface. A Redis backend and an in-memory reference backend implement it, and the handler cannot tell them apart.</li>
    <li><strong>Every Redis mutation is one Lua script.</strong> Create, append, close, and delete each run as a single <code>EVALSHA</code>, so validate, write, and publish are atomic and serialized per stream.</li>
    <li><strong>Subscriptions attach through two hooks.</strong> A durable append fires <code>OnStreamAppend</code>; the webhook <code>Manager</code> turns it into a fenced wake, recovered by a sweep so a crash cannot lose or double-deliver it.</li>
  </ul>
</Tldr>

<section class="block" id="shell">
  <SectionHead num="01" label="handler.go · mount.go" title="The imperative <em>shell</em>" />

  The shell is the only code that knows it is on a network. `Handler.ServeHTTP`
  is the entry point. It sets CORS and the browser security headers, answers
  `OPTIONS` preflight, intercepts the reserved `__ds` subscription namespace, and
  then dispatches on the HTTP method. The dispatch is one `switch`. There is no
  router, no route table, and no regex. A verb maps to a handler, and the handler
  maps to one store call.

  <Step n="1" fn="Handler.ServeHTTP, the dispatch" loc="handler.go:122-141" role="Five verbs, five handlers. This switch is the whole router. Above it is headers and the <code>__ds</code> guard. Below it is one error writer.">

```go title="handler.go" showLineNumbers startLineNumber=122
	var err error
	switch r.Method {
	case http.MethodPut:
		err = h.handleCreate(w, r, streamPath)
	case http.MethodHead:
		err = h.handleHead(w, r, streamPath)
	case http.MethodGet:
		err = h.handleRead(w, r, streamPath)
	case http.MethodPost:
		err = h.handleAppend(w, r, streamPath)
	case http.MethodDelete:
		err = h.handleDelete(w, r, streamPath)
	default:
		w.WriteHeader(http.StatusMethodNotAllowed)
		return
	}

	if err != nil {
		h.writeError(w, err)
	}
```

Each `handle*` method does the same three steps in order. It parses the request
(headers, query, body), calls one method on `h.Store`, and turns the result into
a status code and response headers. Errors return as values and meet a single
`writeError` at the bottom, which maps each `store` error variable to its HTTP
status.

<Aside label="Edge case" summary="How <code>Mount</code> keeps the handler path-relative">
  <code>Mount</code> wraps the handler. It strips the stream-root prefix
  (<code>/v1/stream/</code>) before dispatch and rewrites <code>Location</code>
  headers on the way back out, so the handler core only ever sees store-relative
  paths.
</Aside>

  </Step>

<section class="block" id="protocol">
  <SectionHead num="02" label="protocol/ · store/producer.go" title="The pure <em>core</em>" />

  Everything you can decide without I/O lives here as plain functions over
  immutable inputs. e.g. the TTL and offset grammars (`ParseTTL`, `ParseOffset`),
  the cursor math that busts the CDN cache (`GenerateResponseCursor`), and the
  producer-validation state machine. None of it touches Redis or the network, so
  all of it is tested without mocks.

  The state machine is the main piece. Protocol section 5.2.1, idempotent
  producers, is written once into `ValidateProducer`. Given a producer's stored
  state and an incoming `(epoch, seq)`, it returns exactly one of five outcomes.

  <Step n="1" fn="ValidateProducer, two of five branches" loc="store/producer.go:17" role="State in, decision out, no side effects. The same five branches are written again in Lua so Redis can enforce them atomically, with this function as the test oracle.">

```go title="store/producer.go" showLineNumbers startLineNumber=17
func ValidateProducer(state *ProducerState, epoch, seq, nowUnix int64) (AppendResult, *ProducerState, error) {
	// No existing state - accept as new producer
	if state == nil {
		if seq != 0 {
			// First message from producer must be seq=0
			return AppendResult{
				ProducerResult: ProducerResultNone,
				ExpectedSeq:    0,
				ReceivedSeq:    seq,
			}, nil, ErrProducerSeqGap
		}
		newState := &ProducerState{
			Epoch:       epoch,
			LastSeq:     0,
			LastUpdated: nowUnix,
		}
		return AppendResult{
			ProducerResult: ProducerResultAccepted,
			LastSeq:        0,
		}, newState, nil
	}

	// Epoch validation (client-declared, server-validated)
	if epoch < state.Epoch {
		// Stale epoch - zombie fencing
		return AppendResult{
			ProducerResult: ProducerResultNone,
			CurrentEpoch:   state.Epoch,
		}, nil, ErrStaleEpoch
	}
```

Three more branches follow in the same shape. An **epoch bump**: a higher epoch
must restart at `seq=0`, which fences out a writer that came back from a stale
epoch. A **duplicate**: `seq ≤ lastSeq` returns idempotent success so a retried
append never writes twice. A **gap**: `seq > lastSeq+1` is rejected with the
expected and received sequence numbers. The function returns a new state value
instead of changing its argument. The caller decides whether to persist it.

  </Step>
</section>

<section class="block" id="store">
  <SectionHead num="03" label="store/store.go" title="The storage <em>contract</em>" />

  The handler never names Redis. It depends on `store.Store`, an interface of
  eleven methods, and on a small set of error variables (`ErrStreamNotFound`,
  `ErrStreamClosed`, `ErrStaleEpoch`, and so on) that the handler maps to status
  codes. Two backends satisfy the interface. The Redis store runs in production,
  and an in-memory store is the reference implementation in tests. The handler
  cannot tell them apart, so the in-memory store also serves as a runnable
  specification of what Redis must do.

  <Step n="1" fn="store.Store, the method set" loc="store/store.go:98" role="The whole contract. The doc comments spell out every error case. They are left out here, so follow the locator to read them.">

```go title="store/store.go (the Store interface, doc comments elided)"
type Store interface {
	Create(path string, opts CreateOptions) (*StreamMetadata, bool, error)
	Get(path string) (*StreamMetadata, error)
	Has(path string) bool
	Delete(path string) error
	Append(path string, data []byte, opts AppendOptions) (AppendResult, error)
	CloseStream(path string) (*CloseResult, error)
	CloseStreamWithProducer(path string, opts CloseProducerOptions) (*CloseProducerResult, error)
	Read(path string, offset Offset) ([]Message, bool, error)
	WaitForMessages(ctx context.Context, path string, offset Offset, timeout time.Duration) (messages []Message, timedOut bool, streamClosed bool, err error)
	GetCurrentOffset(path string) (Offset, error)
	Close() error
}
```

The core type in the contract is the `Offset`. It is two zero-padded 64-bit
integers, `ReadSeq` and `ByteOffset`, formatted as `"%016d_%016d"`. The padding
makes the string form sort in the same order as the numbers. The Redis backend
can then store frames in a sorted set keyed by offset string and read a range
with one `ZRANGEBYLEX`. `Read` returns the messages plus an `upToDate` bool.
`WaitForMessages` is the long-poll call that blocks until an append, a close, or
a timeout.

<Complexity
  time="O(log N + M) per read"
  space="O(M) for the M returned frames"
  why="Because the padded offset string sorts in numeric order, a read is one <code>ZRANGEBYLEX</code>: a log-N seek into the sorted set plus the M frames in range. No scan, no per-frame round trip."
/>

  </Step>

<section class="block" id="redis">
  <SectionHead num="04" label="store/redis/" title="The Redis <em>backend</em>" />

  Each stream is four Redis keys. They share a `{path}` hash tag so they live in
  one cluster slot. The keys are a `meta` hash (tail offset, content type, closed
  flag, TTL), a `msg` sorted set (the frames), a `prod` hash (producer state),
  and a `notify` pub/sub channel. Every mutation, which is create, append, close,
  or delete, is one Lua script run with `EVALSHA`. Its whole validate, write, and
  publish sequence is atomic and runs one at a time per stream. The Go side
  prepares arguments, runs the script, and handles retries. The script makes the
  decisions.

  `append.lua` opens with a chain of validation checks in the exact order the
  spec sets. The Go caller has already read the tail and framed the batch. The
  script re-checks everything against live state before it writes a byte.

  <Step n="1" fn="append.lua, the validation chain" loc="store/redis/scripts/append.lua:34" role="Existence, soft-delete, expiry, then the TTL touch, in that order, because that order is the spec. A mismatch returns an error reply and writes nothing.">

```lua title="store/redis/scripts/append.lua" showLineNumbers startLineNumber=34
-- 1. Existence.
local m = meta_map(KEYS[1])
if m == nil then return make_reply('NOTFOUND') end

-- 2. Soft-deleted (before expiry, mirroring MemoryStore order).
if m.softDel == '1' then return make_reply('SOFTDEL') end

-- 3. Lazy expiry.
if is_expired(m, now) then
  expire_cleanup(m)
  return make_reply('NOTFOUND')
end

-- 4. Sliding-TTL touch (upstream touches before the closed check, so even
-- rejected appends refresh the window).
m.accessedAtNs = string.format('%.0f', now)
redis.call('HSET', KEYS[1], 'accessedAtNs', m.accessedAtNs)
```

The closed-stream, content-type, producer, and `Stream-Seq` checks run next. Only
then does the script compare the read tail against the live tail and, if it still
matches, write.

<Aside label="Edge case" summary="Why the retry loop is bounded at 64">
  The Go caller re-snapshots the tail and re-runs the script on every
  <code>RETRY</code>, up to 64 attempts. Past that it returns an error rather
  than spin forever under pathological contention on one stream.
</Aside>

    Atomicity is per primary. Redis replication is asynchronous, so an
    acknowledged append can be lost on failover. Within a healthy primary,
    appends are atomic and producer idempotency is exact.

<section class="block" id="subs">
  <SectionHead num="05" label="webhook/" title="Subscriptions &amp; <em>webhooks</em>" />

  The subscription plane is optional. It attaches through two hooks the handler
  fires: `OnStreamCreated` and `OnStreamAppend`. When a durable append lands on a
  matched stream, the webhook `Manager` looks up the stream's subscribers in a
  fan-out index and wakes any that are idle and have pending work. A wake is not a
  one-way notification. It is a fenced state change in Redis, so a crash on either
  side can be recovered instead of lost or delivered twice. A fence here is a
  counter Redis checks on every later message, and it rejects any message that
  carries an older count.

  `arm_wake.lua` creates a wake. It only fires if the subscription is idle, and it
  increments a `generation` counter that fences every later message about this
  wake.

  <Step n="1" fn="arm_wake.lua, arming a fenced wake" loc="webhook/scripts/arm_wake.lua" role="Guard on idle, then increment the generation and stamp a fresh wake_id. Webhook delivery arms a lease at once. Pull-wake leaves a flag the recovery loop watches.">

```lua title="webhook/scripts/arm_wake.lua"
local sub = KEYS[1]
if redis.call('EXISTS', sub) == 0 then
  return { 'NOSUB' }
end
if redis.call('HGET', sub, 'phase') ~= 'idle' then
  return { 'BUSY', redis.call('HGET', sub, 'generation'), redis.call('HGET', sub, 'wake_id') }
end
local gen = redis.call('HINCRBY', sub, 'generation', 1)
redis.call('HSET', sub, 'wake_id', ARGV[5], 'phase', 'waking', 'holder', '0', 'holder_worker', '')
if ARGV[4] == '1' then
  local until_ns = tonumber(ARGV[2]) + tonumber(ARGV[3]) * 1000000
  redis.call('HSET', sub, 'lease_until_ns', tostring(until_ns))
  redis.call('ZADD', KEYS[2], until_ns, ARGV[1])
else
  redis.call('HSET', sub, 'wake_event_sent_ns', '0')
end
return { 'ARMED', tostring(gen), ARGV[5] }
```

That `generation` is the fence. Every delivery, ack, and claim carries the
generation it was issued under, and Redis rejects any that arrives under an older
one with a `409`. Webhook delivery signs the POST with Ed25519 and runs it past
an SSRF guard. Pull-wake instead appends an event to the worker's wake stream and
lets workers race to `claim` a lease. A recovery loop backs both. It runs every
30 seconds, re-arms wakes stranded by a crash, and expires leases whose holder
went away. It is the durability backstop for the best-effort append-time hook.

  </Step>
