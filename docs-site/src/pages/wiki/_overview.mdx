<Tldr>
  <ul>
    <li><strong>One handler, one switch.</strong> Every request lands in <code>Handler.ServeHTTP</code> and dispatches on the HTTP method to one of five handlers. That switch is the whole router.</li>
    <li><strong>Decisions are pure. Effects are not.</strong> Everything you can decide without I/O lives in the <code>protocol</code> and <code>store</code> packages as plain functions, e.g. header grammar, offset math, and the producer state machine. The handler and the Redis backend only carry data to and from those decisions.</li>
    <li><strong>The store is an interface.</strong> The handler talks to <code>store.Store</code>, an 11-method contract. A Redis backend and an in-memory backend implement it. The handler cannot tell them apart.</li>
    <li><strong>Every mutation is one Lua script.</strong> Create, append, close, and delete each compile to a single <code>EVALSHA</code> call, so validation, write, and wakeup happen atomically inside Redis.</li>
    <li><strong>Subscriptions run from a callback.</strong> A durable append fires <code>onStreamAppend</code>. The webhook manager turns that into a signed wake or a pull-wake event, fenced so a crash cannot lose or double-deliver it.</li>
  </ul>
</Tldr>

<section class="block" id="what">
  <SectionHead num="01" label="Orientation" title="Every package, <em>and its one job</em>" />

  Chronicle implements the [Durable Streams protocol](./../). A stream is a URL,
  and ordinary HTTP verbs create it, append to it, read it, and tail it. The
  [Overview](./../) explains the protocol and the performance envelope. This
  wiki covers the source. It answers a different question. If you opened the
  repository, where would you look, and what would each piece be doing?

  Five packages plus the command. Each row is a directory or file, with the one
  function to open first. Every reference links to the exact lines on GitHub.

  <Table>
    | Package | Job | Start at |

    | [`handler.go`](https://github.com/adityavkk/chronicle/blob/main/handler.go) · [`mount.go`](https://github.com/adityavkk/chronicle/blob/main/mount.go) | The imperative shell: route by method, parse, respond | [`Handler.ServeHTTP`](https://github.com/adityavkk/chronicle/blob/main/handler.go#L83) |
    | [`protocol/`](https://github.com/adityavkk/chronicle/tree/main/protocol) | Pure core: header grammar, cursor math, SSE framing | [`ParseTTL`](https://github.com/adityavkk/chronicle/blob/main/protocol/parse.go#L27) |
    | [`store/`](https://github.com/adityavkk/chronicle/tree/main/store) | The storage contract + offset type + producer rules | [`store.Store`](https://github.com/adityavkk/chronicle/blob/main/store/store.go#L98) |
    | [`store/redis/`](https://github.com/adityavkk/chronicle/tree/main/store/redis) | The Redis backend: Lua mutations, key schema, pub/sub | [`Store.Append`](https://github.com/adityavkk/chronicle/blob/main/store/redis/store.go#L523) |

  </Table>

  Two of these are the core, `protocol` and `store`. The rest is shell and
  effects. The boundary between them is the `store.Store` interface. There are
  two implementations of it, Redis and an in-memory store used in tests, and
  that second implementation is what keeps the boundary honest.

  - **[Modules](./modules)** walks each package in turn, with its main function
    and a snippet.
  - **[Tracer bullets](./tracers)** follow three real requests end to end, from
    the HTTP edge down to Redis and back.

  <Callout icon="info" label="Parity">
    The package names and shapes follow the protocol's reference Caddy plugin.
    Chronicle keeps the same interface, the same error variables, and the same
    handler structure, pinned to a recorded upstream commit. Porting an upstream
    change is a file diff, not a redesign.[^caddy]
  </Callout>

<section class="block" id="architecture">
  <SectionHead num="02" label="Architecture" title="A request's <em>path through the layers</em>" />

  The server splits into two parts. The shell is the HTTP handler. It owns
  everything that touches the network: it parses requests, writes responses, and
  decides which store call each verb maps to. The core is the `protocol` and
  `store` packages. It is plain functions over immutable inputs, with no I/O to
  mock. The store backend runs the effects the core decided on.

  <Diagram name="wiki-arch" label="System architecture" caption="A request enters the <strong>shell</strong> (<code>Handler.ServeHTTP</code>), consults the <strong>core</strong> for parsing and validation decisions, and calls the <strong>store.Store</strong> interface. The Redis backend turns each mutation into one Lua script. A durable append also fires the subscription hook into the webhook manager." />

  The diagram reads left to right as control flow. A durable append also fires
  the `OnStreamAppend` hook, which is where the subscription plane picks up.

  <Aside label="Live readers" summary="How a tailing read waits for the next append">
    For a live read the Redis backend, after running the read script, waits on a
    per-stream pub/sub channel. A durable append publishes to that channel inside
    the same Lua script that writes the message, so the wake cannot race the
    write. The default long-poll timeout is 30 seconds (`config.go`).
  </Aside>
</section>

<section class="block" id="shell-core">
  <SectionHead num="03" label="Design" title="Why the <em>shell is thin</em>" />

  The handler never decides protocol questions. It gathers bytes and headers,
  hands them to a pure function, and acts on the answer. The producer idempotency
  rules from spec section 5.2.1 are the clearest case.

<Step n={1} fn="Handler.ServeHTTP" loc="handler.go:124" role="One switch on the HTTP method routes every request to one of five handlers. No protocol decisions happen here.">
```go title="handler.go"
switch r.Method {
case http.MethodPut:
	err = h.handleCreate(w, r, streamPath)
case http.MethodHead:
	err = h.handleHead(w, r, streamPath)
case http.MethodGet:
	err = h.handleRead(w, r, streamPath)
case http.MethodPost:
	err = h.handleAppend(w, r, streamPath)
case http.MethodDelete:
	err = h.handleDelete(w, r, streamPath)
}
```
</Step>
