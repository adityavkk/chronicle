<Tldr>
  <ul>
    <li><strong>One handler, one switch.</strong> Every request lands in <code>Handler.ServeHTTP</code> and dispatches on the HTTP method to one of five handlers. That switch is the whole router.</li>
    <li><strong>Decisions are pure; effects are not.</strong> Everything decidable without I/O — header grammar, offset math, the producer state machine — lives in the <code>protocol</code> and <code>store</code> packages as plain functions. The handler and the Redis backend only carry data to and from those decisions.</li>
    <li><strong>The store is an interface.</strong> The handler talks to <code>store.Store</code>, an 11-method contract. A Redis backend and an in-memory backend implement it; the handler can't tell them apart.</li>
    <li><strong>Every mutation is one Lua script.</strong> Create, append, close, and delete each compile to a single <code>EVALSHA</code> call, so validation, write, and wakeup happen atomically inside Redis.</li>
    <li><strong>Subscriptions hang off a hook.</strong> A durable append fires <code>onStreamAppend</code>; the webhook manager turns that into a signed wake or a pull-wake event, fenced so a crash can't lose or double-deliver it.</li>
  </ul>
</Tldr>

<section class="block" id="what">
  <SectionHead num="01" label="Orientation" title="What this wiki <em>covers</em>" />

  Chronicle implements the [Durable Streams protocol](./../): a stream is a URL,
  and ordinary HTTP verbs create it, append to it, read it, and tail it. The
  [Overview](./../) explains the protocol and the performance envelope. This
  wiki is the other half — the source. It answers a different question: *if you
  opened the repository, where would you look, and what would each piece be
  doing?*

  It is organized the way the code is layered:

  - **This page** sketches the architecture and maps every package to its job.
  - **[Modules](./modules)** walks each package in turn — the imperative shell,
    the pure core, the store contract, the Redis backend, and the subscription
    plane — with the load-bearing function and a snippet from each.
  - **[Tracer bullets](./tracers)** follow three real requests end to end,
    function by function, from the HTTP edge down to Redis and back.

  Every code reference links to the exact lines on GitHub. Nothing here is
  paraphrased from memory; it was read out of the tree at `main`.
</section>

<section class="block" id="architecture">
  <SectionHead num="02" label="Architecture" title="A request's <em>path through the layers</em>" />

  The server is a functional core wrapped in an imperative shell. The shell —
  the HTTP handler — owns everything that touches the network: parsing requests,
  writing responses, and deciding which store call a verb maps to. The core —
  the `protocol` and `store` packages — is plain functions over immutable
  inputs, with no I/O to mock. The store backend executes the effects the core
  decided on.

  <Diagram name="wiki-arch" label="System architecture" caption="A request enters the <strong>imperative shell</strong> (<code>Handler.ServeHTTP</code>), consults the <strong>pure core</strong> for parsing and validation decisions, and calls the <strong>store.Store</strong> interface. The Redis backend turns each mutation into one Lua script; a durable append also fires the subscription hook into the webhook manager." />

  Read the diagram left-to-right as control flow. A client request hits the
  handler. The handler asks the pure core to parse headers and offsets and to
  rule on producer sequencing, then calls one method on the `store.Store`
  interface. The Redis backend behind that interface runs a Lua script against
  Redis 8 — atomic per stream — and, for live readers, waits on a pub/sub
  channel. A durable append also trips the `OnStreamAppend` hook, which is where
  the subscription plane picks up.
</section>

<section class="block" id="shell-core">
  <SectionHead num="03" label="Design" title="Why the <em>shell is thin</em>" />

  The split is deliberate, and it shows up everywhere in the source. The handler
  never decides protocol questions; it gathers bytes and headers, hands them to
  a pure function, and acts on the answer. The producer-idempotency rules are
  the clearest case: spec section 5.2.1 is transcribed once, into
  `ValidateProducer`, a function that maps a producer's stored state and an
  incoming epoch/sequence pair to one of five outcomes. The handler calls it.
  The Lua script that enforces the same rules inside Redis mirrors it line for
  line, with the Go function as its test oracle.

  The payoff is testability and parity. The core is tested without a running
  Redis. The backend is swappable — the same handler runs against an in-memory
  store in unit tests and Redis in production. The cost is indirection: to
  follow a single append you cross four files, which is exactly what the
  [tracer bullets](./tracers) are for.
</section>

  Two of these — `protocol` and `store` — are the pure core; the rest is shell
  and effects. The boundary between them is the `store.Store` interface, and the
  fact that there are two implementations of it (Redis, and an in-memory store
  used in tests) is what keeps the boundary honest.

  <Callout icon="info" label="Parity">
    The package names and shapes are not arbitrary. Chronicle mirrors the
    protocol's reference Caddy plugin — same interface, same error variables,
    same handler structure — pinned to a recorded upstream commit, so porting an
    upstream change is a file diff rather than a redesign.[^caddy]
  </Callout>
