<Callout icon="alert" label="What this proves, and what it does not">
  The slot-homed, work-sharded design is built and passes locally. PR #18 landed
  the whole horizontal-scale epic on main. Subscriptions are slot-homed into 256
  `{__ds:h}` slots, ownership is leased over a `{ownership}` keyspace, and the
  claim is split per `(subId, g)`. Its safety and liveness suite passes on a real
  Chronicle plus Redis stack: T1 to T5 GREEN, L1 and L3 GREEN, and the contention
  suite C1 to C3 with doc-05 gate #6 GREEN. The cloud rig deployed that same
  post-epic binary, so the numbers on this page measure the sharded design and not
  a pre-sharding baseline. The one thing the cloud rig could not exercise is the
  multi-shard layout. It ran on a single-node Memorystore BASIC, which has one
  hash-slot range on one node, so the slots never spanned nodes. The measured
  gates therefore show that the sharded binary on a single-slot substrate produces
  no fence rejections as replicas grow, holds its sweep at 10,000 subscriptions,
  and keeps its claim lease linearizable under contention. They do **not** yet
  carry the cross-node, multi-shard fan-out number, and they do not re-prove the 12
  replica point. Those two gates, plus the L2, L4, and L5 multi-node liveness
  runs, stayed unmeasured on cloud because GCP project quota blocked the 3-shard
  Redis Cluster and the larger node pool. See [the limits](#limits) for exactly
  which, and why.
</Callout>

<section class="block" id="goal">
  <SectionHead num="01" label="The goal" title="What we set out to <em>prove</em>" />

  Chronicle is one Go binary that serves durable streams and runs the
  subscription control plane. The open question was whether adding replicas
  helps the control plane or hurts it. A prior load test answered "hurts". In
  that test the agents runtime drove chronicle as its Durable-Streams backend.

  At 6 replicas the wake path was clean. At 12 it failed. The warmup on the load
  generators produced 489 to 735 FENCED rejections per pod, and only about 40
  percent of entities ever woke, while every tier sat at 12 percent CPU or less.
  Three documents state the numbers identically: research doc 05
  (`05-proposed-architecture.md` lines 92 to 93), research doc 07
  (`07-jepsen-style-verification.md` lines 75 to 77), and the epic #9 objective
  on GitHub.

  <Table>
    | Tier | CPU at failure |
    | --- | --- |
    | chronicle | 4% |
    | Redis | 4% |
    | Postgres | 12% |
  </Table>

  The cause was not capacity and not the recovery sweep. The agents runtime
  registers one subscription per entity *type*, named `"<type>-handler"`, and
  chronicle holds exactly one single-holder lease per subscription. So every
  entity of a type and every replica hosting one contend for that one lease.
  More replicas means more contenders on one lock. The queue on the single
  control-plane slot grows longer, heartbeats land late, leases lapse, takeovers
  fence the deposed holder's in-flight work, and the loser retries, which adds
  more operations. Research doc 05 (lines 91 to 96) names the binding constraint
  as lock contention on a single subscription's lease. It adds that 2.5 times the
  CPU made it worse, which is the signature of a contention failure and not a
  saturation one.

  The job of the rig was to show the inverse on real cloud infrastructure: that
  adding replicas produces no fence rejections, that the state and sweep cost
  stay inside budget at scale, and that the claim lease stays correct under load
  and under faults.

  <Aside label="Provenance" summary="Where the 489 to 735 FENCED figure comes from">
    The 489 to 735 FENCED figure, the roughly 40 percent woke, and the 12 percent
    CPU come from research doc 05 (line 96), which cites two Electric-rig files
    that were not in the read set for this page. They were verified only as
    restated consistently across the three documents above, not opened at the
    primary source. Research doc 05 (line 109) notes that the exact per-wake
    interleaving at 12 replicas is inferred from the proven halves plus the
    rejection counts.
  </Aside>
</section>

  Two configuration choices drive the results. First, several runs used 1 CPU
  per pod instead of the 2 CPU default, either to fit cost-minimal e2-standard-2
  nodes or to fit 12 replicas on three e2-standard-4 nodes at 12 vCPU total.
  Section 04 shows one result that flips between fail and pass on pod CPU alone.
  Second, every successful Redis phase ran on Memorystore BASIC, which is a
  single node with no sharding and no managed failover. A single-node Redis has
  one hash slot range on one node, so the cross-node maximum round-trip time that
  a sharded cluster would show was never available for those phases.

<section class="block" id="rig">
  <SectionHead num="03" label="The rig" title="Provision, five phases, <em>teardown to zero</em>" />

  A run is a single shell script, `run-gate2.sh`. It provisions cloud
  infrastructure, drives five phases of load, and then destroys everything. The
  load-bearing line is the teardown trap, because a stranded GKE cluster keeps
  billing.

  <Step n={1} fn="teardown_all" loc="loadtest/run-gate2.sh:84" role="A trap on EXIT, INT, and TERM destroys the GKE cluster and both Redis instances on every exit path, then checks no cluster remains. The meter never runs past the test.">

  ```bash title="run-gate2.sh"
  trap teardown_all EXIT INT TERM
  ```

  </Step>

  The trap runs a Terraform destroy of the GKE cluster and both the standalone
  and cluster Redis instances, then checks that no clusters remain in the region
  (`run-gate2.sh` lines 48 to 83). The cost estimate is about 3 dollars per hour,
  with a total under 80 dollars (`run-gate2.sh` line 18). Phase 5 uses a
  self-restarting port-forward loop because the Jepsen nemesis kills pods, which
  would otherwise break a static port-forward (`run-gate2.sh` lines 108 to 125).

  Provisioning builds the container images for `linux/amd64` through Cloud Build
  into Artifact Registry, then applies the cluster and Redis with Terraform in
  `loadtest/terraform` (`run-gate2.sh` lines 29 and 185 to 191).[^apis]

  The five phases run in order, each driven by a purpose-built Go load
  generator (`run-gate2.sh` lines 11 to 16):

  <Table wide>
    | Phase | What it drives | Driver |
    | --- | --- | --- |
    | 1 | Horizontal-scale curve: append fan-out p99 as replicas grow | fanoutscale |
    | 2 | Claim contention: single-holder linearizability at higher worker counts | jepsen single-holder-linz |
    | 3 | Gate #2 cross-node fan-out on a Redis Cluster | fanoutscale |
    | 4 | Recovery sweep at K=10,000 subscriptions | sweepscale |
    | 5 | Jepsen correctness under pod and Redis kills | jepsen-checker |
  </Table>

  A fan-out phase drives appends and scrapes `chronicle_fanout_seconds` from
  `/metrics`. A sweep phase seeds K subscriptions and scrapes the
  `chronicle_sweep_*` metrics.[^drivers]

  <Switch name="phase-driver" label="How a phase is driven" tabs={["fan-out (phase 1, 3)", "sweep (phase 4)"]}>
    <div data-panel="0">

  ```bash title="fanoutscale"
  fanoutscale -subs 8 -rate 5/s -warmup 20s -measure 60s \
    -base-url http://localhost:4437 -metrics-url http://localhost:9090/metrics
  ```

    </div>
    <div data-panel="1">

  ```bash title="sweepscale"
  sweepscale -scenario scenarios/sweep-scale.yaml \
    -base-url http://localhost:4437 -metrics-url http://localhost:9090/metrics
  ```

    </div>
  </Switch>

  <Callout icon="info" label="Teardown confirmed at zero ongoing cost">
    Every campaign confirmed teardown left nothing running. The gate2 trap
    finished its destroy (`run-gate2.sh` lines 81 to 82). The claude-fleet run
    recorded clusters none, Memorystore none, and zero dollars ongoing, checked
    with `gcloud container clusters list` and `gcloud redis instances list`
    against its `-claude`-suffixed resources. The codex run recorded the same
    zero-ongoing result for its `-codex`-suffixed resources.
  </Callout>

<section class="block" id="results">
  <SectionHead num="04" label="The results" title="Flat fan-out, a sweep floor, <em>and a contention point</em>" />

  Three measured gates passed on the cluster `chronicle-gate2` on 2026-06-22,
  recorded in `loadtest/RESULTS-gate2.md`. They are the inverse of the original
  failure. A fourth gate, the one that would have been the most decisive, did
  not run.

  ### The headline: a flat fan-out curve

  Phase 1 is the regression test for the fence rejections. It ramps replicas
  through 1, 2, 4, and 8, with 4 pull-wake subscriptions, an append rate of 10
  per second, a 20 second warmup, and a 60 second measurement window,
  port-forwarded to the first pod for each replica count (`RESULTS-gate2.md`
  lines 13 to 28). The append-to-wake p99 is flat.

  <Table wide>
    | Replicas N | fanout p99 (ms) | fanout p50 (ms) | subs per fanout | append errors | Verdict |
    | --- | --- | --- | --- | --- | --- |
    | 1 | 12.48 | 5.20 | 4 | 0 | PASS |
    | 2 | 12.61 | 5.59 | 4 | 0 | PASS |
    | 4 | 12.58 | 5.45 | 4 | 0 | PASS |
    | 8 | 12.53 | 5.29 | 4 | 0 | PASS |
  </Table>

  The p99 holds at 12.5 milliseconds plus or minus 0.1 across the range, a
  spread of 0.13 milliseconds while the replica count quadruples, with zero
  append errors at every count. If the single control-plane slot were still
  serializing under concurrency, the p99 would climb as replicas grow. It does
  not. The original failure showed fence rejections that grew with more replicas
  and grew further with more CPU. Here the curve is flat.

  The absolute number is not meaningful and the failing replica count was not
  reached. The p99 includes about 10 milliseconds of port-forward round-trip
  time, so the file's claim is about flatness across N and not the absolute 12.5
  milliseconds (`RESULTS-gate2.md` lines 16 to 17). Phase 1 also used 1 CPU per
  pod, not the 2 CPU default, so that 12 replicas could fit on the available
  nodes. So it measures relative latency against N and not absolute throughput.
  And N=12, the exact count that originally failed, was not run, which section 06
  covers.

  ### The K=10,000 regression floor

  Phase 4 measures the recovery sweep at 10,000 subscriptions with 5 links each,
  at the fair node size of 2 CPU on e2-standard-4-class capacity
  (`RESULTS-gate2.md` lines 53 to 60). The limit is a sweep tick p99 under 1500
  milliseconds.

  <Table wide>
    | Run | CPU | K | Links per sub | sweep p50 (ms) | sweep p99 (ms) | Verdict |
    | --- | --- | --- | --- | --- | --- | --- |
    | Sweep-smoke (single link) | 1 | 10,000 | 1 | 384 | 509 | PASS |
    | Claude-fleet | 1 | 10,000 | 5 | 1536 | 2037.8 | FAIL (confounded) |
    | Gate2 Phase 4 (fair) | 2 | 10,000 | 5 | 768 | 1019 | PASS |
  </Table>

  <Complexity
    time="O(N·K) per tick"
    space="O(1) streamed"
    why="Every replica sweeps all K subscriptions each tick, so the cost is CPU-bound on the per-tick read. Shrinking a pod from 2 CPU to 1 CPU is what flipped this gate from pass to fail, not a code change."
  />

  The fair re-run passed at 1019 milliseconds. It seeded all 10,000
  subscriptions in 4.1 seconds with zero seed errors and ran 60 sweep ticks in
  120 seconds. The claude-fleet run failed the same gate at 2037.8 milliseconds,
  but that run had been shrunk to 1 CPU to fit cost-minimal nodes, so the failure
  was the sizing and not a regression (`RESULTS-gate2.md` lines 53 to 60 and 94).
  The earlier 509 millisecond single-link result from the first shakeout is the
  standing baseline that every later campaign cites (`RESULTS-gke.md` lines 7 to
  46). It used 1 link per subscription while the gate uses 5. So the 1019
  millisecond figure is the right like-for-like result at 5 links, and the 509
  millisecond figure is a lighter single-link control.

  ### The contention point

  Phase 2 checks that the single-holder claim lease stays linearizable as the
  number of concurrent claim workers rises. It uses the jepsen
  single-holder-linz checker over a 12,000 millisecond workload
  (`RESULTS-gate2.md` lines 30 to 37).

  <Table wide>
    | Workers W | Operations | Claims | Granted | Linearizable | Verdict |
    | --- | --- | --- | --- | --- | --- |
    | 4 | 451 | 417 | 34 | yes | PASS |
    | 16 | 1267 | 1225 | 42 | yes | PASS |
  </Table>

  The claim is linearizable at both 4 and 16 workers. This is the G=1 point, the
  claim granularity of one lease per type, which is the granularity that
  originally failed. It measures the single-lease G=1 design at higher worker
  counts on one healthy node, so on its own it does not show the worker count at
  which throughput stops scaling. The finer-grained claim fix the original
  failure calls for is built and passes locally. The per-(subId,g) claim split
  lands in `webhook/shard.go` (FNV-1a, default G=16), and the contention suite C1
  to C3 with doc-05 gate #6 is GREEN on a real Chronicle plus Redis stack, so the
  contention point that broke G=1 at N=12 moved past the whole G=16 ramp.[^c16]
  What stays cloud-pending is the multi-node reproduction of that point under
  scale, blocked by quota and not by the mechanism.

  ### Gate #2: the number that was never measured

  Phase 3 was meant to be the deciding gate. It would measure the append fan-out
  p99 at 2, 4, 8, and 256 subscribers on a 3-shard Memorystore Cluster, which is
  the sharded substrate where slots span nodes and a cross-node maximum
  round-trip time appears. It did not run.

  <Table wide>
    | Subscribers S | Substrate | fan-out p99 | Verdict |
    | --- | --- | --- | --- |
    | 2 | Memorystore Cluster (3 shards) | not measured | SKIPPED |
    | 4 | Memorystore Cluster (3 shards) | not measured | SKIPPED |
    | 8 | Memorystore Cluster (3 shards) | not measured | SKIPPED |
    | 256 | Memorystore Cluster (3 shards) | not measured | SKIPPED |
  </Table>

  The Memorystore Cluster never reached a ready state. Provisioning failed with
  an insufficient project quota error on every attempt, twice. Each time it
  entered a create-then-delete loop and never became ready (`RESULTS-gate2.md`
  lines 40 to 51 and 93). The standalone Redis phases were not affected. This is
  the most important unmeasured number across all campaigns. The cross-node
  fan-out latency decides whether the built slot-homing layout is affordable to
  keep enabled in production, and Memorystore BASIC is single-shard so its slots
  never span nodes. The fan-out mechanism itself is built and proven correct
  locally, so gate #2 settles enable-versus-defer for a design that already
  exists and is already correct. Section 06 records this as a hard limit on the
  claim.

  <Aside label="Why the mechanism is still trusted" summary="The slot-isolation differential (T5) is GREEN locally">
    T5, the slot-isolation differential checker (jepsen/checker
    `scenario_slot.go`), is GREEN: 320 subscriptions spread across 204 of 256
    slots, the scatter-gather subscriber set equal to the independent reference
    and to the brute-force union, zero foreign wakes, and a mis-tag detected as
    CROSSSLOT. What single-node Redis removes is the max-node round-trip time,
    which is the only thing gate #2 reads.
  </Aside>
</section>

<section class="block" id="faults">
  <SectionHead num="05" label="Under faults" title="Killed mid-flight, <em>and a bug found</em>" />

  Phase 5 ran a Jepsen nemesis against 2 replicas. The nemesis killed origins,
  and a self-restarting port-forward let a second pod serve traffic while one was
  killed (`RESULTS-gate2.md` lines 62 to 99). It ran three scenarios that work
  without a cluster-to-host webhook back-channel, which the GKE network did not
  allow. Two passed and one failed.

  <Table wide>
    | Scenario | What it checks | Result |
    | --- | --- | --- |
    | expired-lease-takeover | a deposed worker's late ack is fenced | PASS |
    | single-holder-linz | the claim lease is linearizable under a GC-pause nemesis | PASS |
    | pull-wake-arm-crash | a pull-wake cursor recovers after an origin crash | FAIL, then fixed |
  </Table>

  The lease fence passed cleanly. In expired-lease-takeover, Worker A claimed
  generation 1, stalled 2.5 seconds past its lease time-to-live, Worker B took
  over at generation 2, and Worker A's late ack returned a 409 FENCED instead of
  a silent success. In single-holder-linz at 4 workers, 440 operations were
  linearizable. So the fence rotates correctly on a stale-lease takeover, and the
  claim is linearizable under fault injection.

  The third scenario found a real bug. In pull-wake-arm-crash the driver appended
  120 messages across 6 streams, all 120 reachable in Redis. Then the nemesis
  killed all origins once and killed one origin twelve more times. After a 120
  second settle the subscription cursor was stuck at offset `000000000150` for
  all 6 streams while the stream tail had advanced to `000000000293`
  (`RESULTS-gate2.md` lines 80 to 87). About 143 messages were never acked. The
  root cause was that the recovery sweep did not re-emit pull-wake wake events
  after an origin restart. Pull-wake arms no lease, so the lease-expiry path never
  fires. A subscription stuck in the waking phase with a durably emitted but
  unclaimed wake had no way out, so it stayed stuck and later appends only
  coalesced against it.

  The bug was fixed in commit `5b99bb9` and re-verified. The fix adds an
  emit-without-claim recovery block to `sweepOnce` in `webhook/manager.go`
  (lines 1185 to 1202). When a pull-wake subscription has been in the waking
  phase with a sent wake for longer than three sweep intervals, the sweep
  re-emits the same generation and wake id, and a duplicate event with the same
  fence token is claim-safe. The fix also corrects the jepsen checker, whose drain worker had
  sent zero stream acks, so cursors never advanced. The reproduce command is the
  one recorded in the results file (`RESULTS-gate2.md` line 86):

  ```bash title="Reproduce the pull-wake recovery bug"
  jepsen-checker -scenario pull-wake-arm-crash -settle 120s
  ```

  The re-run passed with 6 of 6 streams reaching their tail under 47 nemesis
  actions, with no regression on the two scenarios that already passed and with
  the full test suite green.

  <Callout icon="alert" label="The results file shows this as FAIL">
    `RESULTS-gate2.md` was committed before the fix landed and was never updated,
    so it still records pull-wake-arm-crash as FAIL. The re-verified pass lives in
    the commit message for `5b99bb9` and in the closing comments of GitHub issue
    #24, roughly one hour later, not in the results file. Anyone reading only the
    results file would see the bug marked open.
  </Callout>
</section>

<section class="block" id="limits">
  <SectionHead num="06" label="The limits" title="What quota <em>blocked</em>" />

  Two of the numbers this campaign set out to measure were never measured,
  because the GCP project ran out of quota. These are measurement gaps, things
  that did not run, not observed defects. They bound the claim. The plain
  statement is that the no-fence-rejection verdict rests on a flat trend through
  8 replicas, not on a passing run at the replica count that originally failed.

  <Table wide>
    | Gap | What was blocked | Why | What it bounds |
    | --- | --- | --- | --- |
    | N=12 not re-run | the 12 replica point on the scale curve | insufficient project quota on the SUT node pool | the curve is flat 1 to 8, and the count that failed was not re-tested |
    | Gate #2 multi-shard fan-out skipped | cross-node fan-out p99 at S=2, 4, 8, 256 | the Redis Cluster never reached ready under insufficient project quota | the viability of spreading subscriptions across slots is unproven |
  </Table>

  The first gap is the direct one. The original failure was at 12 replicas, and
  Phase 1 measured only 1 through 8 because GCP returned an insufficient project
  quota error on the SUT node pool (`RESULTS-gate2.md` line 28). The script does
  attempt 12, but the pool was exhausted, so the SUT pods were rescheduled onto
  the loadgen and observation nodes for that run. The file's own claim is that
  the trend through 8 is conclusive, not that 12 passed.

  The second gap is the deeper one. Gate #2, the cross-node fan-out latency on a
  sharded cluster, is described in the research as the number that decides
  whether the built slot-homing layout stays enabled in production or is deferred
  to the single-slot ownership build. It has no measured value in any campaign,
  on any rig, because no run ever stood up a multi-shard Redis. The claude-fleet
  run could not produce the fan-out metric at all because its driver seeded
  subscriptions but drove no appends. The codex run reached an S=256 result from
  only 20 observations rather than a full 100 appends, which is not a valid gate.
  So the headline cross-node fan-out p99 stays unknown. The slot-homing it
  measures is already built and locally correct (T5 GREEN), so this is a missing
  production-readiness number on a shipped mechanism, not a test of one that does
  not exist.

  Other gaps follow from the same constraint. The webhook-based Jepsen scenarios,
  baseline and origin-restart, were skipped because they need a cluster-to-host
  back-channel the GKE network did not provide (`RESULTS-gate2.md` lines 69 to
  70). Gates #3, #4, and #5 test mechanisms that exist on main and pass locally,
  but their cloud numbers were not measured for the same quota reason.[^gates] The
  recommended follow-up is to request a quota increase for `redis.googleapis.com`
  in `adityavkk-prototyping`, then re-run Phase 3 in isolation, and to re-run
  Phase 1 with node pool quota for 12 replicas.
