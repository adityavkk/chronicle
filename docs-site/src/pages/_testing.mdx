<section class="block" id="philosophy">
  <SectionHead num="01" label="Philosophy" title="A pure core behind an <em>imperative shell</em>" />

  Chronicle splits every component into a pure deterministic core and an
  imperative I/O shell, then tests each at the cheapest level that is still
  faithful.[^split] The core is a function over values, e.g. `ValidateProducer`,
  the offset and frame codecs, the webhook fence/claim/ack mirrors, the porcupine
  fence model, and the cursor checker. It has no Redis, no clock, and no cluster.
  The shell is where the bytes move: the Lua scripts, the pub/sub wakeups, the
  durable hashes, and the lease ZSETs. The five layers below name what runs at
  each level, the faults it sees, and the infra it needs.

  <Table>
    | Layer | What runs | Faults | Infra |
    | --- | --- | --- | --- |
    | **Pure core** | unit tables, porcupine model, cursor checker | none, deterministic | none |
    | **I/O shell** | store / webhook integration, the Lua | timing, durability across a fresh store | real Redis (db 15 / 14) |
    | **HTTP surface** | `handler_test.go` via `httptest` | concurrency via goroutine + sleep | in-memory store |
    | **Cross-process** | jepsen scenarios, porcupine histories | pod kill, Redis kill, gcPause | live `k3d` cluster |
    | **Wire contract** | conformance (vitest + fast-check) | property fuzzing | black-box vs a live server |
  </Table>

  The pyramid is intentional. Each load-bearing rule is pinned more than once,
  e.g. `ValidateProducer` from both backends and the fence from Go, Lua, and the
  checker, so a regression has to defeat two independent assertions to slip
  through.

  <Callout icon="info" label="Why the split pays off here">
    The most safety-critical logic in Chronicle is the producer fence, the
    subscription generation fence, and the forward-only cursor. That logic is the
    hardest to test against a live system, because the cases that matter are rare
    races. Lifting it into a pure core turns the rare case into a hand-built table
    row, e.g. a stale bearer token where <code>reqGen</code> matches but
    <code>tokenGen</code> is stale, or a post-expiry <code>FENCED</code> that must
    still linearize. The live drivers cannot reliably reach those states. The unit
    tests reach them by construction.
  </Callout>
</section>

<section class="block" id="store">
  <SectionHead num="02" label="Store" title="The durable log, <em>two backends</em>" />

  The store layer is the append-only durable streams log behind one `Store`
  interface, with two backends: an in-process `MemoryStore` in pure Go, and a
  Redis-backed store built on ZSET frames, hash-tagged keys, and Lua. The pure
  half needs no Redis. The Redis half needs a real server, gated by `-short` and a
  one-time `FlushDB`.[^realredis] Two tests carry the layer and lead here. The
  reference index of all eleven groups follows.

  <Step n={1} fn="TestFrameLexOrderEqualsStreamOrder" loc="store/redis/keys_test.go:140" role="The foundation under the whole ZSET model is one property: bytewise frame order must equal stream offset order for any payload bytes. A pure lexicographic sort must still come out in offset order.">

  ```go title="store/redis/keys_test.go"

  }
  sort.Strings(frames)                 // pure lexicographic sort
  // => 5, 7, 50, 100  — offset order, independent of payload bytes
  ```

  </Step>

  <Callout icon="alert" label="What the store layer does not test">
    Despite the repository name, the store layer has no linearizability, no
    Jepsen-style, and no <code>go test fuzz</code> coverage. Offset and frame
    coverage is example-based tables, not generative. There is no concurrent
    multi-writer race test and no <code>-race</code>-specific store test here.
    Memory-versus-Redis parity is <em>structural</em>. Both back the same
    <code>Store</code> interface and share the producer table, but there is no
    single harness. Most contract cases are duplicated per backend, so catching a
    divergence relies on hand-kept mirroring.
  </Callout>

<section class="block" id="http">
  <SectionHead num="03" label="HTTP" title="The protocol surface, <em>in memory</em>" />

  The HTTP layer is `handler_test.go` with 57 functions, plus the pure protocol
  helpers in `protocol/protocol_test.go`. Together they cover the stream wire
  surface: PUT create and fork, HEAD, POST append and close, idempotent producers,
  DELETE and soft-delete, GET catch-up and conditional reads, JSON-mode
  flattening, long-poll, SSE framing, CORS and security headers, `Mount` prefix
  translation, and the TTL, sub-offset, and cursor codecs. Every handler test runs
  against an in-memory `store.NewMemoryStore` through a shared `do()` helper that
  returns a `*httptest.ResponseRecorder`. There is no real Redis, no miniredis, and
  no mocks in this layer.[^httpharness]

    Error envelopes are plain-text <code>http.Error</code> bodies, not a JSON
    problem shape. Tests assert status codes and diagnostic headers but never the
    body text, so the wording can drift untested. Glob and wildcard matching is not
    tested here, because it lives in the webhook package, and <code>__ds</code>
    subscriptions only reach a <code>501</code> stub. There is no fuzz, property, or
    linearizability test in this area. Negative-input coverage is hand-enumerated
    table slices, so an untried malformed input may slip through.
  </Callout>

<section class="block" id="subscriptions">
  <SectionHead num="04" label="Subscriptions" title="The control plane, <em>three tiers</em>" />

  The `__ds` subscription layer turns stream activity into webhook deliveries and
  pull-wakes. Its tests come in three tiers. The first tier is pure unit tests on
  functions with injected seams: the fence/claim/ack mirrors of the Lua, the
  pending-work core, config hashing, glob, retry backoff, and the SSRF URL
  classifier. They are table-driven and deterministic because `now` and the jitter
  fraction are passed in explicitly. The second tier is real-Redis integration
  tests (db 14, `FlushDB` at setup, `t.Skip` under `-short`) that drive the
  embedded Lua scripts and prove durability across a *fresh store over the same
  Redis*. The third tier is Manager tests with in-memory fakes (`fakeStreams`,
  `fakeLister`, `fakeMetrics`) over a real store.

  The main technique in this layer is differential parity by convention. Each
  pure mirror in `state.go` carries a doc comment naming the Lua file it must move
  with, and the fence rule rejects an ack unless the token generation, the request
  generation, and the request wake all equal current state.

  ```go title="webhook/pure_test.go — the Go mirror of common.lua fenced()"
  // mirror of: fenced() = tok ~= cur or req ~= cur or wake == '' or wake ~= cur
  d := FenceDecision(sub{gen: 7, wake: "w_abc"}, reqGen, reqWake, tokGen)
  // all three equal current      -> ""            (accepted)
  // any stale gen / empty wake    -> ErrCodeFenced (rejected)
  ```

  The crypto tests re-derive the `ed25519.PublicKey` from the JWK and verify over
  the `timestamp.body` string the same way the conformance receiver does, so
  producer and consumer agree on the wire contract the external suite checks.

  <Callout icon="alert" label="Where the convention can drift">
    Parity here is asserted by convention, not enforced. No test
    <code>EVAL</code>s the Lua and runs the Go mirror on identical inputs to
    assert byte-equal output, so the two sides can diverge without warning. True
    concurrency is simulated in sequence. Two-worker races are reproduced by
    advancing a clock, not by running in parallel. Only the jepsen
    single-holder-linz driver runs concurrent claims and acks. The SSRF
    resolve path is exercised only with IP literals, so DNS rebinding and
    multi-IP hosts are not table-covered.
  </Callout>

<section class="block" id="jepsen">
  <SectionHead num="05" label="Jepsen" title="Killed mid-flight, <em>and linearized</em>" />

  The Jepsen suite (`jepsen/checker/`) is a Go generator, nemesis, and checker
  triad over a live `k3d` cluster. It has two halves. The first is final-state
  durability. It kills pods and Redis, lets the faults settle, and asserts every
  cursor reached its tail or that a deposed worker is `FENCED`. The second is a
  per-history safety suite. It uses porcupine linearizability for the single-holder
  invariant (T1) and a non-decreasing checker for cursor monotonicity (T2). Both
  are split into a pure core and an imperative shell, with the pure cores
  unit-tested on hand-built histories and no cluster.[^jepsen-split]

  <Table wide>
    | Test group | Verifies | Code ref |
    | --- | --- | --- |
    | Durability final-state (baseline / origin-restart / redis-restart) | after faults settle, every glob webhook cursor reached its stream tail; duplicates reported (at-least-once) but fence-safe, not failures | [^jep-durability] |
    | Crash-window hardening (arm-crash / glob-create-crash / index-repair) | each hardening slice's property: the sweep re-emits arm-without-emit wakes, the reconcile loop links a stream created under a crash, `ReconcileIndexes` rebuilds a dropped SADD | [^jep-hardening] |
    | Expired-lease-takeover (deterministic) | worker B's takeover rotates the fence to a fresh generation, so A's late ack returns `409 FENCED` — the single-holder property, hand-built | [^jep-takeover] |
    | **T1** single-holder-linz — live porcupine driver | N workers claim/ack under a gcPause nemesis; each outcome is bracketed into a `porcupine.Operation` and `CheckOperationsVerbose` runs the model | [^jep-t1live] |
    | **T1** lease-fence model — the pure oracle | the sequential model of one sub's `(gen, wake, phase)` register; an `OK` ack under a stale token is the two-holder violation | [^jep-t1model] |
    | **T1** model unit tests — legal + violation witnesses | the oracle accepts every legal interleaving and rejects exactly the single-holder violations, from hand-built porcupine histories | [^jep-t1unit] |
    | **T2** cursor-monotonic — live sampling driver | under origin churn, a poller samples each cursor every 150ms; the pure checker reports any backward step after settle | [^jep-t2live] |
    | **T2** cursor-monotonic pure checker + unit tests | per `(sub, path)` the acked offset never decreases; a dip-and-recover reports exactly once; the `-1` sentinel orders below any real offset | [^jep-t2unit] |
  </Table>

  ### The porcupine model for <em>T1 and T2</em>

  The fence is the safety property the whole suite rests on: at no instant may two
  workers hold a token that `ack.lua` will accept for one subscription. The model
  is a pure I/O-free sequential oracle over one subscription's fence register
  `(gen, wake, phase)`, partitioned by subscription so the linearizability search,
  which is NP-hard, stays per-key.

  <Step n={1} fn="stepAckOrRelease" loc="jepsen/checker/model_fence.go:215" role="The load-bearing safety gate. An accepted ack must carry the current fence. An <code>OK</code> under a stale token is two holders at once, and porcupine surfaces the witness.">

  ```go title="jepsen/checker/model_fence.go"
  func stepAckOrRelease(s fenceState, in fenceInput, out fenceOutput) (bool, fenceState) {
      fenced := in.tokenGen != s.gen || in.reqGen != s.gen ||
                in.reqWake == "" || in.reqWake != s.wake
      if out.status == statusOK && fenced {
          // two holders' tokens were both honored — no valid linearization.
          return false, s
      }
      // ... apply a legal OK forward; BUSY / NOSUB / FENCED are no-ops
  }
  ```

  </Step>

  <Aside label="Model detail" summary="Two design choices that a review found load-bearing">
    Time is *absent* by design. `BUSY`, `NOSUB`, and `FENCED` are unconditional
    legal no-ops, so a `FENCED` grants nothing and cannot be half of a violation.
    Modeling it this way also absorbs `expire_lease.lua` clearing `wake_id` without
    rotating `gen`, a server event the history cannot see. Generations start at 1
    in the unit tests to mirror `create_sub.lua`'s `gen=0` plus the first `HINCRBY`.
  </Aside>

  The pure model is exercised two ways. The live driver runs N real workers against
  a real subscription, with an in-process `gcPause` nemesis that stalls about every
  third grant past `lease_ttl_ms` so a peer takes over and the stalled ack races in
  stale. The recorder brackets each outcome and records `BUSY` and `FENCED`
  verbatim, not as errors. It uses the host monotonic clock so `[Call, Return]`
  stays sound under clock skew.

  ```text title="scenario_lease.go — the live T1 driver, in shape"
  N workers, until deadline:
    claim  -> record(opClaim)
    if grants % 3 == 0 { sleep(lease_ttl + 250ms) }   # gcPause: stall past the lease
    ack    -> record(opAck)                            # the stalled one races in stale

  result, info := porcupine.CheckOperationsVerbose(leaseModel(), history, 20s)
  switch result {
    Ok:      pass
    Illegal: VisualizePath(info, "linz-counterexample.html")  # two-holder witness
    Unknown: too concurrent to decide  -> accepted T1 limit (gap #1)
  }
  ```

  The 13 model unit tests pin clauses the live driver cannot reliably reach, e.g.
  a stale bearer token where `reqGen == current` but `tokenGen` is stale, or the
  post-expiry `FENCED` that must linearize. They run with `CheckOperations`, which
  has no timeout and so returns a definitive `Ok` or `Illegal`, with no cluster and
  no Redis.

  ```text title="model_fence_test.go — a stale ack accepted is Illegal"
  h := history{
    claim(A) -> CLAIMED(gen=1, wA),
    claim(B) -> CLAIMED(gen=2, wB),     # B rotated the fence to 2
    ack(A, gen=1, wA, done) -> OK,      # A's token is stale; an OK here is two-holder
  }
  checkLinearizable(t, h, /*want*/ false)   # no valid linearization exists
  ```

  ### A takeover history, <em>step by step</em>

  The single-holder violation T1 exists to catch is the lease takeover that fails
  to rotate the fence.

  <Aside label="Worked example" summary="The takeover the fence must rotate, broken vs fixed">

  ```text title="Expired-lease takeover — the fence must rotate"
  time ─────────────────────────────────────────────────────────────────▶

  worker A   claim ─────[ holds lease, gen=1, wake=wA ]───X stalled past TTL
                                                          │
                                    lease expires ........│
                                                          ▼
  worker B                              claim (takeover) ─[ gen=?, wake=wB ]── ack ─▶ OK
                                                          │
  worker A (resumes) ............................. ack(gen=1, wA) ─▶ ???

   ┌─ BROKEN (pre-slice): B reuses gen=1, wake=wA ───────────────────────────┐
   │   A's ack(gen=1, wA)  matches current  ->  OK   ← two holders, split brain│
   └──────────────────────────────────────────────────────────────────────────┘

   ┌─ FIXED: B rotates to gen=2, wake=wB ────────────────────────────────────┐
   │   A's ack(gen=1, wA)  tokenGen 1 != cur 2  ->  409 FENCED  ← single holder │
   └──────────────────────────────────────────────────────────────────────────┘
  ```

  </Aside>

  T2 is simpler by construction. A single backward step in a cursor is a finite
  counterexample, so it needs no linearization search. It is a direct
  non-decreasing checker over the sampled stream. `CheckCursorMonotonic` keeps a
  per-`(sub, path)` high-water mark. A sample below the mark is one violation, and
  it keeps the existing mark, so a dip and recover reports exactly once rather than
  again on every later sample. Its `offsetGreater` mirrors `common.lua` and the
  webhook a third time, the same rule pinned from three sides, with `-1` and `""`
  as the beginning sentinel.

  ```text title="check_cursor.go — a dip-and-recover is one report, not a cascade"
  samples for one (sub, path):  20 ─▶ 5 ─▶ 21
                                │     │     │
  high-water:                   20    20    20→21
  violation:                          ✗ (5 < 20)
                                            (21 ≥ 20: not re-flagged)
  => exactly 1 violation, not 2
  ```

  <Callout icon="alert" label="What Jepsen does not yet cover">
    <strong>The jepsen pod-kill still approximates the arm-crash, but an
    in-process seam now hits it exactly.</strong> The arm-to-emit window of a few
    microseconds cannot be hit from the external host driver, so the jepsen
    scenario stands in with a coarse pod-kill plus churn, and the checker notes it
    lacks a server-side seam. An in-process gofail-style failpoint now closes the
    gap directly. <code>webhook/failpoint.go</code> crashes at the exact
    arm-to-emit point and the recovery sweep then re-emits the stranded wake (doc
    07 honest-gap #2).
    <strong>The local failover substrate is still single-node.</strong>
    <code>killRedis</code> deletes the one Redis pod and relies on PVC-backed AOF
    replay. A STANDARD_HA substrate with a replica and a promotion drill is now
    built (<code>jepsen/deploy/standard-ha.yaml</code> and
    <code>standard-ha-failover.sh</code>), but its run is cloud-pending, so
    durability under a two-node split and under replication lag is specified, not
    yet measured. The T3 ownership-exclusivity and T5 slot-isolation acceptance
    gates are now built and green, checked by the porcupine CAS shard model and
    the slot differential checker on live local-Redis drivers. The liveness gates
    L2, L4, and L5 are built mechanisms whose verdict is pending a multi-node cloud
    run that has not happened yet. A porcupine <code>Unknown</code> on an
    over-concurrent history is an accepted T1 limit.
  </Callout>

<section class="block" id="findings">
  <SectionHead num="07" label="Findings" title="What the suite <em>does and does not</em> prove" />

  This is the honest ledger of what the suite buys, where it stops, and the
  techniques that do the work. The Gaps and Techniques tables carry the verdict.
  The eight strengths sit one click away.

  ### Strengths

  <Aside label="Strengths" summary="What the suite buys, in eight points">

  - **Differential parity of memory and Lua against the oracle.** One §5.2.1
    producer table is the oracle. The Redis Lua append path is asserted to be
    byte-identical to pure-Go `ValidateProducer` for every row, including the
    persisted state and the write-only-when-accepted rule.[^f-diff]
  - **Pure-core and imperative-shell discipline.** `model_fence.go` and
    `check_cursor.go` are I/O-free oracles unit-tested with hand-built porcupine
    histories (18 jepsen unit tests, no cluster). The `scenario_*.go` files are the
    live drivers.[^f-split]
  - **Porcupine linearizability gate on the lease fence.** A sequential
    single-holder model encodes the claim and ack Lua. An `OK` ack under a stale
    token is the two-holder violation, reported with `VisualizePath`
    counterexamples.[^f-porcupine]
  - **Lua and Go lockstep mirrors in both directions.** `common.lua` `fenced()` and
    `state.go` `FenceDecision` each cite the other. The cursor checker mirrors
    `offsetGreater` a third time, so the same rule is pinned from three sides.[^f-mirror]
  - **Property-style lex invariant over hostile bytes.** Frame and offset tests
    sort `0x00`, `0xff`, and `|` payloads and assert bytewise order equals stream
    order, which is the basis of the ZSET log model.[^f-lex]
  - **Real Redis or skip, with isolated DBs.** Each tier pings a dedicated DB (15
    store, 14 webhook, 13 bench) and skips under `-short` or when unreachable. So
    `go test ./...` stays green with no server while still exercising real Lua and
    the round-trip cost.[^f-redis]
  - **Determinism through injected seams.** `TailFunc`, `IPResolver`, explicit
    `now` clocks, and an explicit jitter fraction make the fence, claim, retry, and
    SSRF paths reproducible. `fakeStreams`, `fakeLister`, and `fakeMetrics` isolate
    Manager orchestration over a real store.[^f-seams]
  - **Conformance plus timing-bound concurrency assertions.** About 330 black-box
    conformance tests pin the wire spec. The store notify tests assert that an
    elapsed time under one second proves pub/sub and under 200ms proves the fast
    path. They assert the mechanism, not only the outcome.[^f-timing]

  </Aside>
