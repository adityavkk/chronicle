<Tldr>
  <ul>
    <li><strong>Five verbs, one script each.</strong> <code>PUT</code> creates, <code>POST</code> appends or closes, <code>GET</code> reads, <code>HEAD</code> probes, <code>DELETE</code> removes. Every write is a single Lua script. Validation, write, tail update, and wakeup land together or not at all.</li>
    <li><strong>The same four parts every time.</strong> The HTTP handler parses the request, the Go store frames the bytes and calls the script, and Redis runs it. Decisions live in the script. The handler only maps the reply to a status code.</li>
    <li><strong>Reads are one <code>ZRANGEBYLEX</code>.</strong> Frames are stored as <code>"&lt;offset&gt;|&lt;bytes&gt;"</code> members in a sorted set, zero-padded so byte order equals stream order. A read is a single lexicographic range from the cursor.</li>
    <li><strong>Live tailing is pub/sub plus a re-read.</strong> A long-poll subscribes to the stream's notify channel, then re-reads before blocking, so an append that lands in the gap is never missed.</li>
    <li><strong>One operation has no script.</strong> <code>HEAD</code> is two pipelined reads and no write. It is the only client-facing path that does not touch a Lua script or slide the TTL.</li>
  </ul>
</Tldr>

<section class="block" id="how-to-read">
  <SectionHead num="00" label="Orientation" title="How to read <em>this page</em>" />

  Every section below has the same four parts. A request you can `curl`, a
  sequence diagram, the Redis command sequence the script runs, and a sentence
  on each command.

  ### Four <em>parts</em>

  A request crosses the same four boundaries on every endpoint.

  <Table>
    | Part | Is | Does |
    | --- | --- | --- |
    | **Client** | `curl`, a browser, an official client library | Sends an HTTP request, stores one offset string to resume |
    | **Handler** | `handler.go` (Go) | Parses headers and query, calls the store, maps store errors to HTTP status codes |
    | **Store** | `store/redis/store.go` (Go) | Frames bytes into messages, snapshots the tail, runs the Lua script, decodes its reply |
    | **Redis** | Redis 8 | Runs one Lua script per mutation, atomically and serialized per stream |
  </Table>

  ### Keys and the <em>notify channel</em>

  Each stream at path `p` is four Redis keys that share a `{p}` hash tag, so
  they occupy one cluster slot, plus one pub/sub channel.

  <Table>
    | Key | Type | Holds |
    | --- | --- | --- |
    | `ds:{p}:meta` | hash | content type, tail offset, closed flag, TTL/expiry, fork lineage, refcount |
    | `ds:{p}:msg` | sorted set | message frames `"<offset>\|<bytes>"`, score `0`, ordered lexicographically |
    | `ds:{p}:prod` | hash | idempotent-producer state: `producerId → "epoch:lastSeq:lastUpdated"` |
    | `ds:{p}:forks` | set | the paths of forks that reference this stream as their source |
    | `ds:notify:{p}` | pub/sub | wakes long-poll and SSE waiters: `a` append, `c` close, `d` delete |
  </Table>

  <Aside label="Encoding" summary="How offsets and frames are laid out for lexicographic reads">
    An offset is two unsigned 64-bit numbers, `ReadSeq_ByteOffset`, each
    zero-padded to 16 digits. A fresh tail is
    `0000000000000000_0000000000000000`.[^offset] `ReadSeq` is reserved for
    future log rotation and stays `0`. `ByteOffset` counts payload bytes, not
    framing. A frame is one sorted-set member holding the message end offset,
    a `|`, then the raw payload, e.g. `0_13|{"tok":"The"}`.[^frame] Because
    the prefix is fixed-width and zero-padded, bytewise order equals stream
    order, so a read is a single `ZRANGEBYLEX` from an exclusive lower bound.
    This page abbreviates offsets to their significant digits (`0_0`, `0_13`,
    `0_29`). On the wire both halves are full width.
  </Aside>

  <Aside label="Shared machinery" summary="Three things that recur inside the mutation scripts">
    <ul>
      <li><strong>Lazy expiry.</strong> The first script to touch an expired stream runs <code>expire_cleanup</code> (a <code>DEL</code>, or a soft-delete if forks still reference it) and reports not found. Expiry is checked on access. There is no sweeper.</li>
      <li><strong>The sliding-TTL touch.</strong> A read or an append writes <code>HSET …:meta accessedAtNs &lt;now&gt;</code>, moving the idle window forward. A bare <code>Get</code>, a <code>HEAD</code>, and a close deliberately do not.</li>
      <li><strong>The TTL backstop.</strong> <code>refresh_backstop</code> arms a key TTL of <code>ttl × 1000 + 60000</code> ms via <code>PEXPIRE</code>, a 60-second grace over the logical TTL. A fork-referenced or soft-deleted stream gets <code>PERSIST</code> instead.</li>
    </ul>
    Every mutation reply is a fixed nine-field string array so int64 values
    survive Lua's double round-trip.[^reply]
  </Aside>

  <Aside label="Cost model" summary="Why each endpoint costs what it costs: three Redis structures carry the work">
    The `:msg` sorted set is the only structure that grows without bound. Past
    Redis's listpack threshold it is a skiplist, so an insert is `O(log N)`
    and a lexicographic range is `O(log N + M)`.[^zset] The `:meta` and
    `:prod` hashes are `O(1)` per field. The `:forks` set is `O(1)` to add or
    remove, and a notify publish is `O(S)` in subscribers. Throughout:
    **N** frames in the stream, **M** frames the call touches, **k** messages
    per batch, **b** bytes per batch, **S** subscribers, **P** producers,
    **d** fork-chain depth.
  </Aside>

  <Callout icon="arrow" label="Subscriptions">
    This page covers the stream operations: five verbs, one Lua script each.
    The protocol's reserved subscription API (`__ds` webhook and pull-wake
    delivery) is also implemented, on a separate plane with its own
    durability story, on [its own page](../subscriptions).
  </Callout>
</section>

<section class="block" id="put-create">
  <SectionHead num="01" label="PUT" title="Create a stream" />

  `PUT {stream-url}` creates a stream, and it is idempotent. A repeated `PUT`
  with matching configuration returns `200` instead of `201`. A `PUT` whose
  content type, TTL, or closed status differs returns `409`.[^create] The body,
  if any, becomes the stream's initial content.

  ```http title="Create a JSON stream"
  PUT /v1/stream/chat HTTP/1.1
  Content-Type: application/json
  Stream-TTL: 3600

  HTTP/1.1 201 Created
  Location: /v1/stream/chat
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000000
  ```

  ### Forking: <em>two streams, two slots</em>

  A `PUT` carrying `Stream-Forked-From` creates a fork. A fork is a new
  stream that inherits the source's data up to a fork offset without copying
  it.[^fork] This is the one operation that is not a single atomic step,
  because the fork's keys and the source's keys live in different cluster
  slots. The store runs two scripts: `incr_ref.lua` takes a reference on the
  source, then `create.lua` builds the child.

  ```http title="Fork an existing stream at offset 0_13"
  PUT /v1/stream/branch HTTP/1.1
  Stream-Forked-From: /v1/stream/chat
  Stream-Fork-Offset: 0000000000000000_0000000000000013

  HTTP/1.1 201 Created
  Stream-Next-Offset: 0000000000000000_0000000000000013
  ```

  />

  <Complexity
    label="Complexity, what forking saves"
    time="O(1)"
    space="O(1), no data copied"
    why="Forking copies nothing, however long the source. <code>incr_ref</code> bumps a refcount, registers the child, and persists the source keys, and <code>create</code> writes the fork's own meta, all O(1). The inherited prefix stays in the source's skiplist and is range-scanned in place on a read. A <code>Stream-Fork-Sub-Offset</code> adds one O(log N + M) read of the source to resolve the boundary byte."
  />

  <Callout icon="alert" label="Non-atomic across the pair">
    Because the two scripts span two slots, a crash between `incr_ref.lua`
    and `create.lua` leaks one reference on the source. That reference is
    reaped later by delete-cascade arithmetic, never by a dangling read. A
    concurrent identical fork is resolved by rolling our extra reference
    back. This window is documented. Everything within a single stream
    stays atomic.
  </Callout>

<section class="block" id="post-append">
  <SectionHead num="02" label="POST" title="Append to a stream" />

  `POST {stream-url}` with a body appends one batch. In JSON mode the body
  is a JSON array, flattened one level into individual messages. Otherwise
  the whole body is one message.[^append] The store frames the bytes against a
  tail snapshot, then runs `append.lua`, which re-validates and writes in one
  atomic step. A concurrent append that moved the tail returns `RETRY`, and the
  store re-frames up to 64 times before giving up.

  ```http title="Append two JSON messages"
  POST /v1/stream/chat HTTP/1.1
  Content-Type: application/json

  [{"tok":"The"},{"tok":"answer"}]

  HTTP/1.1 204 No Content
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ```

  The batch flattens to `{"tok":"The"}` (13 bytes) and `{"tok":"answer"}`
  (16 bytes), with end offsets `0_13` and `0_29`, so the tail advances to
  `0_29`.

  ### With an <em>idempotent producer</em>

  Three optional headers, `Producer-Id`, `Producer-Epoch`, and
  `Producer-Seq`, turn an append into a write that the server deduplicates on
  retry.[^producer] They must all be present or all absent. Producer
  validation runs **before** the `Stream-Seq` check, so a retry deduplicates
  even when its sequence would otherwise conflict, and a duplicate returns
  with **no write**.

  ```http title="Append with producer headers"
  POST /v1/stream/chat HTTP/1.1
  Content-Type: application/json
  Producer-Id: writer-1
  Producer-Epoch: 0
  Producer-Seq: 7

  [{"tok":"now"}]

  HTTP/1.1 200 OK
  Producer-Epoch: 0
  Producer-Seq: 7
  Stream-Next-Offset: 0000000000000000_0000000000000033
  ```

  <Table>
    | Verdict | Condition | HTTP | Extra headers |
    | --- | --- | --- | --- |
    | **Accept** | new `(epoch, seq)` in order | `200 OK` | `Producer-Epoch`, `Producer-Seq` |
    | **Duplicate** | `seq ≤ lastSeq` | `204 No Content` | echoes the stored `Producer-Seq` |
    | **StaleEpoch** | `epoch < stored` (a producer fenced by a newer epoch) | `403 Forbidden` | `Producer-Epoch: <current>` |
    | **SeqGap** | `seq > lastSeq + 1` | `409 Conflict` | `Producer-Expected-Seq`, `Producer-Received-Seq` |
    | **EpochSeq** | new epoch not starting at `seq 0` | `400 Bad Request` | none |
  </Table>

  ### Append <em>and close</em>

  Add `Stream-Closed: true` to an append with a body and the same
  `append.lua` writes the final frame, flips the closed flag, and publishes
  `c` instead of `a` in one step. Readers see the last bytes and the EOF
  together.
</section>

<section class="block" id="post-close">
  <SectionHead num="03" label="POST" title="Close a stream" />

  `POST {stream-url}` with `Stream-Closed: true` and an empty body closes a
  stream without appending. Closure is durable and monotonic. Once closed, a
  stream stays closed, and every later append returns `409`.[^close] The
  close-only path is its own script, `close.lua`.

  ```http title="Close-only"
  POST /v1/stream/chat HTTP/1.1
  Stream-Closed: true

  HTTP/1.1 204 No Content
  Stream-Closed: true
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ```

<section class="block" id="get-catchup">
  <SectionHead num="04" label="GET" title="Read, catch-up" />

  `GET {stream-url}?offset=<offset>` returns every message past the offset,
  up to a server chunk limit. `offset=-1` (or an absent offset) starts from
  the beginning, and `offset=now` resolves to the current tail.[^read] The
  read is one `ZRANGEBYLEX` inside `read.lua`.

  ```http title="Read from the beginning"
  GET /v1/stream/chat?offset=-1 HTTP/1.1

  HTTP/1.1 200 OK
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ETag: "0000000000000000_0000000000000029"
  Cache-Control: public, max-age=60, stale-while-revalidate=300

  [{"tok":"The"},{"tok":"answer"}]
  ```

      { cmd: "HGETALL", key: "ds:{p}:meta", does: "Return the fresh meta alongside the frames so the handler can compute the next offset and whether the client is up to date.", loc: "read.lua:28" },
    ]}
    caption="A fork read adds a second ZRANGEBYLEX, in Go, against the source's :msg in its own slot, capped at the fork offset, so the inherited prefix comes back ahead of the fork's own frames, and post-fork source appends stay invisible."
  />

  <Complexity
    time="O(log N + M)"
    space="O(M), the frames returned"
    why="<code>ZRANGEBYLEX</code> descends the skiplist to the cursor in O(log N), then walks the M frames at or past it, and the two bounded <code>HGETALL</code>s on meta are O(1). A fork read repeats the seek-and-walk on each source up the chain, which is O(d · (log N + M)) for chain depth d, and the server caps M at a chunk limit, so one response stays bounded."
  />

  <Callout icon="info" label="Why the bound looks the way it does">
    The lower bound `"(<offset>\xff"` is exclusive and ends in the byte
    `0xFF`. Frames sharing the requested offset prefix have `|` (`0x7C`) as
    their next byte, which sorts below `0xFF`, so they are excluded, because
    the client already has those bytes. Any frame with a greater offset
    prefix sorts above the bound and is included. Payload bytes never enter
    the comparison. The fixed-width prefix decides first.[^lex]
  </Callout>
</section>

<section class="block" id="get-longpoll">
  <SectionHead num="05" label="GET" title="Read, live, long-poll" />

  `GET {stream-url}?offset=<offset>&live=long-poll` holds the request open
  until data arrives past the offset, the stream closes, or a 30-second
  timeout returns `204`.[^longpoll] The waiting runs in Go
  (`WaitForMessages`), not in a script, and it re-reads before blocking so an
  append that lands in the gap is never missed.

  ```http title="Long-poll from the tail"
  GET /v1/stream/chat?offset=now&live=long-poll HTTP/1.1

  HTTP/1.1 200 OK            ← data arrived within 30 s
  Stream-Next-Offset: 0000000000000000_0000000000000033
  Stream-Cursor: 7f3a…

  HTTP/1.1 204 No Content    ← 30 s passed with nothing new
  Stream-Up-To-Date: true
  Stream-Cursor: 7f3a…
  ```

  <Diagram
    name="seq-longpoll"
    label="GET, long-poll"
    caption="The waiter never publishes. The wake is some other request's PUBLISH inside append.lua or close.lua. A 1-second defensive poll recovers any wakeup dropped by pub/sub delivery, which does not guarantee delivery."
  />

  <RedisTrace
    label="The wait loop, step by step"
    script="—  Go orchestration"
    source="store/redis/notify.go"
    steps={[
      { cmd: "EVAL read.lua", key: "fast path", does: "If frames already sit past the offset, respond <strong>200</strong> now and skip the subscribe entirely. Most caught-up polls that race an append end here.", loc: "notify.go:33", kind: "read" },
      { cmd: "SUBSCRIBE", key: "ds:notify:{p}", does: "Open a dedicated pub/sub connection to the stream's channel and confirm it before waiting.", loc: "notify.go:48", kind: "read" },
      { cmd: "EVAL read.lua", key: "re-check", does: "Re-read <em>before</em> blocking. An append that landed between the fast-path read and the SUBSCRIBE would otherwise be lost until the timeout. This re-read closes that race.", loc: "notify.go:57", kind: "read" },

    ]}
  />

  <Callout icon="alert" label="Fork waiters never hang">
    An offset in a fork's inherited range (below `forkOffset`) can only be
    served by the source, and source appends never publish to the fork's
    channel. Blocking there would hang until the timeout, so the store
    detects this case and returns empty immediately instead of
    subscribing.[^forkwait]
  </Callout>

  <Complexity
    time="O(log N + M) per read"
    space="O(M) returned + O(1) subscription"
    why="Every read is the same skiplist seek-and-walk. The <code>SUBSCRIBE</code> is O(1), and the block costs no compute while idle. One request does the fast-path read, the re-check read, and one read per wakeup, which is a small constant number of reads, independent of how long it waits."
  />
</section>

<section class="block" id="get-sse">
  <SectionHead num="06" label="GET" title="Read, live, SSE" />

  `GET {stream-url}?offset=<offset>&live=sse` keeps a Server-Sent Events
  stream open and pushes each batch as it arrives.[^sse] The handler flushes
  the SSE headers, then loops over `read.lua`, and every 60 seconds it cycles
  the connection so a CDN can collapse concurrent tails.

  ```http title="SSE tail"
  GET /v1/stream/chat?offset=now&live=sse HTTP/1.1

  HTTP/1.1 200 OK
  Content-Type: text/event-stream

  event: data
  data: [{"tok":"now"}]

  event: control
  data: {"streamNextOffset":"0000000000000000_0000000000000033","streamCursor":"7f3a…"}
  ```

    caption="The data event carries the bytes. The control event after it carries the next offset and a cursor. A closed stream gets a final control with streamClosed:true, then the connection ends, and clients must not reconnect."
  />

<section class="block" id="head">
  <SectionHead num="07" label="HEAD" title="Read metadata" />

  soft-deletion and expiry in Go. A metadata probe is not access, so it does

  ```http title="Probe a stream"
  HTTP/1.1 200 OK
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000029
  Stream-TTL: 3600
  Cache-Control: no-store
  ```

<section class="block" id="delete">
  <SectionHead num="08" label="DELETE" title="Delete a stream" />

  `DELETE {stream-url}` removes a stream and its data. If forks still
  reference it, the stream cannot be removed, because its data is needed for
  their inherited prefix. So `delete.lua` soft-deletes instead. The URL
  answers `410` to clients while the data stays for fork reads.[^delete] When
  the last fork detaches, garbage collection cascades up the fork chain.

  ```http title="Delete a fork"
  DELETE /v1/stream/branch HTTP/1.1

  HTTP/1.1 204 No Content
  ```

  <Diagram

The cascade runs one O(1) <code>decr_ref</code> per ancestor and frees each hard-deleted ancestor's frames, so a chain of depth d costs O(d) scripts plus the frames reclaimed."
