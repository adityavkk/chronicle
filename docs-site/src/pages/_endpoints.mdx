<Tldr>
  <ul>
    <li><strong>Five verbs, one script each.</strong> <code>PUT</code> creates, <code>POST</code> appends or closes, <code>GET</code> reads, <code>HEAD</code> probes, <code>DELETE</code> removes. Every write is a single Lua script — validation, write, tail update, and wakeup land together or not at all.</li>
    <li><strong>The same four lifelines every time.</strong> The HTTP handler parses the request, the Go store frames the bytes and calls the script, Redis executes it. Decisions live in the script; the handler only maps the reply to a status code.</li>
    <li><strong>Reads are one <code>ZRANGEBYLEX</code>.</strong> Frames are stored as <code>"&lt;offset&gt;|&lt;bytes&gt;"</code> members in a sorted set, zero-padded so byte order equals stream order. A read is a single lexicographic range from the cursor.</li>
    <li><strong>Live tailing is pub/sub plus a re-read.</strong> A long-poll subscribes to the stream's notify channel, then re-reads before blocking, so an append that lands in the gap is never missed.</li>
    <li><strong>One operation has no script.</strong> <code>HEAD</code> is two pipelined reads and no write — the only client-facing path that does not touch a Lua script or slide the TTL.</li>
  </ul>
</Tldr>

<section class="block" id="how-to-read">
  <SectionHead num="00" label="Orientation" title="How to read <em>this page</em>" />

  Every section below has the same four parts: a request you can `curl`, a
  sequence diagram of the path that request travels, the Redis command
  sequence the script runs, and a sentence on each command. The machinery
  that repeats across endpoints is defined once here, so each section can
  skip it.

  ### Four <em>lifelines</em>

  A request crosses the same boundaries on every endpoint. Naming them once:

  <Table>
    | Lifeline | Is | Does |
    | --- | --- | --- |
    | **Client** | `curl`, a browser, an official client library | Sends an HTTP request, stores one offset string to resume |
    | **Handler** | `handler.go` (Go) | Parses headers and query, calls the store, maps store errors to HTTP status codes |
    | **Store** | `store/redis/store.go` (Go) | Frames bytes into messages, snapshots the tail, runs the Lua script, decodes its reply |
    | **Redis** | Redis 8 | Executes one Lua script per mutation, atomically and serialized per stream |
  </Table>

  ### Keys and the <em>notify channel</em>

  Each stream at path `p` is four Redis keys sharing a `{p}` hash tag (so
  they occupy one cluster slot) and one pub/sub channel:

  <Table>
    | Key | Type | Holds |
    | --- | --- | --- |
    | `ds:{p}:meta` | hash | content type, tail offset, closed flag, TTL/expiry, fork lineage, refcount |
    | `ds:{p}:msg` | sorted set | message frames `"<offset>\|<bytes>"`, score `0`, ordered lexicographically |
    | `ds:{p}:prod` | hash | idempotent-producer state: `producerId → "epoch:lastSeq:lastUpdated"` |
    | `ds:{p}:forks` | set | the paths of forks that reference this stream as their source |
    | `ds:notify:{p}` | pub/sub | wakes long-poll and SSE waiters: `a` append, `c` close, `d` delete |
  </Table>

  ### Offsets and <em>frames</em>

  An offset is two unsigned 64-bit numbers, `ReadSeq_ByteOffset`, each
  zero-padded to 16 digits: a fresh stream's tail is
  `0000000000000000_0000000000000000`.[^offset] `ReadSeq` is reserved for
  future log rotation and stays `0`; `ByteOffset` counts payload bytes, not
  framing. This page abbreviates offsets to their significant digits —
  `0_0`, `0_13`, `0_29` — but on the wire both halves are full width.

  A frame is one sorted-set member: the message's **end** offset, a `|`
  separator, then the raw payload — `0_13|{"tok":"The"}`.[^frame] Because the
  offset prefix is fixed-width and zero-padded, bytewise lexicographic order
  of members equals stream order, and the prefix makes every member unique.
  A read is therefore a single `ZRANGEBYLEX` from an exclusive lower bound.

  ### Machinery that <em>runs in most scripts</em>

  Three things recur inside the mutation scripts; rather than repeat them in
  every section, here is what they do:

  - **Lazy expiry.** `is_expired(meta)` checks whether the absolute
    `expAtNs` has passed, or `accessedAtNs + ttl` has. The first script to
    touch an expired stream runs `expire_cleanup` — a `DEL` of all keys, or
    a soft-delete if forks still reference it — and then reports the stream
    as not found. Expiry is evaluated on access; there is no sweeper.
  - **The sliding-TTL touch.** A read or an append writes
    `HSET …:meta accessedAtNs <now>`, sliding the idle window forward. A
    bare `Get`, a `HEAD`, and a close deliberately do **not**.
  - **The TTL backstop.** `refresh_backstop` arms a Redis key TTL of
    `ttl × 1000 + 60000` ms — a 60-second grace over the logical TTL — via
    `PEXPIRE`, so a stream nobody ever touches again is still reaped. A
    fork-referenced or soft-deleted stream gets `PERSIST` instead and never
    carries a key TTL.

  Every mutation reply is a fixed nine-field array of strings —
  `{status, tail, producerResult, currentEpoch, expectedSeq, receivedSeq,
  lastSeq, closed, alreadyClosed}` — so int64 values survive Lua's double
  round-trip.[^reply] The sections show the `status` that matters and skip
  the rest.

  <Callout icon="info" label="Out of scope">
    Chronicle does not implement the protocol's optional reserved
    subscriptions API (`__ds` webhooks and pull-wake) — the same posture as
    the reference Caddy plugin's conformance gate. This page covers the
    operations Chronicle serves.
  </Callout>
</section>

<section class="block" id="put-create">
  <SectionHead num="01" label="PUT" title="Create a stream" />

  `PUT {stream-url}` creates a stream, idempotently. A repeated `PUT` with
  matching configuration returns `200` instead of `201`; a `PUT` whose
  content type, TTL, or closed status differs from the existing stream
  returns `409`.[^create] The body, if any, becomes the stream's initial
  content. Everything happens in one script, `create.lua`, which touches
  only the new stream's own four keys.

  ```http title="Create a JSON stream"
  PUT /v1/stream/chat HTTP/1.1
  Content-Type: application/json
  Stream-TTL: 3600

  HTTP/1.1 201 Created
  Location: /v1/stream/chat
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000000
  ```

  The store does an existence pre-check before the script — the same check
  `create.lua` repeats atomically inside — so an idempotent match returns
  before any fork validation or body parsing, mirroring the reference
  implementation's precedence.

  ### Forking: <em>two streams, two slots</em>

  A `PUT` carrying `Stream-Forked-From` creates a fork: a new stream that
  inherits the source's data up to a fork offset without copying it.[^fork]
  This is the one operation that is not a single atomic step. The fork's
  keys and the source's keys live in different cluster slots, so the store
  runs two scripts: `incr_ref.lua` takes a reference on the source, then
  `create.lua` builds the child.

  ```http title="Fork an existing stream at offset 0_13"
  PUT /v1/stream/branch HTTP/1.1
  Stream-Forked-From: /v1/stream/chat
  Stream-Fork-Offset: 0000000000000000_0000000000000013

  HTTP/1.1 201 Created
  Stream-Next-Offset: 0000000000000000_0000000000000013
  ```

  <Callout icon="alert" label="Non-atomic across the pair">
    Because the two scripts span two slots, a crash between `incr_ref.lua`
    and `create.lua` leaks one reference on the source — reaped only later
    by delete-cascade arithmetic, never by a dangling read. A concurrent
    identical fork is resolved by rolling our extra reference back. This
    window is documented; everything within a single stream stays atomic.
  </Callout>

<section class="block" id="post-append">
  <SectionHead num="02" label="POST" title="Append to a stream" />

  `POST {stream-url}` with a body appends one batch. In JSON mode the body
  is a JSON array, flattened one level into individual messages; otherwise
  the whole body is one message.[^append] The store frames the bytes
  against a snapshot of the tail, then runs `append.lua`, which re-validates
  and writes in one atomic step. If another writer advanced the tail between
  the snapshot and the script, the script returns `RETRY` and the store
  re-frames — up to 64 times before giving up.

  ```http title="Append two JSON messages"
  POST /v1/stream/chat HTTP/1.1
  Content-Type: application/json

  [{"tok":"The"},{"tok":"answer"}]

  HTTP/1.1 204 No Content
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ```

  The batch flattens to two messages, `{"tok":"The"}` (13 bytes) and
  `{"tok":"answer"}` (16 bytes). Their end offsets are `0_13` and `0_29`, so
  the stored frames are `0_13|{"tok":"The"}` and `0_29|{"tok":"answer"}`,
  and the tail advances to `0_29`.

  ### With an <em>idempotent producer</em>

  Three optional headers — `Producer-Id`, `Producer-Epoch`, `Producer-Seq`
  — turn an append into a fire-and-forget write that deduplicates on retry,
  Kafka-style.[^producer] They must all be present or all absent. Producer
  validation runs **before** the `Stream-Seq` check, so a retry deduplicates
  even when its sequence would otherwise conflict, and a duplicate returns
  with **no write**.

  ```http title="Append with producer headers"
  POST /v1/stream/chat HTTP/1.1
  Content-Type: application/json
  Producer-Id: writer-1
  Producer-Epoch: 0
  Producer-Seq: 7

  [{"tok":"now"}]

  HTTP/1.1 200 OK
  Producer-Epoch: 0
  Producer-Seq: 7
  Stream-Next-Offset: 0000000000000000_0000000000000033
  ```

  The five verdicts map to five HTTP responses. The store never writes
  frames on anything but Accept:

  <Table>
    | Verdict | Condition | HTTP | Extra headers |
    | --- | --- | --- | --- |
    | **Accept** | new `(epoch, seq)` in order | `200 OK` | `Producer-Epoch`, `Producer-Seq` |
    | **Duplicate** | `seq ≤ lastSeq` | `204 No Content` | echoes the stored `Producer-Seq` |
    | **StaleEpoch** | `epoch < stored` (a fenced zombie) | `403 Forbidden` | `Producer-Epoch: <current>` |
    | **SeqGap** | `seq > lastSeq + 1` | `409 Conflict` | `Producer-Expected-Seq`, `Producer-Received-Seq` |
    | **EpochSeq** | new epoch not starting at `seq 0` | `400 Bad Request` | — |
  </Table>

  ### Append <em>and close</em>

  Add `Stream-Closed: true` to an append with a body and the same
  `append.lua` writes the final frame and flips the closed flag in one step:
  it adds `HSET …:meta closed 1` (plus the closing producer tuple
  `cbId`/`cbEpoch`/`cbSeq` when producer headers are present) and publishes
  `c` instead of `a`. Readers see the last bytes and the EOF together.
</section>

<section class="block" id="post-close">
  <SectionHead num="03" label="POST" title="Close a stream" />

  `POST {stream-url}` with `Stream-Closed: true` and an empty body closes a
  stream without appending. Closure is durable and monotonic — once closed,
  a stream stays closed, and every later append returns `409`.[^close] The
  close-only path is its own script, `close.lua`, which (matching the memory
  store) does **not** slide the TTL and does **not** check soft-deletion.

  ```http title="Close-only"
  POST /v1/stream/chat HTTP/1.1
  Stream-Closed: true

  HTTP/1.1 204 No Content
  Stream-Closed: true
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ```

<section class="block" id="get-catchup">
  <SectionHead num="04" label="GET" title="Read — catch-up" />

  `GET {stream-url}?offset=<offset>` returns every message past the offset,
  up to a server chunk limit. `offset=-1` (or an absent offset) starts from
  the beginning; `offset=now` resolves to the current tail.[^read] The read
  is one `ZRANGEBYLEX` inside `read.lua`. The handler brackets it with two
  metadata fetches: one before, to turn a missing or soft-deleted stream
  into the right status, and one after, to decide whether the client has
  caught up to the tail.

  ```http title="Read from the beginning"
  GET /v1/stream/chat?offset=-1 HTTP/1.1

  HTTP/1.1 200 OK
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000029
  ETag: "0000000000000000_0000000000000029"
  Cache-Control: public, max-age=60, stale-while-revalidate=300

  [{"tok":"The"},{"tok":"answer"}]
  ```

  <Callout icon="info" label="Why the bound looks strange">
    The lower bound `"(<offset>\xff"` is exclusive and ends in the byte
    `0xFF`. Frames sharing the requested offset prefix have `|` (`0x7C`) as
    their next byte, which sorts below `0xFF`, so they are excluded — the
    client already has those bytes. Any frame with a greater offset prefix
    sorts above the bound and is included. Payload bytes never enter the
    comparison; the fixed-width prefix decides first.[^lex]
  </Callout>
</section>

<section class="block" id="get-longpoll">
  <SectionHead num="05" label="GET" title="Read — live, long-poll" />

  `GET {stream-url}?offset=<offset>&live=long-poll` holds the request open
  until data arrives past the offset, the stream closes, or a 30-second
  timeout returns `204`.[^longpoll] The waiting is orchestrated in Go
  (`WaitForMessages`), not in a script: the store reads first, subscribes to
  the stream's notify channel, then re-reads before blocking. That re-read
  is the load-bearing step — it closes the window where an append could land
  between the first read and the subscribe.

  ```http title="Long-poll from the tail"
  GET /v1/stream/chat?offset=now&live=long-poll HTTP/1.1

  HTTP/1.1 200 OK            ← data arrived within 30 s
  Stream-Next-Offset: 0000000000000000_0000000000000033
  Stream-Cursor: 7f3a…

  HTTP/1.1 204 No Content    ← 30 s passed with nothing new
  Stream-Up-To-Date: true
  Stream-Cursor: 7f3a…
  ```

<section class="block" id="get-sse">
  <SectionHead num="06" label="GET" title="Read — live, SSE" />

  `GET {stream-url}?offset=<offset>&live=sse` keeps a Server-Sent Events
  stream open and pushes each batch as it arrives.[^sse] The handler flushes
  the SSE headers, then loops: read with `read.lua`, emit an `event: data`
  and an `event: control`, wait ~100 ms for a wakeup, repeat. Every 60
  seconds it cycles the connection so a CDN can collapse concurrent tails;
  the client reconnects from the last `streamNextOffset`.

  ```http title="SSE tail"
  GET /v1/stream/chat?offset=now&live=sse HTTP/1.1

  HTTP/1.1 200 OK
  Content-Type: text/event-stream

  event: data
  data: [{"tok":"now"}]

  event: control
  data: {"streamNextOffset":"0000000000000000_0000000000000033","streamCursor":"7f3a…"}
  ```

<section class="block" id="head">
  <SectionHead num="07" label="HEAD" title="Read metadata" />

  `HEAD {stream-url}` returns the tail offset, content type, TTL/expiry, and
  closed status with no body.[^head] It is the canonical way to learn a
  stream's tail and whether it is closed without transferring data. This is
  the only client-facing operation with no Lua script: the store reads the
  two hashes in one pipeline and checks soft-deletion and expiry in Go. A
  metadata probe is not access, so — unlike a read or an append — it does
  not slide the TTL.

  ```http title="Probe a stream"
  HEAD /v1/stream/chat HTTP/1.1

  HTTP/1.1 200 OK
  Content-Type: application/json
  Stream-Next-Offset: 0000000000000000_0000000000000029
  Stream-TTL: 3600
  Cache-Control: no-store
  ```

<section class="block" id="delete">
  <SectionHead num="08" label="DELETE" title="Delete a stream" />

  `DELETE {stream-url}` removes a stream and its data. If forks still
  reference it, the stream cannot vanish — its data is needed for their
  inherited prefix — so `delete.lua` soft-deletes instead: the URL answers
  `410` to clients while the data stays for fork reads.[^delete] When the
  last fork detaches, garbage collection cascades up the fork chain. The
  delete of one stream is atomic; the cascade across the chain spans cluster
  slots and is orchestrated in Go.

  ```http title="Delete a fork"
  DELETE /v1/stream/branch HTTP/1.1

  HTTP/1.1 204 No Content
  ```
