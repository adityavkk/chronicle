  <ul>
    <li><strong>The protocol is plain HTTP.</strong> A stream is a URL. <code>PUT</code> creates it, <code>POST</code> appends, and <code>GET</code> reads from any offset. Long-poll and SSE serve live tails.</li>
    <li><strong>Redis is the only dependency.</strong> Chronicle is one Go binary; stream data, producer state, and reader wakeups live in Redis 8.</li>
    <li><strong>Every mutation is one Lua script.</strong> Validation, write, tail update, and notification run atomically and serialized per stream. No partial appends and no lost wakeups.</li>
    <li><strong>It tracks the reference implementation.</strong> The store contract and handler mirror the upstream Caddy plugin, and ~330 official conformance tests gate every change.</li>
    <li><strong>Load-tested against that reference.</strong> One writer per stream sustains 1,500 appends/s with zero errors. Concurrent writers on one stream collapse near 1,000/s. Both results are in the benchmark report.</li>
  </ul>

<section class="block" id="protocol">
  <SectionHead num="01" label="Protocol" title="What a durable stream <em>is</em>" />

  A durable stream is an append-only sequence of bytes that lives at a URL.
  The [Durable Streams protocol](https://github.com/durable-streams/durable-streams)
  uses HTTP verbs for the lifecycle. `PUT` creates a stream, `POST` appends
  a batch, `GET` reads from an offset, and `DELETE` removes it. Every response
  carries a `Stream-Next-Offset` header. Pass it back as `?offset=` to
  resume where you left off. Resuming needs no consumer groups and no
  server-side cursor state. The client stores one string.

  ```bash title="The protocol in four core requests"
  # Create a stream
  curl -i -X PUT http://localhost:4437/v1/stream/demo -H 'Content-Type: application/json'

  # Append two messages (one request = one batch)
  curl -i -X POST http://localhost:4437/v1/stream/demo \
    -H 'Content-Type: application/json' -d '[{"hello":"world"},{"n":2}]'

  # Read from the beginning
  curl -i 'http://localhost:4437/v1/stream/demo?offset=-1'

  # Close the stream (EOF for all readers)
  curl -i -X POST http://localhost:4437/v1/stream/demo -H 'Stream-Closed: true'
  ```

  Reads are pull-based, in three modes. A catch-up read returns everything
  from a given offset. A long-poll read holds the request open until data
  arrives or a timeout returns `204` after 30 seconds. An SSE read keeps a
  tail open continuously, and the server cycles connections every 60 seconds
  so CDNs can collapse concurrent tails. The two live modes are one query
  parameter apart:

  The protocol also covers explicit closure (an EOF every reader observes),
  TTL expiry, stream forking, idempotent producers, and the reserved
  [subscriptions API](./subscriptions) of durable cursors that wake workers by
  webhook or pull-wake. OpMap below indexes each operation. Chronicle
  implements the full surface,[^spec] and the official client libraries
  (TypeScript, Python, Go, Rust) work against its base URL.

<section class="block" id="design">
  <SectionHead num="02" label="Design" title="Pure core, <em>imperative edge</em>" />

  Everything decidable without I/O lives in the `protocol/` package as plain
  functions over immutable inputs. These cover header and TTL-grammar
  parsing, offset and cursor math, SSE event framing, and the
  producer-validation state machine. The state machine is spec section
  5.2.1 transcribed into one pure function that maps a producer's state and
  an incoming epoch and sequence pair to `Accept`, `Duplicate`,
  `StaleEpoch`, `SeqGap`, or `InvalidEpochSeq`. These functions are tested
  without mocks. The Lua script that enforces producer rules inside Redis
  mirrors the state machine, and the pure function is its test oracle.

  The HTTP handler parses requests and calls the store, the Redis store runs
  the scripts, and the decisions stay in the core.

  ```text title="Layout"
  protocol/      pure protocol logic: headers, parsing, cursors, producer rules
  store/         the storage contract (mirrors the Caddy plugin's store package)
  store/redis/   the Redis backend: Lua scripts, frames, pub/sub waiters
  handler.go     HTTP layer (mirrors the Caddy plugin's handler)
  cmd/chronicle/ the server binary
  ```

  ### Parity with the <em>reference implementation</em>

  Chronicle mirrors the protocol's reference Caddy plugin, pinned to a
  recorded upstream commit.[^caddy] The `store.Store` interface, type names,
  error variables, handler structure, and behavioral details stay in
  lockstep. Where the Caddy plugin persists to memory or files, Chronicle
  persists to Redis.

  <Aside label="Tradeoff" summary="Why mirror upstream, and what it costs">
    The payoff is maintenance. When upstream changes, porting is a file diff
    against a parity table. The cost is design freedom. Chronicle inherits
    upstream's shapes even where a Go server written from scratch might choose
    differently.
  </Aside>
</section>

<section class="block" id="storage">
  <SectionHead num="03" label="Storage" title="The Redis <em>data model</em>" />

  Each stream at path `p` maps to three Redis keys and one pub/sub channel:

  <Table>
    | Key | Type | Holds |
    | --- | --- | --- |
    | `ds:{p}:meta` | hash | content type, tail offset, closed flag, TTL/expiry, fork lineage, refcount |
    | `ds:{p}:msg` | sorted set | message frames `"<offset>\|<bytes>"`, ordered lexicographically by offset |
    | `ds:{p}:prod` | hash | idempotent-producer state: `producerId → epoch:lastSeq` |
    | `ds:notify:{p}` | pub/sub | wakes long-poll/SSE waiters on append, close, and delete |
  </Table>

  Every mutation runs as a single Lua script via `EVALSHA`. Redis runs
  scripts atomically,[^eval] so validation, the write, the tail update, and
  the waiter notification land together or not at all, serialized per stream.
  A long-poll reader subscribes to the notify channel and then re-checks the
  stream, so an append that lands between the check and the subscribe is never
  missed. The OpMap below walks each operation's Redis steps.

  <OpMap />

  The model requires this of the Redis deployment:

  - **Redis ≥ 7.4, 8.x recommended.** Chronicle uses `EVALSHA` scripts,
    pub/sub, and hash-field TTLs (`HEXPIRE`).[^hexpire]
  - **`maxmemory-policy noeviction`.** Any eviction policy can drop keys and
    truncate a stream.[^evict] Chronicle warns at startup if it can read the
    config and the policy differs.
  - **Cluster mode works, with one caveat.** All keys for a stream share a
    `{path}` hash tag, so they occupy a single slot and scripts stay valid
    in a cluster. Fork lifecycle operations span two streams and are not
    atomic across slots.

  <Callout icon="alert" label="Durability">
    Redis replication is asynchronous, so an acknowledged append can be lost
    on failover.[^repl] Within a healthy primary, appends are atomic and
    producer idempotency is exact.
  </Callout>
</section>

<section class="block" id="running">
  <SectionHead num="04" label="Quickstart" title="Run it <em>locally</em>" />

  Requirements: Go ≥ 1.26 and Docker (for Redis).

  ```bash
  make redis-up      # start Redis 8 (docker compose, AOF persistence)
  make run           # build + start chronicle on :4437
  ```

  Port 4437 is the protocol's IANA-selected default. Configuration comes
  from flags or environment variables. Flags take precedence.

  <Table>
    | Flag | Env | Default | Meaning |
    | --- | --- | --- | --- |
    | `--listen` | `CHRONICLE_LISTEN` | `:4437` | HTTP listen address |
    | `--redis-url` | `CHRONICLE_REDIS_URL` | `redis://localhost:6379` | Redis connection URL |
    | `--store` | `CHRONICLE_STORE` | `redis` | Storage backend: `redis` or `memory` (dev/testing) |
    | `--stream-root` | `CHRONICLE_STREAM_ROOT` | `/v1/stream/` | URL prefix streams live under |
    | `--long-poll-timeout` | `CHRONICLE_LONG_POLL_TIMEOUT` | `30s` | How long `live=long-poll` waits before `204` |
    | `--sse-reconnect-interval` | `CHRONICLE_SSE_RECONNECT_INTERVAL` | `60s` | SSE connection cycling (enables CDN collapsing) |
  </Table>
</section>

<section class="block" id="performance">
  <SectionHead num="05" label="Performance" title="Throughput, latency, <em>two bottlenecks</em>" />

  Chronicle is load-tested with `dsload`, a declarative load generator for
  Durable Streams servers that lives on the repo's `loadgen` branch.[^rig]
  Scenarios are YAML and set stream counts, writer rates, payload sizes, and
  reader populations (SSE and long-poll). Writers are open-loop. They send on a
  fixed schedule and never wait for responses, so a stalling server cannot hide
  its latency. Each message carries its send timestamp, and tailers record
  write-to-receipt delivery latency into HDR histograms.

  The 2026-06-10 run[^report] compares Chronicle against the reference
  Caddy plugin (memory and file stores) under the same scenarios. The setup
  is one Apple Silicon host (14 cores, 48 GiB), loopback HTTP, Redis 8 in
  Docker with `appendfsync everysec`, measured windows of 60 to 120 s, and
  one run per cell. Conclusions rest on differences of an order of
  magnitude, not single-digit percentages.

  <Table>
    | Scenario | Offered | Chronicle measured |
    | --- | --- | --- |
    | `fanout`, 1 stream, 400 SSE + 100 long-poll readers | 10 appends/s, 5,000 deliveries/s | full rate, zero errors. SSE delivery p50 386 ms, p99 1.2 s |
    | `producer-sessions`, 50 streams, one idempotent producer each | 1,500 appends/s | 1,486/s, zero errors. Delivery p50 33 ms, p99 1.0 s |
    | `append-steady`, 20 streams, open-loop writers | 1,000 appends/s | 503/s achieved. 4,521 HTTP 500s. Append p99 9.3 s |
    | `append-sweep`, 16 streams, ramping writers | 200 to 2,000/s | tracks the ramp to about 1,000/s, then goodput falls to zero within about 20 s |
    | `sessions-light`, 10 streams, unsaturated | 20 appends/s | append p50 8.4 ms, p90 287 ms, p99 534 ms |
  </Table>

  Both negative results have a named mechanism.

  ### Append collapse under <em>concurrent writers</em>

  `store/redis` retries a contended append optimistically, up to 64 times
  with no backoff. With several appends in flight against one stream the
  retries collide, and past about 1,000/s in total every request uses all 64
  attempts and fails. Goodput drops to zero and stays there while load
  continues. With one in-flight append per stream, Chronicle instead sustains
  1,500/s across 50 streams with zero errors. The collapse is specific to
  concurrent appends on a single stream.

  ### A latency floor in <em>the wakeup path</em>

  Delivery p90 sits near 300 ms even at 20 msg/s with nothing saturated. The
  Caddy memory store served the same fanout scenario at delivery p50 9.4 ms,
  so the gap is Chronicle's wakeup path, not Redis round-trip time.

  <Aside label="Mechanism" summary="Why the floor exists, and the two proposed fixes">
    The SSE loop waits in 100 ms steps, and each iteration opens a fresh Redis
    pub/sub subscription with a dedicated connection per `Subscribe`. The
    benchmark Redis logged 96,564 connections received and 65,852 SUBSCRIBE
    calls over the suite. The report names two fixes. On the write path, use
    jittered backoff or per-stream single-flight. On the wakeup path, use a
    persistent per-stream pub/sub hub.
  </Aside>

  <Callout icon="alert" label="Caveats">
    Single host and loopback HTTP, so no network jitter and no TLS. Redis
    ran behind Docker Desktop, which adds about 0.4 ms median per round
    trip. One run per scenario cell. The report lists every threat to
    validity.
  </Callout>
</section>

<section class="block" id="conformance">
  <SectionHead num="06" label="Conformance" title="~330 tests, <em>plus the reserved APIs</em>" />

  Chronicle is tested against the official conformance suite.[^conf] It is
  about 330 protocol tests that treat the server as a black box. They cover
  idempotent producers, stream closure, forks, JSON mode, SSE, and
  property-based fuzzing, plus the reserved subscription group. All of them
  run against a live Redis.

  ```bash
  make conformance                              # full suite vs live Redis
  make conformance-filter FILTER="Idempotent"   # one group while iterating
  ```

  The optional reserved subscriptions API (`__ds` webhook and pull-wake
  delivery) is implemented and passes its conformance group too. It is
  backed by Redis and a recovery sweep, so cursors and leases survive a
  restart. See [Subscriptions](./subscriptions) for the design and the
  fault-injection results.
