<section class="block" id="shape">
  <SectionHead num="01" label="Shape" title="One rule shapes the <em>whole suite</em>" />

  Chronicle splits each component into a pure deterministic core and an imperative
  shell that does input and output. It tests each part at the cheapest level that
  stays faithful. Proofs and per-input properties test the cores. A real Redis tests
  the shell. Some properties only appear across processes, e.g. a failover or a race
  between two workers, and fault injection and linearizability checks test those. A
  black-box suite checks the wire protocol from outside the code. The table below
  maps each level to what it gives you and the infrastructure it needs.

  <Table wide>
    | Level | What runs | What it gives you | Infrastructure |
    | --- | --- | --- | --- |
    | Pure core | unit tables, Lean proofs, the cursor and fence model checkers | per-input correctness, and for the proofs, correctness for every input | none |
    | Differential | the same input through Go, Lua, and the compiled Lean model | the three implementations agree on every input | real Redis |
    | Property and model based | `rapid` generates random operation sequences against two backends | the backends agree across reachable states, with a shrunk counterexample on failure | real Redis |
    | Fuzzing | Go's coverage-guided fuzzer drives the same model | reaches rare branches that random sampling misses | real Redis |
    | Linearizability | `porcupine` checks recorded concurrent histories | the concurrent history has a valid sequential order | real Redis |
    | Concurrent protocol | TLA+ model checking, Apalache induction, Alloy | safety and liveness over all interleavings, and for Apalache, over all sizes | Java |
    | Fault injection | a `k3d` cluster and a Redis failover nemesis | end to end safety under crashes and failover | k3d or local Docker |
    | Wire contract | the vendored conformance suite | the HTTP surface matches the protocol | a live server |
    | Scale | the load generator and the GKE rig | throughput and latency under load | managed Redis |
  </Table>

  <Callout icon="info" label="How to read each section">
    Every section that follows states four things. What the test is, what it gives
    you, where it lives, and its downside. The downside is the point. A test
    described as all upside is being oversold. The limit of each kind tells you which
    other kind covers the gap.
  </Callout>
</section>

<section class="block" id="kinds">
  <SectionHead num="02" label="Kinds" title="The thirteen kinds of <em>test and proof</em>" />

  ### 1. Unit tests on the pure cores

  Table-driven Go tests on the pure functions. They cover `ValidateProducer`, the
  idempotent producer state machine, `Offset.Compare` and the offset codec,
  `GenerateResponseCursor`, and the webhook reducers such as `FenceDecision` and
  `MergeAcks`. They run in milliseconds with no Redis and no clock, because the
  caller passes the clock in. The downside is that they only check the cases you
  wrote, so they miss the inputs you did not think of. The proofs and the property
  tests below close that gap.

  Where: `ValidateProducer` at [`store/producer.go:17`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go#L17),
  `Compare` at [`store/offset.go:128`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go#L128),
  and the webhook reducers `FenceDecision` ([`webhook/state.go:119`](https://github.com/adityavkk/chronicle/blob/main/webhook/state.go#L119))
  and `MergeAcks` ([`webhook/state.go:235`](https://github.com/adityavkk/chronicle/blob/main/webhook/state.go#L235)).
  Run: `make test-unit`.

  ### 2. Integration tests against a real Redis

  The `store/redis` and `webhook` packages run against a live Redis 8. Each uses
  its own database, flushes once at start, and skips cleanly when Redis is not
  reachable or under `-short`. These run the Lua scripts and the full read-write
  round trip that the in-memory store cannot. The downside is that they need Redis,
  so they do not run in a bare unit pass.

  Where: [`store/redis/integration_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/integration_test.go)
  and [`webhook/redis_store_test.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/redis_store_test.go).
  Run: `make test`, which needs Redis. Run `make redis-up` first.

  ### 3. Differential tests

  The same input goes through three independent implementations and the results must
  match. The three are the pure Go function, which is the oracle, the Lua script
  Redis runs, which is the mirror, and the Lean model compiled to C, which is the
  proof and is kind 9 below. The producer state machine, the fence and slot
  predicates, the content-type and config normalizers, the `Stream-Seq` comparison,
  and the JSON flatten path each have a differential test.

  This gives you confidence that the three copies of a rule have not drifted apart.
  A differential test found the `10^14` producer-reply bug, issue
  [#47](https://github.com/adityavkk/chronicle/issues/47). The Lua reply rendered
  large sequence numbers in scientific notation, which the Go reply parser rejected.
  The downside is that a differential test only catches a disagreement. If all three
  copies share the same wrong assumption, they agree and the test passes.
  The independent properties and the proofs cover that gap.

  Where: [`store/redis/differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_test.go),
  [`webhook/predicate_differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/predicate_differential_test.go),
  and [`store/leanoracle`](https://github.com/adityavkk/chronicle/blob/main/store/leanoracle),
  which is the compiled-Lean oracle behind the `leanoracle` build tag. Run:
  `go test -run Differential ./store/redis/`, e.g. the producer differential property
  in [`differential_producer_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_producer_test.go).

  ### 4. Property-based and model-based tests

  The `rapid` library generates random, shrinking sequences of operations and runs
  each operation against both backends. One backend is the in-memory `MemoryStore`,
  the reference model. The other is the live Redis backend, the subject. After every
  step the two must agree on the result, the error, the tail offset, the metadata,
  and the bytes a read returns. A single fake clock drives both, so expiry is
  reproducible. When it fails it shrinks the failure to the smallest sequence that
  still breaks it. The downside is that it samples the space rather than covering it,
  and a bug in a pure core both backends share agrees in both. The independent
  metamorphic checks and the proofs cover that gap.

  Where: [`store/redis/equivalence_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/equivalence_test.go),
  [`store/redis/differential_producer_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_producer_test.go),
  and [`store/offset_property_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/offset_property_test.go).

  ### 5. Coverage-guided fuzzing

  The same `rapid` model is wrapped as a native Go fuzz target with `rapid.MakeFuzz`,
  so Go's coverage-guided fuzzer steers the generated sequences toward rare branches
  that uniform random sampling under-reaches, e.g. an epoch bump at a nonzero
  sequence or a fork sub-offset overshoot. A committed corpus of seeds, including
  four branch-verified fixtures, runs on every pull request. A nightly job runs the
  fuzzer for a fixed time.

  This gives you a search that reaches branches random sampling keeps missing. The
  downside is that fuzzing finds crashers and divergences but proves nothing. It is
  a search, not a guarantee.

  Where: [`store/redis/equivalence_fuzz_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/equivalence_fuzz_test.go),
  the corpus under [`store/redis/testdata/fuzz/`](https://github.com/adityavkk/chronicle/blob/main/store/redis/testdata/fuzz),
  and [`.github/workflows/fuzz-nightly.yml`](https://github.com/adityavkk/chronicle/blob/main/.github/workflows/fuzz-nightly.yml).

  ### 6. Linearizability checking with Porcupine

  `porcupine` takes a recorded concurrent history and a sequential model and decides
  whether the history has a valid sequential order. Chronicle uses it for the lease
  fence, the owner-epoch slot fence, the stream data plane of append, read, and
  close, and a composed model that carries both fences at once. It gives you a
  guarantee that concurrent operations behaved as if they ran one at a time, the
  correctness condition Herlihy and Wing defined. A torn read, a lost append, or a
  reordering has no valid order, and the checker prints the witness.

  <Complexity
    time="NP-hard in the history length"
    space="bounded history, small worker count"
    why="Deciding whether a concurrent history has a valid sequential order is NP-hard, so Chronicle runs it over bounded histories; a history that is too large returns Unknown, which counts as a failure."
    label="Cost of the linearizability check"
  />

  <Aside label="Downside" summary="The search is NP-hard, so it runs over bounded histories">
    Because the search is NP-hard, Chronicle runs it over bounded histories and a
    small number of workers. A history that is too large returns Unknown, which counts
    as a failure.
  </Aside>

  Where: [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go),
  [`jepsen/checker/model_shard.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_shard.go),
  [`jepsen/checker/model_store.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_store.go),
  and the `*-linz` scenarios. Run via the `jepsen-checker` binary.

  ### 7. Fault injection and the failover nemesis

  One nemesis crashes origin pods and Redis on a `k3d` cluster while load runs. A
  Redis failover nemesis kills the primary and promotes a replica. The checks assert
  that every subscription
  cursor reaches its stream tail within a settle window and that a deposed worker's
  late acknowledgement is fenced. The failover check asserts the central safety
  property. An acknowledged write that is lost on failover degrades only to
  at-least-once delivery, deduplicated by the consumer's monotone offset, and never
  to a safety violation.

  This gives you end-to-end evidence under real crashes. The downside is that the
  full multi-node run under load needs a managed cluster. The local run covers
  single-node failover, and the multi-node verdict is pending a cloud run.

  Where: [`jepsen/checker/nemesis.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/nemesis.go),
  [`jepsen/checker/scenario_failover.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/scenario_failover.go),
  and the [`jepsen/`](https://github.com/adityavkk/chronicle/blob/main/jepsen) tree.

  ### 8. Black-box conformance

  A vendored suite of about 330 tests, with `fast-check` property fuzzing, drives a
  live server over HTTP and checks it against the Durable Streams protocol. It covers
  idempotent producers, closure, forks, JSON mode, server-sent events, and the
  subscription APIs. The downside is that it sees only the HTTP surface, not the
  internal state.

  Where: [`test/conformance/conformance.test.ts`](https://github.com/adityavkk/chronicle/blob/main/test/conformance/conformance.test.ts).
  Run: `make conformance`, which needs Redis and a built server.

  ### 9. Machine-checked proofs in Lean 4

  The pure cores are transcribed into Lean and proven correct for every input,
  checked by Lean's kernel. The proofs cover the producer state machine, meaning its
  totality, determinism, epoch monotonicity, and idempotency. They cover the offset
  order, which is a strict total order, and the lexical-equals-numeric property below
  `10^16`. They cover cursor progression. They include a parametric lemma that a
  monotone token implies a single holder, instantiated for both the generation fence
  and the owner-epoch fence. The proofs rest only on the standard axioms `propext`,
  `Classical.choice`, and `Quot.sound`, with no `sorry` and no `native_decide`.

  This gives you correctness for every input, not for a sampled list. The downside is
  that a proof covers the Lean model, not the Go code directly. The differential
  oracle closes that gap.

  <Aside label="How it ties to the code" summary="A proof covers the Lean model, not the Go code directly">
    The proof is tied to the running code in two ways. The differential tests check
    the same <code>Stream-Seq</code> and producer rules, and the Lean model is
    compiled to C and run as the third differential oracle, so the proof and the Go
    code are pinned to one statement.
  </Aside>

  Where: [`lean/`](https://github.com/adityavkk/chronicle/blob/main/lean). Run:
  `cd lean && lake build`.

  ### 10. Model checking in TLA+

  The concurrent subscription protocol is specified in TLA+ and checked by TLC over
  every interleaving of workers, crashes, and lease expiries, up to a bounded size.
  The specs cover the wake, claim, acknowledge, and release fence, the owner-epoch
  layering, and liveness. The safety properties checked are single-holder, generation
  monotonicity, stale-operation inertness, the soundness of treating a fenced
  operation as a no-op, and forward-only cursors. The liveness properties are that
  pending work eventually issues a wake and that an expired lease is eventually
  reclaimed.

  This gives you safety and liveness over every interleaving up to the size checked.
  The downside is that TLC explores a bounded number of workers, so a passing run is
  evidence up to that size, not a proof for all sizes. Apalache removes that limit for
  the central property.

  <Aside label="Trace validation" summary="What binds the TLA+ spec to the running code">
    The running engine emits a log of its fence transitions, and TLC checks that each
    real trace is a behavior the spec allows. A tampered trace is rejected, which
    shows the check is not vacuous.
  </Aside>

  Where: [`formal/tla/`](https://github.com/adityavkk/chronicle/blob/main/formal/tla).
  Run: `cd formal/tla && make tlc`, then `make trace-validate`.

  ### 11. Inductive proof with Apalache

  Apalache proves single-holder as an inductive invariant by symbolic reasoning
  rather than enumeration. It shows the invariant holds in the initial state and is
  preserved by every action. This establishes single-holder for all sizes, not just
  the bounded number of workers TLC explores. A negative control confirms that an
  unsound lease expiry breaks the invariant, so the proof is not vacuous.

  This gives you the single-holder property for all sizes. The downside is that
  finding the inductive strengthening is manual work, and the proof is scoped to the
  fence core, not the whole engine.

  Where: [`formal/tla/FenceCore.tla`](https://github.com/adityavkk/chronicle/blob/main/formal/tla/FenceCore.tla).
  Run: `cd formal/tla && make apalache`.

  ### 12. Bounded relational checking with Alloy

  Alloy checks relational invariants over every configuration up to a scope. A scope
  is a fixed upper bound on the number of objects of each kind. The fan-out index
  equals the projection of the canonical links, so reconciliation never invents
  membership. The slot-homing scatter set equals the reference set, so no subscriber
  reads another's data. The slot-homing model also shows, formally, that never
  clearing the occupied-slots bit on de-index is a correctness requirement, not an
  optimization.

  This gives you an exhaustive check over every configuration within the scope. The
  downside is that Alloy is exhaustive only up to the scope it is given.

  Where: [`formal/alloy/`](https://github.com/adityavkk/chronicle/blob/main/formal/alloy).

  ### 13. Load, stress, and scale

  The load generator and the GKE rig measure throughput and latency, and sweep
  subscription recovery to tens of thousands of subscriptions on managed Redis. The
  full results are on the <a href={`${import.meta.env.BASE_URL.replace(/\/$/, "")}/load-testing`}>load and stress page</a>.
  The downside is cost. The rig runs on cloud infrastructure and must be torn down
  after each run.

  Where: [`loadgen/`](https://github.com/adityavkk/chronicle/blob/main/loadgen)
  and [`loadtest/`](https://github.com/adityavkk/chronicle/blob/main/loadtest).

<section class="block" id="line">
  <SectionHead num="03" label="The line" title="Proven, sampled, or <em>not provable</em>" />

  What Redis guarantees sets the line between what is proven, what is only sampled,
  and what cannot be proven. The three groups below say which kind of test carries
  each claim, so you can tell a proof from a sample and a sample from an honest gap.

  <Table wide>
    | Class | What is in it | How it is established |
    | --- | --- | --- |
    | Proven or checked exhaustively | the pure-core function properties; the single-holder fence for all sizes; per-stream linearizability of the data plane[^linz]; generation monotonicity; forward-only cursors | Lean proofs; Apalache induction; Porcupine over the single Lua commit point |
    | Sampled | the property and fuzz tests, which cover many reachable states but not all of them; the TLA+ runs, which cover a bounded number of workers | `rapid`, Go fuzzing, TLC up to a bounded size |
    | Not provable | durability of an acknowledged write across a failover[^durab] | tested with the failover nemesis and documented, not proven |
  </Table>

  Durability across a failover is separated from safety, so a lost write degrades to
  at-least-once delivery and never to a safety violation.

  <Callout icon="alert" label="Read a green check for exactly what it proves">
    A passing TLA+ run is evidence up to the number of workers TLC explored, not a
    proof for all sizes. A passing property run sampled the space. It did not cover
    it. A passing failover run is evidence under the crashes it injected, not a
    durability proof. The proofs in the first group are the only claims that hold for
    every input or every size.
  </Callout>
</section>

<section class="block" id="bugs">
  <SectionHead num="04" label="Bugs" title="Three bugs the <em>formal work</em> found" />

  The property and differential tests found three real defects that example tests
  had missed. Each is tracked with the `found-by-formal-methods` label, which names
  the method that found it and the intended fix.

  <Table wide>
    | Issue | The defect | Found by | State |
    | --- | --- | --- | --- |
    | [#47](https://github.com/adityavkk/chronicle/issues/47) | a producer sequence gap with a value at or above `10^14` makes the Redis backend error while parsing its own reply, instead of returning a clean 409 | the producer differential property | client reachable, so it is the most serious of the three |
    | [#46](https://github.com/adityavkk/chronicle/issues/46) | the offset string format uses a minimum width, so lexical order inverts against numeric order at or above `10^16` | the offset property | the fix is a format migration, so it is deferred |
    | [#48](https://github.com/adityavkk/chronicle/issues/48) | the in-memory read compares only the byte offset while Redis compares the full offset string, so they will diverge once log rotation makes the read sequence nonzero | the property test | [armed with a failing test](https://github.com/adityavkk/chronicle/blob/main/store/readseq_divergence_test.go) that fires when the code changes |
  </Table>
</section>

<section class="block" id="run">
  <SectionHead num="05" label="Run" title="How to run <em>everything</em>" />

  Run each layer in turn:

  ```bash title="run every layer"
  make test-unit                 # pure cores, no infrastructure
  make redis-up && make test     # unit and integration against Redis
  make conformance               # the black-box protocol suite
  cd lean && lake build          # the Lean proofs
  cd formal/tla && make tlc      # the TLA+ model checks
  cd formal/tla && make apalache # the Apalache inductive proof
  cd formal/tla && make trace-validate   # trace validation against the spec
  go test -tags leanoracle ./store/redis/   # the triple oracle, incl. the Lean model
  ```

[^linz]: Every mutation is a single Lua script over keys in one hash-tag slot, so writes to a stream are totally ordered and serialized on the Redis primary. That single commit point is the one place the order is decided, which makes the linearizability claim hold.

[^durab]: Redis replication is asynchronous, and `WAIT` and `WAITAOF` do not make it strongly consistent, so an acknowledged write can be lost on a primary failure. Safety rests only on the monotone fence, so the lost write degrades to at-least-once delivery rather than a safety violation.
