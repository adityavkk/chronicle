<section class="block" id="shape">
  <SectionHead num="01" label="Shape" title="One rule shapes the <em>whole suite</em>" />

  Chronicle splits each component into a pure deterministic core and an imperative
  shell that does input and output, then tests each part at the cheapest level that
  is still faithful. The cores are tested by proofs and by properties that hold for
  every input. The shell is tested against a real Redis. Properties that only
  appear across processes, such as a failover or a race between two workers, are
  tested with fault injection and linearizability checks. The wire protocol is
  checked from outside the code by a black-box suite.

  The table below lists the levels at a glance. Each row names what runs at that
  level, what the level gives you, and the infrastructure it needs. The sections
  after the table take each kind of test one at a time.

  <Table wide>
    | Level | What runs | What it gives you | Infrastructure |
    | --- | --- | --- | --- |
    | Pure core | unit tables, Lean proofs, the cursor and fence model checkers | per-input correctness, and for the proofs, correctness for every input | none |
    | Differential | the same input through Go, Lua, and the compiled Lean model | the three implementations agree on every input | real Redis |
    | Property and model based | `rapid` generates random operation sequences against two backends | the backends agree across reachable states, with a shrunk counterexample on failure | real Redis |
    | Fuzzing | Go's coverage-guided fuzzer drives the same model | reaches rare branches that random sampling misses | real Redis |
    | Linearizability | `porcupine` checks recorded concurrent histories | the concurrent history has a valid sequential order | real Redis |
    | Concurrent protocol | TLA+ model checking, Apalache induction, Alloy | safety and liveness over all interleavings, and for Apalache, over all sizes | Java |
    | Fault injection | a `k3d` cluster and a Redis failover nemesis | end to end safety under crashes and failover | k3d or local Docker |
    | Wire contract | the vendored conformance suite | the HTTP surface matches the protocol | a live server |
    | Scale | the load generator and the GKE rig | throughput and latency under load | managed Redis |
  </Table>

  <Callout icon="info" label="How to read each section">
    Every section that follows states four things: what the test is, what it gives
    you, where it lives, and its downside. The downside is the point. A test that
    is described as all upside is being oversold. The honest limit of each kind is
    what tells you which other kind covers the gap.
  </Callout>
</section>

<section class="block" id="kinds">
  <SectionHead num="02" label="Kinds" title="The thirteen kinds of <em>test and proof</em>" />

  ### 1. Unit tests on the pure cores

  Table-driven Go tests on the pure functions. They cover `ValidateProducer`, which
  is the idempotent producer state machine, `Offset.Compare` and the offset codec,
  `GenerateResponseCursor`, and the webhook reducers such as `FenceDecision` and
  `MergeAcks`. They run in milliseconds with no Redis and no clock, because the
  clock is passed in.

  They give you correctness on the specific cases you wrote. The downside is that
  they only check the cases you wrote, so they miss the inputs you did not think
  of. That gap is closed by the proofs and the property tests below.

  Where: the pure-Go cores under [`store/`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go)
  and [`webhook/`](https://github.com/adityavkk/chronicle/blob/main/webhook/state.go). Run: `make test-unit`.

  ### 2. Integration tests against a real Redis

  The `store/redis` and `webhook` packages run against a live Redis 8. Each uses
  its own database, flushes once at start, and skips cleanly when Redis is not
  reachable or under `-short`. These exercise the Lua scripts and the full
  read-write round trip that the in-memory store cannot.

  They give you confidence that the bytes actually move correctly through the real
  storage layer. The downside is that they need Redis, so they do not run in a bare
  unit pass.

  Where: [`store/redis/integration_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/integration_test.go)
  and [`webhook/redis_store_test.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/redis_store_test.go).
  Run: `make test`, which needs Redis. Run `make redis-up` first.

  ### 3. Differential tests

  The same input is sent through three independent implementations and the results
  are required to match. The three are the pure Go function, which is the oracle,
  the Lua script Redis runs, which is the mirror, and the Lean model compiled to C,
  which is the proof and is described in kind 9. The producer state machine, the
  fence and slot predicates, the content-type and config normalizers, the
  `Stream-Seq` comparison, and the JSON flatten path each have a differential test.

  This gives you confidence that the three copies of a rule have not drifted apart.
  A differential test found the `10^14` producer-reply bug, issue
  [#47](https://github.com/adityavkk/chronicle/issues/47). The Lua reply rendered
  large sequence numbers in scientific notation, which the Go reply parser
  rejected. The downside is that a differential test only catches a disagreement.
  If all three copies share the same wrong assumption, they agree and the test
  passes. That residual gap is covered by the independent properties and the
  proofs.

  Where: [`store/redis/differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_test.go),
  [`webhook/predicate_differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/predicate_differential_test.go),
  and [`store/leanoracle`](https://github.com/adityavkk/chronicle/blob/main/store/leanoracle),
  which is the compiled-Lean oracle behind the `leanoracle` build tag.

  ### 4. Property-based and model-based tests

  The `rapid` library generates random, shrinking sequences of operations and runs
  each operation against both backends. One backend is the in-memory `MemoryStore`,
  which is the reference model. The other is the live Redis backend, which is the
  subject. After every step the two must agree on the result, the error, the tail
  offset, the metadata, and the bytes a read returns. A single fake clock drives
  both, so expiry is reproducible.

  This gives you agreement across many reachable states rather than a fixed list,
  and when it fails it shrinks the failure to the smallest sequence that still
  breaks it. The downside is that it samples the space rather than covering it, and
  both backends call the same pure cores, so a bug in a shared core agrees in both.
  Independent metamorphic checks and the proofs cover that.

  Where: [`store/redis/equivalence_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/equivalence_test.go),
  [`store/redis/differential_producer_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_producer_test.go),
  and [`store/offset_property_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/offset_property_test.go).

  ### 5. Coverage-guided fuzzing

  The same `rapid` model is wrapped as a native Go fuzz target with
  `rapid.MakeFuzz`, so Go's coverage-guided fuzzer steers the generated sequences
  toward rare branches that uniform random sampling under-reaches. Examples are an
  epoch bump at a nonzero sequence or a fork sub-offset overshoot. A committed
  corpus of seeds, including four branch-verified fixtures, runs on every pull
  request, and a nightly job runs the fuzzer for a fixed time.

  This gives you a search that reaches branches random sampling keeps missing. The
  downside is that fuzzing finds crashers and divergences but proves nothing. It is
  a search, not a guarantee.

  Where: [`store/redis/equivalence_fuzz_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/equivalence_fuzz_test.go),
  the corpus under [`store/redis/testdata/fuzz/`](https://github.com/adityavkk/chronicle/blob/main/store/redis/testdata/fuzz),
  and [`.github/workflows/fuzz-nightly.yml`](https://github.com/adityavkk/chronicle/blob/main/.github/workflows/fuzz-nightly.yml).

  ### 6. Linearizability checking with Porcupine

  `porcupine` takes a recorded concurrent history and a sequential model and decides
  whether the history has a valid sequential order. Chronicle uses it for the lease
  fence, the owner-epoch slot fence, the stream data plane of append, read, and
  close, and a composed model that carries both fences at once.

  This gives you a guarantee that concurrent operations behaved as if they ran one
  at a time, which is the correctness condition Herlihy and Wing defined. A torn
  read, a lost append, or a reordering has no valid order and the checker prints the
  witness. The downside is that the search is NP-hard, which means the time it takes
  can grow very fast as the history grows. So it runs over bounded histories and a
  small number of workers, and a too-large history can return Unknown, which is
  counted as a failure.

  Where: [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go),
  [`jepsen/checker/model_shard.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_shard.go),
  [`jepsen/checker/model_store.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_store.go),
  and the `*-linz` scenarios. Run via the `jepsen-checker` binary.

  ### 7. Fault injection and the failover nemesis

  A nemesis crashes origin pods and Redis on a `k3d` cluster while load runs. A
  Redis failover nemesis kills the primary and promotes a replica. A nemesis is a
  process that injects faults on purpose. The checks assert that every subscription
  cursor reaches its stream tail within a settle window and that a deposed worker's
  late acknowledgement is fenced. The failover check asserts the load-bearing safety
  property. An acknowledged write that is lost on failover degrades only to
  at-least-once delivery, deduplicated by the consumer's monotone offset, and never
  to a safety violation.

  This gives you end-to-end evidence under real crashes. The downside is that the
  full multi-node run under load needs a managed cluster, so the local run covers
  single-node failover and the multi-node verdict is pending a cloud run.

  Where: [`jepsen/checker/nemesis.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/nemesis.go),
  [`jepsen/checker/scenario_failover.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/scenario_failover.go),
  and the [`jepsen/`](https://github.com/adityavkk/chronicle/blob/main/jepsen) tree.

  ### 8. Black-box conformance

  A vendored suite of about 330 tests, with `fast-check` property fuzzing, drives a
  live server over HTTP and checks it against the Durable Streams protocol. It
  covers idempotent producers, closure, forks, JSON mode, server-sent events, and
  the subscription APIs.

  This gives you protocol compliance independent of the implementation. The downside
  is that it sees only the HTTP surface, not the internal state.

  Where: [`test/conformance/conformance.test.ts`](https://github.com/adityavkk/chronicle/blob/main/test/conformance/conformance.test.ts).
  Run: `make conformance`, which needs Redis and a built server.

  ### 9. Machine-checked proofs in Lean 4

  The pure cores are transcribed into Lean and proven correct for every input,
  checked by Lean's kernel. The proofs cover the producer state machine, which
  means its totality, determinism, epoch monotonicity, and idempotency. They cover
  the offset order, which is a strict total order, and the lexical-equals-numeric
  property below `10^16`. They cover cursor progression. They include a parametric
  lemma that a monotone token implies a single holder, instantiated for both the
  generation fence and the owner-epoch fence. The proofs rest only on the standard
  axioms `propext`, `Classical.choice`, and `Quot.sound`, with no `sorry` and no
  `native_decide`.

  This gives you correctness for every input, not for a sampled list. The proof is
  tied to the running code in two ways. The same `Stream-Seq` and producer rules are
  checked by the differential tests, and the Lean model is compiled to C and run as
  the third differential oracle, so the proof and the Go code are pinned to one
  statement. The downside is that a proof covers the Lean model, not the Go code
  directly. The differential oracle is what closes that gap.

  Where: [`lean/`](https://github.com/adityavkk/chronicle/blob/main/lean). Run:
  `cd lean && lake build`.

  ### 10. Model checking in TLA+

  The concurrent subscription protocol is specified in TLA+ and checked by TLC over
  every interleaving of workers, crashes, and lease expiries, up to a bounded size.
  The specs cover the wake, claim, acknowledge, and release fence, the owner-epoch
  layering, and liveness. The safety properties checked are single-holder,
  generation monotonicity, stale-operation inertness, the soundness of treating a
  fenced operation as a no-op, and forward-only cursors. The liveness properties are
  that pending work eventually issues a wake and that an expired lease is eventually
  reclaimed.

  This gives you safety and liveness over every interleaving up to the size checked.
  Trace validation binds the spec to the code. The running engine emits a log of its
  fence transitions, and TLC checks that each real trace is a behavior the spec
  allows. A tampered trace is rejected, which shows the check is not vacuous. The
  downside is that TLC explores a bounded number of workers, so a passing run is
  evidence up to that size, not a proof for all sizes. Apalache removes that caveat
  for the central property.

  Where: [`formal/tla/`](https://github.com/adityavkk/chronicle/blob/main/formal/tla).
  Run: `cd formal/tla && make tlc`, then `make trace-validate`.

  ### 11. Inductive proof with Apalache

  Apalache proves single-holder as an inductive invariant by symbolic reasoning
  rather than enumeration. It shows the invariant holds in the initial state and is
  preserved by every action. This establishes single-holder for all sizes, not just
  the bounded number of workers TLC explores. A negative control confirms that an
  unsound lease expiry breaks the invariant, so the proof is not vacuous.

  This gives you the single-holder property for all sizes. The downside is that
  finding the inductive strengthening is manual work and the proof is scoped to the
  fence core, not the whole engine.

  Where: [`formal/tla/FenceCore.tla`](https://github.com/adityavkk/chronicle/blob/main/formal/tla/FenceCore.tla).
  Run: `cd formal/tla && make apalache`.

  ### 12. Bounded relational checking with Alloy

  Alloy checks relational invariants over every configuration up to a scope. A scope
  is a fixed upper bound on the number of objects of each kind. The fan-out index
  equals the projection of the canonical links, so reconciliation never invents
  membership. The slot-homing scatter set equals the reference set, so there is no
  cross-subscriber leakage. The slot-homing model also shows, formally, that never
  clearing the occupied-slots bit on de-index is a correctness requirement and not
  just an optimization.

  This gives you an exhaustive check over every configuration within the scope. The
  downside is that Alloy is exhaustive only up to the scope it is given.

  Where: [`formal/alloy/`](https://github.com/adityavkk/chronicle/blob/main/formal/alloy).

  ### 13. Load, stress, and scale

  The load generator and the GKE rig measure throughput and latency, and sweep the
  subscription recovery sweep to tens of thousands of subscriptions on managed
  Redis. The full results are written up on the <a href={`${import.meta.env.BASE_URL.replace(/\/$/, "")}/load-testing`}>load and
  stress page</a>.

  This gives you throughput and latency numbers under real load at scale. The
  downside is cost. The rig runs on cloud infrastructure and must be torn down after
  each run.

  Where: [`loadgen/`](https://github.com/adityavkk/chronicle/blob/main/loadgen)
  and [`loadtest/`](https://github.com/adityavkk/chronicle/blob/main/loadtest).
</section>

<section class="block" id="line">
  <SectionHead num="03" label="The line" title="Proven, sampled, or <em>not provable</em>" />

  The line between what is proven, what is only sampled, and what cannot be proven
  is set by what Redis guarantees. The three groups below say which kind of test
  carries each claim, so a reader can tell a proof from a sample and a sample from
  an honest gap.

  <Table wide>
    | Class | What is in it | How it is established |
    | --- | --- | --- |
    | Proven or checked exhaustively | the pure-core function properties; the single-holder fence for all sizes; per-stream linearizability of the data plane; generation monotonicity; forward-only cursors | Lean proofs; Apalache induction; Porcupine over the single Lua commit point |
    | Sampled | the property and fuzz tests, which cover many reachable states but not all of them; the TLA+ runs, which cover a bounded number of workers | `rapid`, Go fuzzing, TLC up to a bounded size |
    | Not provable | durability of an acknowledged write across a failover | tested with the failover nemesis and documented, not proven |
  </Table>

  Per-stream linearizability is in the proven group for a specific reason. Every
  mutation is a single Lua script over keys in one hash-tag slot, so writes to a
  stream are totally ordered and serialized on the Redis primary. That single commit
  point is the one place the order is decided, which is what makes the
  linearizability claim hold.

  Durability across a failover is the one thing that cannot be proven. Redis
  replication is asynchronous, and `WAIT` and `WAITAOF` do not make it strongly
  consistent, so an acknowledged write can be lost on a primary failure. The design
  is correct anyway because durability is separated from safety. A lost write
  degrades to at-least-once delivery, never to a safety violation, since safety
  rests only on the monotone fence. This is tested with the failover nemesis, not
  proven, and it is stated as a durability-honest property.

  <Callout icon="alert" label="Read a green check for exactly what it proves">
    A passing TLA+ run is evidence up to the number of workers TLC explored, not a
    proof for all sizes. A passing property run sampled the space, it did not cover
    it. A passing failover run is evidence under the crashes it injected, not a
    durability proof. The proofs in the first group are the only claims that hold
    for every input or every size.
  </Callout>
</section>

<section class="block" id="bugs">
  <SectionHead num="04" label="Bugs" title="Three bugs the <em>formal work</em> found" />

  The property and differential tests found real defects that example tests had
  missed. Each is tracked with the `found-by-formal-methods` label, which names the
  method that found it and the intended fix. There are three.

  <Table wide>
    | Issue | The defect | Found by | State |
    | --- | --- | --- | --- |
    | [#47](https://github.com/adityavkk/chronicle/issues/47) | a producer sequence gap with a value at or above `10^14` makes the Redis backend error while parsing its own reply, instead of returning a clean 409 | the producer differential property | client reachable |
    | [#46](https://github.com/adityavkk/chronicle/issues/46) | the offset string format uses a minimum width, so lexical order inverts against numeric order at or above `10^16` | the offset property | the fix is a format migration, so it is deferred |
    | [#48](https://github.com/adityavkk/chronicle/issues/48) | the in-memory read compares only the byte offset while Redis compares the full offset string, so they will diverge once log rotation makes the read sequence nonzero | the property test | armed with a failing test that fires when the code changes |
  </Table>

  Issue #47 is client reachable, which means a real client can trigger it, so it is
  the most serious of the three. Issue #46 needs a value at or above `10^16`, which
  is far beyond any value seen in practice today, and the fix is a format migration,
  so it is deferred rather than fixed now. Issue #48 cannot happen until log
  rotation makes the read sequence nonzero. It is held in place by a test that will
  fail the moment the code changes in a way that would trigger it, so it cannot
  regress silently.
</section>

<section class="block" id="run">
  <SectionHead num="05" label="Run" title="How to run <em>everything</em>" />

  The commands below run each layer in turn, from the pure cores that need no
  infrastructure up to the proofs and the model checks. Each line names what it
  runs.

  ```bash title="run every layer"
  make test-unit                 # pure cores, no infrastructure
  make redis-up && make test     # unit and integration against Redis
  make conformance               # the black-box protocol suite
  cd lean && lake build          # the Lean proofs
  cd formal/tla && make tlc      # the TLA+ model checks
  cd formal/tla && make apalache # the Apalache inductive proof
  cd formal/tla && make trace-validate   # trace validation against the spec
  go test -tags leanoracle ./store/redis/   # the triple oracle, incl. the Lean model
  ```
