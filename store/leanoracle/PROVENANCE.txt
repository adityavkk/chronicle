Vendored Lean->C differential oracle — provenance and regeneration
==================================================================
Issue #31 (P1.2). This file records exactly how the vendored artifact in this
directory was produced, so it is reproducible and the CI drift guard can assert
byte-identity. Routine Go CI consumes the vendored artifact and needs NO Lean
toolchain; only this regeneration step and the gated drift-guard job need Lean.

Regeneration (the command of record)
------------------------------------
  PATH="$HOME/.elan/bin:$PATH" store/leanoracle/scripts/build-lean-oracle.sh

  The script: lake build (emits C) -> compile each C with the SYSTEM clang at
  -fvisibility=default and -mmacosx-version-min=15.0 -> ld -r partial-link the
  three objects with the referenced members of libleanrt/libInit/libgmp/libuv,
  keeping the public entry points global -> ZERO_AR_DATE=1 ar to pack a
  deterministic (zero-timestamp) archive. The compile and partial-link steps are
  deterministic; ZERO_AR_DATE makes the ar step deterministic, so the archive is
  byte-identical across rebuilds on the same toolchain.

  NOTE: use the SYSTEM clang, not leanc. leanc bakes -fvisibility=hidden, which
  demotes the @[export] symbols to private and breaks cgo linking.

Drift guard
-----------
  PATH="$HOME/.elan/bin:$PATH" store/leanoracle/scripts/build-lean-oracle.sh --check
  rebuilds into a temp dir and asserts byte-identity with the committed archive.
