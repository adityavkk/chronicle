# dsui UI review — Make dsui more beautiful, more intuitive, and more capable — without leaving the lightweight track

Tracking epic: **#60**. Filed from a multi-agent review (6 dimensions, 41 findings) distilled to 10 prioritized, GitHub-ready issues. Every item is client-side only — the Durable Streams API stays frozen and the lightweight Preact + @preact/signals stack is preserved.

dsui is already a tasteful, hand-built Preact + signals console, but a multi-dimension review surfaced a clear pattern: the craft is uneven. A handful of moments feel polished (the start-screen radial wash, the brand badge, the entrance animations, the copy-as-curl disclosures) while the everyday surfaces underneath them — the message grid, the inspector idle states, the read toolbar, the popover menus — are flatter, lower-contrast, and missing the power-user affordances a streams console lives on. This epic closes that gap.

The work splits into three threads that match the team's stated priorities. **Aesthetics:** stop overloading the single indigo accent so status, kind, and link semantics are tellable apart at a glance, and widen a type scale that has collapsed into an 11–15px band so explanatory prose stops sitting at an 11px reading floor. **User experience:** fix the core publish flow, which today lands the user on a blank grid after a successful send, and make the surfaces around reading and writing self-explanatory (discoverable composer, actionable error toasts, keyboard-operable menus). **Capability:** add the table-stakes power-user tools the frozen read/write API can fully support client-side — in-grid search, export, a revisitable read-cursor history, and a Cmd/Ctrl-K command palette.

Every item here is deliberately in-bounds: pure client-side changes against the frozen Durable Streams API, no new runtime dependencies, and built in the existing hand-written idiom (semantic CSS tokens, signals in the store, small pure helpers in `lib/` with unit tests, the established roving-tabindex pattern). Several are token-only or single-function edits that lift the whole app at once.

The issues are ordered so the cheapest high-impact wins land first (the publish fix, the color and type passes) before the larger capability builds (command palette). Taken together they turn dsui from a clean-but-monochrome viewer into a console that feels intentionally designed and genuinely fast to work in.

## Themes

- Visual hierarchy: differentiate semantic color and widen the type scale so the UI reads as designed, not monochrome
- Fix and finish the core read/write flows so publishing, erroring, and writing-to-empty-streams behave the way users expect
- Add the power-user capabilities the frozen API already supports client-side: search, export, history, command palette
- Close the largest keyboard and contrast accessibility gaps with the patterns already proven elsewhere in the tree

### Visual design & aesthetics

- **Give status, kind, and link semantics their own colors instead of overloading the indigo accent** _(high, M)_  
  Introduce two more semantic accent slots in tokens.css derived from existing oklch hues (e.g. a teal/cyan around hue 195 and a violet-distinct hue, both with -soft variants mirroring --accent-soft), and reassign: keep --accent reserved for the primary action / selection only; map content-type kind chips to a stable per-kind hue (json=accent, text=info, binary stays neutral is fine), give the webhook vs pull-wake subchips genuinely different hues (they are the two delivery mechanisms and should be tellable apart at a glance), and let the method chips keep their semantic mapping but stop sharing accent-soft with kind chips. Pure CSS token + class edits in tokens.css and app.css; no component changes.
- **Widen the type scale and stop overloading --text-2xs (11px) for readable body copy** _(high, M)_  
  Reserve --text-2xs strictly for all-caps eyebrow labels and chips (where the letterspacing + uppercase compensate), and promote the explanatory prose currently on it (kvnote, proto note, primer body, playground does/lead, metric tile note, sighint) up to --text-xs. Add a --weight-bold (700) and apply it to .dsui-metrictile__value and section ledes so numeric/headline content gains a real display tier. Keep clamp-based fluid sizing. This is a search-and-replace within app.css plus one token addition; it tightens vertical rhythm without touching layout.
- **Make the bare placeholder/empty states match the crafted firstrun empty state** _(medium, S)_  
  Wrap the Inspector Raw/Headers idle states in the same <div class="dsui-empty"> structure used elsewhere, give each a relevant icon (IconCode for Raw bytes, an icon for Headers, IconStream for 'no row selected') and a title + hint, matching the firstrun treatment. Optionally add a faint --bg-raised card behind idle panels so the empty pane reads as a deliberate surface, not blank space. Pure JSX class/markup changes in Inspector.tsx reusing existing .dsui-empty classes; no new CSS needed beyond an optional centering tweak.
- **Add depth and a quiet gradient to the header and three-pane seams** _(medium, S)_  
  Deepen the rail/inspector separation: make --panel-2 a touch more distinct from --bg in light mode (or give the nav/inspector regions a subtle inset shadow on their inner edge using the existing --shadow-sm), and add a faint top-edge inner highlight to the header (inset 0 1px 0 of a low-alpha white in light, transparent in dark) so it reads as a raised surface like the brand badge does. Carry a very low-intensity version of the start screen's radial accent wash into the workspace background. All achievable with the existing shadow tokens and color-mix; edits confined to .dsui-header, .dsui-region--nav, .dsui-region--inspector in app.css and possibly the panel-2 token.
- **Smooth the center-view and selection transitions that currently snap** _(medium, M)_  
  Give the routed center workspace the same dsui-proto-reveal-style entrance: add a short keyed fade/translate on the CenterWorkspace root when centerView changes (a CSS animation on the .dsui-ws / workspace root, re-triggered via a key or data-view attribute). Apply a brief reveal to the inspector tabpanel on tab change the same way. Ensure every new motion is already covered by the global prefers-reduced-motion rule in base.css (it is). Reuse existing --dur-* / --ease-out tokens and the existing reveal keyframes; no new dependencies.
- **Reconcile the divergent --warn tokens so amber is consistent across light and dark** _(low, S)_  
  Pick one amber and keep it consistent: either raise the light --warn lightness to stay legibly amber (keep hue near 80, lightness ~62-66%) so it is clearly distinct from --danger, or define a single --warn-600 step and reference it from both light and dark semantic tokens the way --ok and --danger already do. Verify the warn-soft halo and the warn text-on-soft contrast hold in both themes. A few-line edit in tokens.css only.
- **Replace the bullet-character protocol-header marker with a real inline-SVG glyph** _(low, S)_  
  Add a tiny IconDot (a single filled or stroked <circle> in the existing Icon wrapper) to icons.tsx and use it for both the per-row marker and the legend dot, sized via the size prop and colored with --accent. This makes the marker crisp at any zoom and brings the Inspector in line with the documented inline-SVG idiom. Small additions to icons.tsx and Inspector.tsx; the .dsui-headers__star rule can be dropped.

### Interaction & UX flows

- **Stop publishing into a blank grid: refresh the just-published messages, not the empty tail** _(high, S)_  
  In store.appendMessages, after a successful publish to the selected stream, re-read from a position that actually shows the appended rows instead of from the tail. The cleanest hand-written fix: capture the read's prior nextOffset (the cursor that was the tail before this append) and re-read from there so the new batch lands in the grid; or, simpler, re-read using the toolbar's current resolved offset (readFromToolbar) rather than the hardcoded "now". Keep the demo-producer path (which intentionally tails) as-is. No new state — reuse lastRead.nextOffset / resolveOffset that already exist.
- **Make failure toasts actionable: surface the under-the-hood request and a one-click retry** _(medium, M)_  
  Add an inline action to error/warning toasts in the write actions: a 'Show details' action that expands the Under the hood panel (e.g. a store signal protocolOpen the ProtocolPanel <details> binds its open to), or a 'Retry' action that re-invokes the same store action. For the producer-conflict warning, add a 'Use expected seq' action that sets producerSeq to the server's expectedSeq so the next publish succeeds. All of this rides the existing Toast.action seam and existing store actions — no new infra.
- **Let the connection form save without forcing a manual Test, and auto-test on a valid URL** _(medium, S)_  
  Keep Save & connect as the single primary action but have it run the probe inline: on submit, if untested, kick off testCandidate and show the existing <output> result; if the probe fails, still allow saving but downgrade the button to a confirming 'Save anyway' state rather than blocking. Optionally debounce an automatic test once baseUrl validates (the form already invalidates the test result on edit via onEdit, so the plumbing exists). This collapses two clicks into one without removing the explicit Test affordance for power users.
- **Auto-open the publish composer when a stream is empty, so creation flows straight into writing** _(medium, S)_  
  Drive the composer's open state from a signal rather than leaving it a static collapsed <details>: open it automatically when the selected stream has no messages (lastRead rows empty / freshly created), and remember the user's manual collapse otherwise. Follow the established persisted-toggle idiom (playgroundOpen / inspectorCollapsed) — a store signal plus the <details open> binding — so it stays in the hand-written stack. Even just defaulting the composer open on a freshly created stream (createStream already calls selectStream) is a high-value S change.
- **Disclose the live curl on the connection card / switcher so copy-as-curl reaches the read path too** _(low, M)_  
  Add a CurlPreview next to the read toolbar's primary button, fed by a pure preview of the GET (origin + streamRoot + path + resolved offset), mirroring how the publish composer shows its curl before sending. This needs a small preview helper alongside the existing previewTailOperation/previewAppendOperation in lib (e.g. previewReadOperation) so the read curl is exact and unit-testable, then a CurlPreview in the Toolbar. It makes the most-used flow as learnable as the write flows.
- **Add a Fork entry point from the read pager/offset, where the user actually decides where to fork** _(low, S)_  
  Add a small 'Fork here' affordance in the workspace head's batch-offset readout (the dsui-ws__offsets block) and/or next to the pager, calling openForkDialog(stream.path, read.nextOffset). It reuses the existing store action and seed; it just puts the entry point next to the offset the user is staring at, turning fork from a hidden menu item into an obvious in-context action.
- **Reduce add-stream confusion: signal that a manually added path is unverified and may not exist** _(low, S)_  
  When a manual add yields a 404/network read, make the empty state explicit that this is an unverified manually-added path (e.g. distinguish status===0 / 404 on a manual stream with copy like 'This path was added by hand and the server has no stream here yet — create it, or check the path') and offer a 'Create this stream' action that opens the create dialog pre-filled with the path. The store already knows stream.manual and the exchange status; this is copy + one action button in the existing MessagesWorkspace empty-state branch.

### Capability & feature gaps

- **Add an in-grid message search/filter over the loaded batch and the tail buffer** _(high, M)_  
  Add a pure matcher in lib/messages.ts, e.g. rowMatches(row, query, opts) that tests query against row.preview, the decoded JSON value (stringified), and the formatted time, with a plain substring default and an optional /regex/ form. Add a component-local useSignal filter input to the dsui-ws__grid header (mirroring Navigator's dsui-nav__filter idiom: IconSearch + input, instant, never touches the store). Render only matching rows, keep the # index honest (it is the element's batch index, not a row counter), and show a 'showing N of M' note + a clear button. Reuse the same input in TailPanel's dsui-tail__bar so a fast tail can be filtered live without dropping the connection. Keep roving-tabindex correct over the filtered subset.
- **Export/download the current batch or tail buffer as NDJSON / CSV / raw bytes** _(high, M)_  
  Add a pure lib/export.ts with rowsToNdjson(rows), rowsToCsv(rows) (index, byteSize, time, preview/value columns; quote-escape per RFC 4180), and a tiny downloadBlob(filename, mime, data) helper guarded for missing document. Add a small 'Export' control to the pager row in MessagesWorkspace (next to 'Read next batch') and to TailPanel's dsui-tail__foot, offering NDJSON / CSV, plus 'Save raw body' for read.rawBytes (the exact bytes, mirroring the Inspector's Raw view) using the stream kind to pick the extension. Name files from the stream path + offset so successive exports are distinguishable.
- **Keep a per-stream read history of visited offset cursors with one-click revisit** _(high, M)_  
  Add a bounded readHistory signal in store.ts (a capped array of {path, requestedOffset, nextOffset, rowCount, at}) appended in readSelected() on success, reset on stream/connection switch. Render a compact 'History' strip in the MessagesWorkspace head or pager (a row of clickable offset chips, newest last) whose click calls readSelected(entry.requestedOffset). Also expose a 'copy cursor' affordance on the honest batch-offset range already shown in dsui-ws__offsets so a user can grab a cursor for a script. Keep it per-connection ephemeral (no persistence needed); cap it like TAIL_BUFFER_CAP.
- **Add a per-row Copy (value + curl-to-reproduce-this-read) directly in the grid and tail rows** _(medium, S)_  
  Add a small CopyButton (icon variant, the existing component) to the Row and TailRow layouts, revealed on hover/focus via CSS (a dsui-row__copy class, opacity transition behind prefers-reduced-motion), copying the row's decoded value (prettyJson for json, String for text, or a hex/base64 note for binary — reuse the Inspector's prettyJson/decode logic by lifting it into lib). Stop click propagation so copying does not also change row selection. This reuses CopyButton's existing single-confirmation keying (copyKey per row index).
- **Add a keyboard command palette (Cmd/Ctrl-K) for stream jump and core actions** _(medium, L)_  
  Build a CommandPalette component (reuse the existing Modal shell for focus-trap + Escape) opened by a global keydown listener registered once in App (Cmd/Ctrl-K, ignored while typing in an input). Source its items from store signals: every stream (selectStream), every tracked subscription (selectSubscription), Metrics (setCenterView), New stream / New subscription (open*Dialog), Read / Start tail / Stop tail, theme cycle, inspector toggle. Filter items with a plain hand-written substring matcher (no fuzzy library), arrow-key roving over results matching the existing tree pattern. All actions already exist in the store; the palette is pure layout over them.
- **Support pinned multi-stream tabs so reads survive switching between streams** _(medium, L)_  
  Add an openStreams signal (an ordered list of paths) plus a per-path read-state map (lastRead, startMode, customOffset, selectedRow, tailMode) in store.ts; selectStream pins the path into openStreams and restores its saved state instead of always resetting to Earliest. Render a tab strip above the MessagesWorkspace head (the Inspector tab-strip roving-tabindex pattern is the template per AGENTS) with a close affordance per tab and a '+' that focuses the Navigator filter. Switching connections clears the set. Keep the single active tail (only the foreground stream tails) to avoid leaking EventSources.
- **Add a two-message diff view for comparing JSON/text elements** _(low, M)_  
  Add a lightweight pin: a 'Compare' affordance on a row that stashes it as a diffBaseline (a local module signal), then a 'Diff' tab in the Inspector that renders a line-by-line (or JSON-key) diff of the baseline vs the currently selected row. Write the diff as a small pure lib/diff.ts (a Myers or even simple LCS line diff over prettyJson output) with a .test.ts, and render added/removed lines with semantic tokens (--ok / --danger tints). No library — a compact LCS is a few dozen lines and matches the hand-written-helper philosophy.

### Accessibility

- **Make the role=menu popovers keyboard-operable: move focus in and wire arrow keys** _(high, M)_  
  In StreamActionsMenu.tsx and ConnectionManager.tsx, on open, focus the first menuitem (a useEffect that queries the first [role^="menuitem"] in the popover ref and calls .focus()). Add an onKeyDown on the popover that handles ArrowDown/ArrowUp (move focus to next/prev menuitem, wrapping), Home/End (first/last), and give menuitems a roving tabIndex (0 on the focused item, -1 otherwise) so Tab does not leak out — mirror the activateAt/onTabKeyDown helpers in Inspector.tsx and the focusItem/onItemKeyDown helpers in Navigator.tsx. Escape-restores-focus is already correct in both; keep it.
- **Raise the contrast of --fg-subtle small text to meet WCAG AA** _(high, S)_  
  In tokens.css, darken --fg-subtle one ramp step for light mode — point it at --gray-600 (oklch 47%, ~6:1) or define a dedicated value around oklch(50%); verify the dark-mode --fg-subtle (currently --gray-500 on --gray-950) clears 4.5:1 too and bump it to --gray-400 if not. Because every component reaches the token rather than a raw step, this is a one-line-per-mode change that lifts all 130+ usages at once. Keep --fg-muted as the brighter tier; the goal is that no text token falls below AA on its intended background.
- **Add a skip-to-content link so keyboard users can bypass the Navigator** _(medium, S)_  
  In app.tsx, render a 'Skip to content' anchor as the first child of the shell (href="#dsui-main") and give the <main> id="dsui-main" plus tabIndex={-1} so the anchor target can take focus. Style it with the existing .sr-only helper in base.css plus a :focus rule that brings it on-screen (position fixed, top/left spacing, --bg-raised, --shadow-md) — the same visually-hidden-until-focused idiom already in the codebase. No new dependency, a dozen lines of hand-written CSS.
- **Keep the roving tab stop and selection in sync on the message listbox (and tie selection to the inspector)** _(medium, M)_  
  In MessagesWorkspace.tsx and TailPanel.tsx, after a successful read/next-batch, restore focus to the tabbable row (the focusRow(0) helper already exists — call it in an effect keyed on the batch identity when the grid had focus). Add aria-activedescendant on the listbox container pointing at the active option's id (give each Row a stable id). Give the Inspector body (the role="tabpanel" in Inspector.tsx) an aria-live="polite" or, better, move focus to the inspector heading when a row is activated, so activating a row produces an announcement. Keep all of this in the components reading existing store signals.
- **Give the Modal a real focus trap that includes the close button and reflows when the body changes** _(medium, M)_  
  In Modal.tsx, recompute the focusable list fresh on each Tab (already done) but drop the offsetParent filter in favor of checking the element is not disabled and has a non-negative tabindex, so disclosures that expand mid-dialog are handled. To address background reachability without showModal (which would fight the custom backdrop), set aria-hidden="true" (or inert, where supported) on the shell root while a dialog is open — drive it from the same activeDialog signal app.tsx already switches on, toggling an attribute on the .dsui-shell wrapper. Keep it hand-written; no library.
- **Announce read results and the selected-stream change to screen-reader users** _(medium, S)_  
  Add one polite aria-live region to MessagesWorkspace.tsx (a visually-hidden span, using the existing .sr-only class) that reflects a short result summary derived from lastRead — e.g. 'Read 12 rows from playground/demo, up to date' or 'Server responded 404'. Update it whenever lastRead changes. This reuses the same pattern Toaster already proves; it needs no store change, just a computed string rendered into an sr-only live span. Optionally wire the same span to announce the selected stream path on change.

### Responsiveness & performance

- **Window the catch-up grid and the live-tail list so a full batch/buffer is not 1000 live DOM rows** _(high, M)_  
  Add a tiny hand-written fixed-height windowing helper (rows are a uniform 30px — block-size:30px on .dsui-row — which makes this trivial and keeps to the lightweight idiom: no library). Put the pure math in a new lib/ helper (e.g. lib/virtual.ts: given scrollTop, clientHeight, rowHeight, total -> {startIndex, endIndex, padTop, padBottom}) with a unit test, and have both grids render only the visible slice inside a top/bottom spacer div. Keep the role=listbox/option semantics and roving tabindex on the rendered rows. The sticky header (.dsui-grid__header) and the tail's stick-to-bottom logic both keep working since total scrollHeight is preserved by the spacers. Gate it behind a small threshold (e.g. only window when total > ~120 rows) so small reads stay simple.
- **Make the tail-buffer append amortized O(1) instead of copying the full 1000-row buffer on every batch** _(high, S)_  
  Trim before concatenating, and only allocate when actually over cap: if batch.rows.length >= TAIL_BUFFER_CAP keep just its tail slice; otherwise compute the keep-count from the existing buffer and build one array of exactly TAIL_BUFFER_CAP. Better still, drop in larger chunks (e.g. evict in blocks of ~10% of the cap rather than exactly `overflow` each time) so steady-state appends usually hit the cheap `else` branch and the expensive slice runs rarely. Keep tailDropped accounting correct. This is a localized change to the one function with no API or component impact.
- **Cap the per-message decoded body (pretty-printed JSON / hex) the Inspector renders** _(medium, S)_  
  Give prettyJson a max-length budget like hexDump's: stringify, and if the result exceeds ~64-100 KB, render the truncated head with a `… (N more characters, full bytes in Raw)` footer line (the exact-bytes Raw view already exists for the complete payload). Keep the full string available to the Copy button so copy is still complete, or have copy fall back to the Raw bytes. Pure change in the one component; mirrors the existing hexDump truncation idiom.
- **Give the live tail an incoming-batch coalescing step so a burst of SSE events is not one render per message** _(medium, M)_  
  Buffer incoming tail rows in a module-level scratch array in the store and flush to the tailRows signal on a requestAnimationFrame (or a short setTimeout) coalescer, so multiple batches within a frame become a single signal write and a single render. Keep the cap/drop accounting in the flush. This is a store-only change behind the existing startTail/stopTail seam (clear the pending rAF in stopTail), invisible to components, and pairs naturally with the O(1) append fix. Honor prefers-reduced-motion as today by keeping the new-row flash unchanged.
- **Add an intermediate breakpoint and let the toolbar/composer survive high zoom and tablet widths gracefully** _(medium, M)_  
  Add a @container shell (max-width: ~860px) (or a workspace container query, since .dsui-region--workspace is already a named container) that: tightens toolbar gaps and lets the Rows select and segmented labels shrink/hide their uppercase text labels to icons-or-shorter; and switches the timed grid template to drop or narrow the Size/Time tracks (e.g. grid-template-columns: 3.5rem 1fr with size folded into the preview line) so the preview stays readable. Keep it token-driven and hand-written. This is independent of the in-flight left-rail overlap fix (that is the nav rail; this is the center toolbar + grid).
- **Stack the Wake Monitor split below the breakpoint where each pane is too narrow, and make stacked panes height-robust** _(low, S)_  
  Raise the stack breakpoint (e.g. ~820-860px) so panes stack before either is uncomfortably narrow, and in the stacked rule replace grid-auto-rows: minmax(0,1fr) with content-driven rows (e.g. auto rows with a min-block-size floor per pane) so each pane keeps a usable minimum height and the container scrolls, rather than dividing a short viewport in half. CSS-only, in the existing token idiom.
- **Bound the SSE per-batch row count so one giant frame cannot inject thousands of rows at once** _(low, S)_  
  Apply a per-batch ceiling in pushTailBatch (or where decodeSseBatch hands off): if a single batch exceeds, say, TAIL_BUFFER_CAP, keep only its last TAIL_BUFFER_CAP rows and bump tailDropped by the remainder, so a pathological frame can never allocate far past the buffer size. Pure store-side guard, consistent with the catch-up cap, no protocol change.

### Consistency & design-system / code quality

- **Extract a shared Field primitive for the label/control/error markup repeated across every form** _(high, M)_  
  Add `components/Field.tsx` exporting a small `Field`/`TextField` layout primitive (typed props: `id`, `label`, `required?`, `value`, `error`, `hint`, `mono?`, input attrs) that renders the exact existing `dsui-field` markup and centralizes the `aria-invalid`/`aria-describedby` switch and the `-err`/`-hint` id derivation. It stays pure layout (no store, no fetch) in the AGENTS.md 'small sub-components / shared primitives' idiom — the same justification that produced `Modal` and `CurlPreview` — and reuses the existing CSS classes unchanged. Migrate the forms one at a time.
- **Unify the three hand-rolled live-status badges into one StatusBadge primitive** _(high, M)_  
  Add `components/StatusBadge.tsx` taking `{ status, label?, variant? }` that owns the tone derivation, the pulse rule, the aria-live politeness, and the dot+label+detail markup once. Have all three call sites render it (the ProtocolPanel 'pill' becomes a `variant` prop). Collapse `dsui-wm__feedstatus` to reuse `dsui-tail__status--{tone}` (it already piggybacks on it) and decide one canonical tone→color set so the proto pill stops re-mapping. Keep all classification in `lib/tail`.
- **De-duplicate the verbatim dark-theme token block between the media query and the [data-theme] override** _(high, S)_  
  Hoist the dark values into a single reusable declaration. The cleanest hand-written option with no tooling: define the dark overrides once on a shared selector list — `@media (prefers-color-scheme: dark){:root:not([data-theme="light"]){…}} :root[data-theme="dark"]{…}` cannot share a body directly, so instead put the dark values in one `@layer`/class-free custom-property group applied via a single selector group where the cascade allows, or factor the dark values into a documented block that both selectors `@import`-free reference by duplicating only the selector, not the body. Practically: keep one authored source block and a short comment forbidding edits to a generated twin, or restructure so `[data-theme="dark"]` and the media query both target one rule. At minimum, add a guard comment pairing the two so they cannot drift unnoticed.
- **Pick one disclosure idiom — native <details> with a real chevron icon — and stop re-implementing the caret per component** _(medium, M)_  
  Standardize on one `Disclosure` shape: native `<details class="dsui-disclose">` with the shared `IconChevronRight` + a single `dsui-disclose__caret` rotation rule, matching `dsui-curl`. Replace the `::before "▸"` glyph with the SVG so all carets are visually identical and themed via currentColor. Reconcile ForkDialog to the same native idiom unless it genuinely needs the controlled-button behavior; if it does, document why so it is the exception, not a silent fork.
- **Promote MetaRow to one shared primitive instead of two divergent copies plus a hand-inlined third** _(medium, S)_  
  Add `components/MetaRow.tsx` with `{ label, children, as?: 'dl' | 'plain' }` (default the `<dt>/<dd>` semantics; the `<dl>` wrapper stays the caller's choice via a sibling `MetaList`). Replace both local definitions and the WakeMonitor inline rows with it. Pure layout over the existing `dsui-meta__*` classes — same idiom as the other shared primitives.
- **Tokenize the modal scrim — the one raw oklch literal in component CSS** _(medium, S)_  
  Add a `--scrim` semantic token to `tokens.css` (e.g. light and dark values built from the existing oklch model, like the `--shadow-*` tokens already are) and change `.dsui-modal { background: var(--scrim); }`. This restores token discipline for the shared dialog shell and lets the backdrop darken correctly per theme. Consider the same treatment for any popover/menu backdrop that should match.
- **Make the CurlPreview labels follow one naming convention** _(low, S)_  
  Settle on the dominant pattern — 'Equivalent curl' as the bare label and 'Equivalent curl — <operation>' when disambiguating — and update the two `StreamActionsMenu` call sites ('curl for close' → 'Equivalent curl — close', etc.). Likewise align the `CopyButton` `label` strings to one phrasing. Trivial text-only change, no behavior impact.
