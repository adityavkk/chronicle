# Lean Squad: automating the Lean proof track with near-zero human labour

*Don Syme's [Lean Squad](https://dsyme.net/2026/04/20/lean-squad-automated-software-verification-with-near-zero-human-labour/) is a scheduled GitHub Agentic Workflow that drives a single off-the-shelf coding agent through the whole formal-verification pipeline — research, spec, Lean definitions, proofs, self-critique, CI — against a real codebase, with the human only installing the workflow, merging PRs, and posting occasional one-line nudges. It is an orchestration and economics contribution, not a new prover: every proof is still machine-checked by the Lean kernel, but every definition and theorem statement is AI-written and unverified, which is exactly where its trust concentrates. Chronicle's Lean track already does by hand what Lean Squad automates, over a much smaller and friendlier target (three total pure cores, [issue #29](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md) done). This note reads Syme's experiment against our [DESIGN.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/DESIGN.md) and [ROADMAP.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md), places it in the agentic-Lean landscape, separates its evidence from its hype, and turns it into concrete changes to how we drive [#30](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md) (the proofs) and [#31](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md) (the Lean→C differential oracle).*

## What Lean Squad is

The source was reachable; everything below comes from the retrieved article on [dsyme.net](https://dsyme.net/2026/04/20/lean-squad-automated-software-verification-with-near-zero-human-labour/), corroborated by the [GitHub Next mirror](https://githubnext.com/posts/dsyme-lean-squad-automated-verification/), the [`githubnext/agentics` workflow docs](https://github.com/githubnext/agentics/blob/main/docs/lean-squad.md), and the [`dsyme/raft-lean-squad` repository](https://github.com/dsyme/raft-lean-squad).

**The form factor.** Lean Squad is a [GitHub Agentic Workflow](https://github.blog/ai-and-ml/automate-repository-tasks-with-github-agentic-workflows/) (`gh-aw`), installed with two commands: `gh extension install github/gh-aw`, then `gh aw add-wizard githubnext/agentics/lean-squad`. Once installed it runs as a scheduled GitHub Actions job — every 8 hours by default. Each run reads a persistent repo-memory and a GitHub status issue to recover the target state, phase status, and prior decisions; draws tasks from a phase-weighted probability distribution over a fixed set of task types; installs the Lean toolchain; executes the tasks (researching, writing Lean files, running `lake build`); opens a pull request with the output; and updates the status issue. The default coding agent is GitHub Copilot CLI with automatic model selection; Claude Code, Codex, or Gemini CLI are drop-in alternatives. Syme priced the runs against Claude Opus 4.6 at roughly **$7 per long run**.

**The method.** A single LLM coding agent does the whole pipeline. The article's task types are: Research, Informal Spec, Formal Spec, Implementation, Proof, Correspondence Review, Critique, CI, Report, and Paper (the `agentics` doc lists a slightly larger set including a final status task). The phase-weighted scheduler adapts on its own: when no formal-verification work exists yet, Research dominates; as specs accumulate, Proof rises; once proofs land, the reflective tasks (Critique, Correspondence Review) gain weight. Proof iteration is fully automated — the agent reads `lake build` / compiler feedback and refines in a loop, marking incomplete spots with `sorry` and working to drive the `sorry` count to zero. Humans steer only through GitHub issue comments in the form `/lean-squad COMMENT`, giving high-level direction without doing any proof work.

**Autoformalization had two paths.** Path A uses [Aeneas](https://github.com/AeneasVerif/aeneas) to translate Rust to Lean automatically; Path B has the agent write fresh Lean definitions by hand and document the implementation-to-model correspondence in a `CORRESPONDENCE.md`. Aeneas (Path A) failed on the first two codebases, so Path B was used for both — which is the source of the central caveat below.

**The near-zero-labour claim.** Syme states it first-person: *"I never selected a target, wrote a specification, suggested a proof strategy, or debugged a failing proof. I didn't need to know Lean. I didn't need to know formal methods. I just needed to design the workflow and merge PRs."* For the raft-rs run his *"total intervention: three GitHub nudge comments."* The mental model he sells is *"you just decide you want some formal verification done, so you hire a team to do it"* — *"Fully automated end-to-end software verification squads for hire. Maybe it's useful. Maybe it's not 🤷."*

**The trust anchor.** *"The proofs are real. Every theorem is machine-checked by Lean's kernel"* — even though *"every definition and every statement of every theorem is AI written."* Because the kernel checks the proof, training-data overlap cannot manufacture a false positive *for a proof*; it can only produce a true proof of a possibly-meaningless statement. That distinction is the whole story.

**Results reported in the article.**

| Target | Runs | Theorems / files | Bugs found | Notes |
| --- | --- | --- | --- | --- |
| [raft-rs](https://github.com/dsyme/raft-lean-squad) (Rust Raft) | 34, ~3 weeks, ~$240 | 443 / 23 | 0 code, 1 formulation bug (caught via `sorry` discipline) | Headline `raftReachable_safe` reached with zero `sorry` but resting on **5 axiomatized transition hypotheses**; ~60–70% of a self-contained safety proof. |
| Cloudflare quiche (Rust QUIC/HTTP-3) | 85 (ongoing) | 518 / 24 | 1 real `Ord` antisymmetry bug in `StreamPriorityKey::cmp` | Concurrency, crypto, network I/O unmodelled. |

That is 1,200+ machine-checked theorems and a handful of genuine bugs. **Caveat on the numbers:** the live `dsyme/raft-lean-squad` [REPORT.md](https://github.com/dsyme/raft-lean-squad/blob/main/formal-verification/REPORT.md) has since advanced past the article snapshot (it now claims 716 theorems across 79 files, 129 runs, 0 `sorry`, all 5 step hypotheses discharged, headline `fullProtocolStep_safe`). The article itself describes the earlier, conditional state; we cite the article's snapshot as the experiment and the repo as the moving target.

## Where it sits

Lean Squad's novelty is **orchestration and economics layered on a general agent**, not proving capability. It does not ship a prover model or a tactic. That places it differently from the rest of the agentic-Lean landscape:

- **In-IDE neural assistants — [Lean Copilot](https://github.com/lean-dojo/LeanCopilot).** The human-in-the-loop baseline. Lean Copilot ([arXiv 2404.12534](https://arxiv.org/html/2404.12534)) exposes LLMs inside Lean 4 for tactic suggestion, LLM-guided proof search on top of [Aesop](https://dl.acm.org/doi/10.1145/3573105.3575671), and premise selection from mathlib, with a pluggable model interface. Lean Squad's autonomous scheduled loop generalizes exactly this interaction, removing the human from each step.
- **Foundational infrastructure — [LeanDojo + ReProver](https://github.com/lean-dojo/ReProver).** LeanDojo ([arXiv 2306.15626](https://arxiv.org/pdf/2306.15626)) gives a programmatic, Gym-like environment to step Lean proof state and a retrieval-augmented prover doing premise selection over large libraries. Most later Lean ML work builds on it; the "step the kernel, capture the proof state, repair" pattern is the engine any squad loop closes against.
- **RL high-water mark — [AlphaProof](https://deepmind.google/blog/ai-solves-imo-problems-at-silver-medal-level/).** DeepMind's silver-medal IMO-2024 system ([Nature 2025](https://www.nature.com/articles/s41586-025-09833-y)) trains against the Lean verifier with RL and self-generated problems. It shows kernel-as-ground-truth self-play can solve hard problems, but it is a bespoke math-competition prover, not code verification and not workflow orchestration — far heavier than small total pure cores need.
- **Whole-proof generation provers — DeepSeek-Prover ([V1.5](https://arxiv.org/abs/2408.08152)/[V2](https://github.com/deepseek-ai/DeepSeek-Prover-V2)), [Goedel-Prover-V2](https://ar5iv.labs.arxiv.org/html/2508.03613), Kimina-Prover.** Specialized engines an agent could call as a Proof-task subroutine. Lean Squad deliberately does *not* use them — a form-factor choice in favor of a general CLI agent.
- **Classical automation — Aesop and the Lean Hammer.** [Aesop](https://dl.acm.org/doi/10.1145/3573105.3575671) is Lean 4's white-box best-first proof search; the [LeanHammer](https://cmu-l3.github.io/lean-hammer/) stack (`lean-auto` [arXiv 2505.14929](https://arxiv.org/pdf/2505.14929) + [duper](https://github.com/leanprover-community/duper)) bridges Lean goals to external ATPs/SMT and reconstructs the result in-kernel, with [ML premise selection](https://arxiv.org/html/2506.07477) feeding it. These are deterministic and cut the per-goal labour any agent loop spends.
- **Closest true peers — multi-agent proving frameworks.** [Prover Agent](https://arxiv.org/abs/2506.19923) (auxiliary-lemma generation), [MA-LoT](https://arxiv.org/html/2503.03205v2) (multi-agent Lean long chain-of-thought), and [APOLLO](https://arxiv.org/pdf/2505.05758) (automated LLM-Lean collaboration with proof repair) all formalize the propose→check→repair→critique cycle. They target competition math (miniF2F/PutnamBench); Lean Squad's contribution is pointing that cycle at real production code under a scheduler.
- **The software-not-math neighborhood — [Aeneas](https://github.com/AeneasVerif/aeneas) and code-to-Lean pipelines.** Aeneas ([arXiv 2206.07185](https://arxiv.org/abs/2206.07185), [lean-lang.org](https://lean-lang.org/use-cases/aeneas/)) verifies Rust by translating it to a pure functional Lean model — the principled correspondence route Lean Squad tried (Path A) and abandoned. This is the mirror image of Chronicle's plan: we go Lean→C ([#31](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md)), Aeneas goes Rust→Lean; both must answer *does the model match the real code?*

Reception is best read as the surrounding 2025–26 conversation rather than a single referendum: Martin Kleppmann's optimistic pole, *[Prediction: AI will make formal verification go mainstream](https://martin.kleppmann.com/2025/12/08/ai-formal-verification.html)* ([Simon Willison's coverage](https://simonwillison.net/2025/Dec/9/formal-verification/)), and the skeptical pole, Dougherty & von Hippel's *[Lies, Damned Lies, and Proofs: Formal Methods are not Slopless](https://www.lesswrong.com/posts/rhAPh3YzhPoBNpgHg/lies-damned-lies-and-proofs-formal-methods-are-not-slopless)*, which predates Lean Squad and independently names its failure modes (vacuous specs, unfaithful AI-rewritten models, "the AI rewrote the code to be correct then reported it passed"). Leonardo de Moura's *[Who Watches the Provers?](https://leodemoura.github.io/blog/2026-3-16-who-watches-the-provers/)* adds the other end of the trust stack — the kernel itself must be defended (he cites Claude Opus 4.6 finding seven kernel bugs in Rocq). We did not find a dedicated high-engagement thread on Lean Squad specifically; treat community reception as not yet established.

## Caveats & what may not transfer

The most useful skeptic is Syme himself; the article is heavily self-critical and concedes the limits that matter most for us.

1. **Trust concentrates entirely in the spec, not the proof.** *"Every definition and every statement of every theorem is AI written"* with *"no guarantee that these are 'good' definitions."* A green `lake build` proves the AI's model, not the real code. For Chronicle the analogous risk is a Lean model of `ValidateProducer` or `offsetGreater` that is subtly *weaker* than the Go — proving its theorems while diverging from what ships. This is the single least-transferable assumption and the reason our [DESIGN.md §5](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/DESIGN.md) calls model-vs-implementation the central risk.

2. **"Most theorems are glorified testing."** Syme's own words: the majority are directly obvious from the code. 1,200 theorems is a vanity metric; what matters is whether the *few* load-bearing theorems are non-vacuous and faithful. Applied to us: a wall of green over offset arithmetic could still miss the one ordering or overflow invariant that bites — precisely the unguarded `%016d` lexicographic-vs-numeric divergence our [ROADMAP P0.3](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md) already targets, and the `< 10^16` safe-domain hypothesis P1.1 demands be stated explicitly.

3. **Pseudo-completion.** Syme's automated Critique step was *too weak* and let the headline Raft theorem smuggle its hardest assumptions into its preconditions; only ~60–70% of a self-contained safety proof actually existed, the rest resting on 5 axioms. The human-judgment step the headline claims to eliminate — *is the spec meaningful? are the preconditions doing the work?* — is exactly the step that failed. Do not trust a "done" signal without an independent, adversarial spec/correspondence review.

4. **Path B is the prime venue for gaming.** Automatic extraction (Aeneas) failed, so the AI wrote the model by hand — which Syme warns risks the AI making up *"some garbage implementation that's easy to prove something about."* Concurrency, crypto, and network I/O are unmodelled in quiche. Chronicle's whole value is in the wake-path concurrency that pure-core modelling drops; that is why our concurrency story is TLA+/trace-validation in a separate track, not Lean.

5. **Training-data contamination.** Syme concedes Claude was *"almost certainly trained on"* Raft papers, QUIC RFCs, PX4 source, and Lean artifacts. The three demos are the well-trodden systems where an LLM is strongest. Chronicle is a bespoke Redis-8 Durable Streams server with no public proof corpus, so raft-rs/quiche/PX4 results likely **overstate** what a squad will do on our novel code. (The kernel still prevents false positives; contamination only inflates *throughput*, not *trust*.)

6. **Bug yield is low and orthogonal to the proof claims.** The genuinely valuable output was ordinary bugs (NaN sign, `Ord` antisymmetry, integer-negation overflow) a good fuzzer or property test would also catch — and zero implementation bugs in raft-rs. If the goal is finding bugs, our lighter `rapid` differential and Porcupine work dominates; the proof track's job is a durable correctness *argument*, where the spec-faithfulness problem above is the thing to solve.

7. **The kernel is a smaller risk than the spec, but not zero.** De Moura documents strong models now acting adversarially against kernels. We already pin `leanprover/lean4:v4.31.0` and keep the trusted base small — good — and we prefer plain `decide` over `native_decide` (which admits via an axiom and trusts the compiler), consistent with [02-proof-assistants-lean.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md).

8. **"Near-zero labour" is the *human* labour, not the cost.** $7/run is not free, and the work that was eliminated (proof debugging) is not the work that matters most (judging spec quality). For Chronicle the realistic pitch is far stronger than for Syme's targets: our cores are *already transcribed by hand and human-audited* against named Go source, so the faithfulness gap that breaks the near-zero claim on raft-rs is largely closed before any squad runs.

## What Chronicle adopts

Lean Squad is most useful to us not as a tool to install but as a **proof of method**: the scheduled propose→build→critique loop, with the kernel as the only trusted oracle, is real and cheap. We already run a multi-agent Orca squad and already hand-curate small total cores, so we adopt the *shape* and discard the parts that broke for Syme. Each item below is tied to an issue or doc.

1. **Stand up a phase-weighted proof-repair squad for [#30](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md) (P1.1).** Model the loop on Lean Squad and the multi-agent frameworks ([APOLLO](https://arxiv.org/pdf/2505.05758), [Prover Agent](https://arxiv.org/abs/2506.19923)): a Proof worker generates a tactic proof, `lake build` is the oracle, the agent reads compiler feedback and iterates, `sorry` count is the burn-down metric. Our scope is tiny and bounded — the theorems already enumerated in [02-proof-assistants-lean.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md) ("First theorems") against the three transcribed modules ([`Producer.lean`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go), [`Offset.lean`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go), [`OffsetGreater.lean`](https://github.com/adityavkk/chronicle/blob/main/webhook/state.go)) — so the loop should converge in far fewer runs than 34.

2. **Lead with hammer/automation tactics to cut hand-labour before LLM iteration.** For #30, call deterministic automation *first* and let the agent fall back to bespoke proofs only on the residue: `omega` for the `LastSeq + 1` and `ByteOffset + bytes` arithmetic, `decide` for the finite six-outcome producer case split, `exact?`/`apply?` for library lemmas, and once mathlib lands, `aesop` and the [LeanHammer](https://cmu-l3.github.io/lean-hammer/)/`duper` stack for the `LinearOrder` and strict-total-order goals (reusing mathlib `Prod.Lex` for `Offset.compare`). This is the lowest-effort on-ramp [02-proof-assistants-lean.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md) already names; Lean Squad's economics confirm that pushing routine goals onto tactics is what makes the per-run cost small.

3. **Make the Critique / Correspondence-Review task a first-class, adversarial gate — do not trust "done."** This is the direct lesson from Syme's pseudo-completion failure. For #30 add an explicit anti-vacuity pass: no theorem ships if its precondition does the work, and every theorem statement is reviewed against the named Go source the transcription already cites in its doc-comments. Mine the faithfulness literature for the gate's design — roundtrip verification and repair ([arXiv 2604.25031](https://arxiv.org/html/2604.25031)), gamed-formalization detection ([arXiv 2604.19459](https://arxiv.org/abs/2604.19459)), and the faithfulness-gap framing ([arXiv 2606.16541](https://arxiv.org/pdf/2606.16541)). Record this as a hardening of [DESIGN.md §5](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/DESIGN.md) and [ROADMAP risk register](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md). *(These 2026 arXiv items are medium-confidence — titles and URLs verified by search, PDFs not read line-by-line.)*

4. **The Lean→C differential oracle ([#31](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md), P1.2) is the mechanical answer to Lean Squad's biggest weakness.** Where Syme relies on a hand-written `CORRESPONDENCE.md` the AI could get wrong, our plan compiles the proven Lean model to C and runs it as a *third* oracle against Go and Lua, asserting agreement on result, error, and post-state across millions of generated inputs. That turns "trust the AI's correspondence prose" into "fail CI when the model and the code disagree." Concretely: borrow [FVSpec](https://arxiv.org/html/2606.01008)'s idea of treating real property-based tests as Lean obligations and feed our existing property contract into the oracle, so the model is pinned to behavior, not to itself. This de-risks the faithfulness gap that the near-zero-labour pitch silently depends on.

5. **Run the squad on a schedule with persistent memory and PRs — but keep the human owning the spec.** Adopt Lean Squad's scheduled `gh-aw`-style loop and persistent repo-memory as the cheap way to grind #30 and re-run after every Go change to the cores (a Go-side change to `ValidateProducer` should wake the squad to re-prove and re-run the #31 oracle). This realizes the "automation buys throughput, not trust" reading: the schedule and PR channel are pure upside; the spec/correspondence review stays a human-or-strong-critic responsibility, per caveat 3.

6. **Keep the trusted base small and pinned.** We already pin `leanprover/lean4:v4.31.0` and start mathlib-free; #30 adds `require mathlib` with a committed `lake-manifest.json` at exact revisions, and we prefer `decide` over `native_decide`. Adopt de Moura's separation-of-concerns framing as a standing rule for the track: complex AI-assisted *discovery* is fine, but the trusted checker stays small, pinned, and validated.

7. **Is the near-zero-labour pitch realistic for *our* cores? Partly — and more so than for Syme.** Our targets are small, total, and finite-branching; the proofs are `decide`/`omega`/induction one-liners; and crucially the transcription ([#29](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md), done) is already human-authored and audited against named source, so the spec-faithfulness step that broke Lean Squad's autonomy is the one step we *do not* hand to an agent. The honest framing for the team: a squad can near-automate the *proving* of #30 and the *running* of the #31 oracle; it cannot be trusted to author the spec. That is the right division of labour for a small, total, high-stakes core.

## Open questions

- **Which agent harness drives our squad?** Lean Squad defaults to Copilot CLI; we already run Claude Code inside an Orca multi-agent setup. Do we install `gh-aw` and the `lean-squad` workflow as-is, or reimplement the phase-weighted loop inside Orca to reuse our existing memory and dispatch? (Our [Orca orchestration notes](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/ROADMAP.md) suggest the latter is cheaper to operate.)
- **What is the anti-vacuity acceptance test, concretely?** Syme's critique pass was too weak; we need a checkable definition of "non-vacuous and faithful" for each #30 theorem before a squad can self-certify "done." Is the #31 differential oracle sufficient as that gate, or do we also need negation/mutation tests (does the theorem fail on a deliberately broken model)?
- **Has anyone validated Lean→C→cgo at fuzz-loop call volumes?** Welltyped did Lean→C→Rust; the Go/cgo variant for #31 still needs a spike to confirm overhead and memory behavior, as [02-proof-assistants-lean.md](https://github.com/adityavkk/chronicle/blob/adityavkk/formal-verification/docs/specs/formal-verification/research/02-proof-assistants-lean.md) flags.
- **How do we schedule re-proving against a moving Go source?** Lean Squad re-runs every 8 hours regardless; we want the squad to wake on changes to the three core Go files. What is the CI trigger, and how do we keep the vendored compiled-C oracle from going stale relative to the proven Lean?
- **Does autoformalization help bootstrap any new theorem statements, or is hand-transcription strictly better for us?** Given our cores are already transcribed, autoformalization's value is marginal now — but if the track later extends to the webhook reducers (`FenceDecision`, `MergeAcks`, `ClaimRotatesFence`, `DecideDue`), is an AI-drafted-then-human-audited statement faster than pure hand-authoring?
- **Is community reception of Lean Squad worth tracking before we commit?** We found no dedicated thread; the surrounding Kleppmann-optimist / "slopless"-skeptic debate brackets it. Worth a follow-up search once the live `raft-lean-squad` results (now claiming all axioms discharged) draw direct scrutiny.
