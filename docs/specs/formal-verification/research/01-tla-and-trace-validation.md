# TLA+, TLC/Apalache, and trace validation for the concurrent protocol

*TLA+ is the dominant industrial tool for specifying distributed logs and replicated state machines, and the decisive recent development — **trace validation** — makes it productive on an **already-built** system like Chronicle: write a medium-grained spec, instrument the running code to emit a trace at linearization points, and run TLC as a constrained model checker to confirm the real execution is a legal spec behavior. This note surveys the evidence, ranks the techniques by maturity and effort, and recommends modeling Chronicle's subscription wake/lease/generation-fencing + recovery-sweep protocol first — the [single-holder fence](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) the team has already articulated — then binding it to the Go/Lua code with trace validation.*

## Why this matters for Chronicle now

Chronicle is built. The pure-Go cores ([`store/producer.go`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go), [`store/offset.go`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go), [`protocol/cursor.go`](https://github.com/adityavkk/chronicle/blob/main/protocol/cursor.go)) and their Lua mirrors are already covered well by oracle + differential testing, and the property-based-testing worktree extends that further (see [`./03-property-and-model-based-testing.md`](./03-property-and-model-based-testing.md)). The open risk lives in the *concurrent* subscription engine: a `(generation, wake_id)` fence enforced by leases and a recovery sweep, with four hardened crash windows in [`docs/research/11-subscription-hardening-implemented.md`](https://github.com/adityavkk/chronicle/blob/main/docs/research/11-subscription-hardening-implemented.md). That subsystem is exactly the shape TLA+ was built for, and exactly the shape that defeats code review, unit tests, and even Porcupine histories (see [`../INVARIANTS.md`](../INVARIANTS.md) and [`./04-linearizability-checking.md`](./04-linearizability-checking.md)).

The headline question is whether it is "too late" to get value from TLA+ on a finished system. The evidence below says no — emphatically — provided you write the spec at the implementation's grain and bind it back to the code rather than letting it drift into abstract documentation.

### Finding 1 in detail — the trace-validation mechanism

Cirstea, Kuppe, Loillier & Merz (Univ. Lorraine / CNRS / Inria / LORIA + Microsoft Research) formalize validation as `S ∩ T ≠ ∅`: at least one spec behavior matches the trace. TLC reconstructs untraced variables non-deterministically and, per spec action, conjoins an `IsEvent(e)` predicate that increments a line counter, validates the logged event name, applies the trace's variable updates via `UpdateVariables`, then evaluates the original action. Success is decided by the postcondition `TraceAccepted` — the longest path equals the trace length. The authors ship a `TLATracer` Java API (`notifyChange`/`log`) and applied it to Two-Phase Commit, EWD998 termination detection, and Raft/etcd, finding discrepancies in all three. ([arXiv 2404.16075v2](https://arxiv.org/html/2404.16075v2))

### Finding 2 in detail — CCF's bug ledger

Howard, Kuppe, Ashton, Chamayou & Crooks (NSDI'25) report a clean split by technique: TLC model checking found 1 bug (election quorum tally); **trace validation found 4** (commit advance on AE-NACK, truncation from early AE, inaccurate AE-ACK, premature node retirement); simulation found 1. The consensus spec has 17 actions, 13 variables, 28 properties. They wrote a `Trace` spec reusing the high-level actions, constrained it to match logged events, switched TLC from BFS to DFS for "orders of magnitude faster" validation, and composed actions to handle grain-of-atomicity mismatches. They call the overall practice "smart casual verification" and integrated trace validation into CI. ([arXiv 2406.17455v1](https://arxiv.org/html/2406.17455v1))

### Finding 6 in detail — the MongoDB failure as a design guide

Davis, Hirschhorn & Schvimer ("eXtreme Modelling in Practice", PVLDB 13(9), 2020) abandoned model-based trace-checking after ~10 weeks: `RaftMongo.tla` modeled leader election as atomic while the C++ did it sequentially, and most effort went to snapshotting concurrent state, so marginal cost exceeded marginal benefit. The companion engineering blog states the successful modern recipe is to deliberately write a *new* spec at a level matching the implementation rather than conformance-check an old abstract one — and, separately, to use model-based test generation for algorithmic cores. ([arXiv 2006.00915](https://arxiv.org/abs/2006.00915), [MongoDB blog](https://www.mongodb.com/company/blog/engineering/conformance-checking-at-mongodb-testing-our-code-matches-our-tla-specs))

### Finding 9 in detail — Chronicle's fence is already spec-shaped

[`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) states the safety property as "at most one worker holds a token [`ack.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/ack.lua) will accept" — an invariant over a `(generation, wake_id)` register — and explicitly invokes Kleppmann's fencing token and the Jepsen etcd-lock double-grant bug. It already enumerates the phases (idle/waking/live) and the Lua reply statuses (`ARMED` / `CLAIMED` / `BUSY` / `OK` / `FENCED`) that would become the trace-event alphabet. The four hardened crash windows in [doc 11](https://github.com/adityavkk/chronicle/blob/main/docs/research/11-subscription-hardening-implemented.md) — fence rotation on expired-lease takeover, durable pull-wake recovery, glob-link reconcile, fan-out index repair — are precisely the non-atomic origin-restart interleavings TLC explores well.

## Techniques, maturity, and effort

| Technique | Maturity | Effort | Best Chronicle target |
|-----------|----------|--------|-----------------------|
| Trace validation (constrained model checking) | mature | medium | Bind the running Go/Lua subscription engine to the fence spec; drive from Jepsen + conformance runs |
| TLC explicit-state checking of the leases-and-fencing protocol | mature | medium | The wake/claim/ack/release + lease-TTL + generation-rotation protocol and the recovery sweep |
| Apalache symbolic checking + inductive-invariant proof | emerging | high | The single-holder fence property, as a size-independent guarantee — phase 3 only |
| Model-based test-case generation | mature | medium | The pure-core oracles + Lua mirrors (an exhaustive extension of the differential test) |
| Interactive spec exploration / animation (Spectacle) | emerging | low | Living documentation of the four crash windows + the rotate-vs-coalesce decision |

### Trace validation (constrained model checking) — *mature, medium effort*

Write a medium-level TLA+ spec of the protocol. Add lightweight logging that records state changes at linearization points (the moment a Lua script commits, a wake is armed, a claim/ack/release returns). A driver feeds the JSON trace into a `Trace.tla` wrapper that constrains the spec: each spec action gains an `IsEvent` predicate enabling it only when the next trace line matches; TLC reconstructs untraced variables non-deterministically and checks whether *some* spec behavior reproduces the whole trace. If it cannot, you have a real divergence between code and design. Run BFS for small checks, DFS for long traces.

This is the late-binding bridge to the running Go/Lua code and the single highest-value technique for an already-built Chronicle. The subscription engine already emits the exact discrete events trace validation needs (`ARMED` / `CLAIMED` / `BUSY` / `OK` / `FENCED` with `generation`/`wake_id`), so the trace alphabet is nearly defined. Validate traces from the Jepsen nemesis runs and the conformance suite against the generation-fence spec; a failure pinpoints any remaining gap among the four crash windows or a future change. This mirrors exactly what CCF did to a built C++ consensus implementation. ([arXiv 2404.16075v2](https://arxiv.org/html/2404.16075v2), [arXiv 2406.17455v1](https://arxiv.org/html/2406.17455v1))

### TLC explicit-state model checking — *mature, medium effort*

Model the protocol's state and legal transitions in TLA+/PlusCal, state the safety invariants (at most one holder of the current fence; an accepted ack carries the current generation; the cursor only advances forward), and let TLC exhaustively explore all interleavings of concurrent workers, crashes, and recovery up to a bounded instance. TLC returns a minimal counterexample when an invariant breaks — this is how AWS found the 35-step DynamoDB bug and how Kafka explores replication/fencing changes.

For Chronicle this directly models the wake/claim/ack/release + lease-TTL + generation-rotation protocol and the recovery sweep. TLC's exhaustive interleaving is the right tool for the four origin-restart windows in [doc 11](https://github.com/adityavkk/chronicle/blob/main/docs/research/11-subscription-hardening-implemented.md) — e.g., a crash between [`arm_wake.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/arm_wake.lua) and the wake-event emit recorded by [`record_wake_sent.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/record_wake_sent.lua); an expired-lease takeover via [`expire_lease.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/expire_lease.lua) racing a deposed worker's late [`ack.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/ack.lua). These are exactly the "failures interleaved with recovery" class that defeats review and testing. Expect it to confirm the single-holder invariant or surface a missed interleaving that the Porcupine model and conformance tests don't reach. ([CACM 2015](https://cacm.acm.org/research/how-amazon-web-services-uses-formal-methods/), [kafka-tlaplus KIP-966](https://github.com/Vanlightly/kafka-tlaplus/blob/main/kafka_data_replication/kraft/kip-966/kafka_replication_kip_966.tla))

### Apalache symbolic checking + inductive invariants — *emerging, high effort*

Instead of enumerating states, Apalache compiles TLA+ transitions to quantifier-free SMT for Z3/CVC5 and either bounded-model-checks deeper than TLC or proves an *inductive* invariant: show the invariant holds initially and is preserved by every action, which (unlike TLC) establishes it for *all* reachable states regardless of instance size. It requires type annotations on the spec.

This is a second-phase upgrade after the TLC spec stabilizes. The single-holder fence is a strong inductive-invariant candidate: proving "rotate ⇒ strictly-greater generation, coalesce ⇒ identical generation, accepted ack ⇒ current generation" as inductive gives a size-independent guarantee no bounded TLC run or Porcupine history can. The effort (type annotations, finding the inductive strengthening) means it should be done only for the fence core, not the whole engine. ([Apalache README](https://github.com/apalache-mc/apalache/blob/main/README.md), [OOPSLA'19](https://dl.acm.org/doi/10.1145/3360549))

### Model-based test-case generation — *mature, medium effort*

Use TLC to enumerate the spec's full behavior graph for a small instance, emit one concrete test case per behavior, run each against the implementation, and assert the implementation reaches the same outcome. MongoDB used this on the Realm Sync operational-transformation algorithm to get 100% branch coverage and find a real infinite-recursion bug — the same project where trace-checking the abstract replication spec had failed.

For Chronicle this is the best fit for the *pure-core* oracles ([`ValidateProducer`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go), [`Offset.Compare`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go), [`GenerateResponseCursor`](https://github.com/adityavkk/chronicle/blob/main/protocol/cursor.go)) and their Lua mirrors. Generate the full behavior set of a small `ValidateProducer` or `Offset.Compare` spec and run each case through both the Go core and the Redis Lua script — a formal, exhaustive extension of the existing Go-vs-Lua agreement check in [`store/redis/differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_test.go). It pairs naturally with the Lean 4 pure-core work in [`./02-proof-assistants-lean.md`](./02-proof-assistants-lean.md), and has lower payoff for the nondeterministic concurrent engine, where trace validation fits better. ([arXiv 2006.00915](https://arxiv.org/abs/2006.00915), [MongoDB blog](https://www.mongodb.com/company/blog/engineering/conformance-checking-at-mongodb-testing-our-code-matches-our-tla-specs))

### Interactive spec exploration / animation (Spectacle) — *emerging, low effort*

Spectacle ([will62794/spectacle](https://github.com/will62794/spectacle), formerly tla-web) runs the TLA+ spec in the browser, steps actions forward and backward, renders an SVG animation of the protocol state, and shares a specific trace via URL. It turns a spec from a verification artifact into onboarding and design-review documentation. Once the wake/lease/fence spec exists, an animated, URL-shareable walkthrough of each crash window (and the rotate-vs-coalesce decision) is a far better artifact than the doc 07/09/10/11 prose for onboarding and for the design doc this research feeds. ([will62794/spectacle](https://github.com/will62794/spectacle))

## Recommendation for Chronicle

Model the subscription wake/lease/generation-fencing + crash-recovery-sweep protocol **first**, in three phases, and bind it to the running code with trace validation. Do **not** start with the durable-stream store CRUD — the pure-core oracles plus the Lua differential test already cover that well, and Lean 4 + model-based generation are the better fits there (see [`./02-proof-assistants-lean.md`](./02-proof-assistants-lean.md)).

**Phase 1 — TLC spec at the implementation's grain.** Write a medium-level TLA+ spec whose state is one subscription's `{phase ∈ {idle, waking, live}, generation, wake_id, lease_deadline, cursor}` plus a small set of concurrent workers and an origin-restart action that can fire between any durable Redis write and its non-atomic Go follow-up. Encode the safety invariants the team already articulated in [`model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go):

- **single-holder** — at most one `(generation, wake_id)` that [`ack.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/ack.lua) will accept;
- **rotate ⇒ strictly-greater generation**, **coalesce ⇒ identical generation**;
- **accepted-ack ⇒ current generation**;
- **forward-only cursor**.

Let TLC exhaustively interleave the four hardened crash windows from [doc 11](https://github.com/adityavkk/chronicle/blob/main/docs/research/11-subscription-hardening-implemented.md). It will confirm the invariants or surface a missed interleaving. Keep the spec at the implementation's grain — the explicit lesson from MongoDB's failure and CCF's success — so it can be trace-validated rather than drifting into unbindable abstraction. Timebox a 1-week spike before committing the worktree: CCF's ~2 engineer-months was for a full 17-action consensus protocol, and Chronicle's single-subscription fence is smaller, so the spec is plausibly faster — but calibrate before promising.

**Phase 2 — trace validation (~1 engineer-day of logging, per the CCF figure).** Emit a JSON trace of the existing `ARMED` / `CLAIMED` / `BUSY` / `OK` / `FENCED` events with their `generation`/`wake_id` from [`webhook/manager.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/manager.go) and the Lua scripts, drive it from the Jepsen scenarios and conformance suite, and run TLC as a constrained checker (DFS) to prove the real traces are legal spec behaviors. Wire it into CI exactly as CCF did. Treat the spec as the living source of truth for the four crash windows and animate it in Spectacle for the design doc and onboarding.

**Phase 3 — Apalache inductive invariant (deferred).** Once the TLC spec is stable, escalate the single-holder fence to an Apalache inductive invariant to convert "no counterexample up to bound N" into a size-independent guarantee. This is the only step that closes the bounded-instance gap; do it for the fence core alone.

A natural follow-on target is the producer-idempotency / zombie-fencing protocol (`Producer-Id` / `Epoch` / `Seq`) in [`ValidateProducer`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go) — the other monotonic-fencing-token subsystem in Chronicle, which closely parallels Kafka's leader-epoch fencing, so the modeling vocabulary can be reused.

## Pitfalls

- **Spec too abstract / written after the code** is the documented failure mode. MongoDB abandoned trace-checking after ~10 weeks because `RaftMongo.tla` modeled leader election as atomic while the C++ did it sequentially. Write Chronicle's spec at the implementation's grain — phase transitions, generation rotation, the `arm_wake → record_wake_sent` split — so a real trace can match it.
- **Grain-of-atomicity mismatch.** A single spec action often maps to several implementation events (or vice versa); CCF had to compose actions (`UpdateTerm + HandleAppendEntries`) to validate. Chronicle's non-atomic [`arm_wake.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/arm_wake.lua) → [`record_wake_sent.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/record_wake_sent.lua) and Lua-then-Go index updates will need the same action-composition or finite-stuttering treatment.
- **Trace validation is not black-box and gives no guarantee on what you don't log.** An empty or under-instrumented trace validates trivially. You must log enough at the right linearization points (each Lua commit, each claim/ack/release outcome) or you get false confidence.
- **State-space explosion.** Validating with only variables (no event names) or with too many free variables timed out in the trace-validation paper (millions of states / timeouts). Log key event names *and* key arguments, and use DFS for long traces, as CCF did, to keep TLC tractable.
- **TLC checks bounded instances, not the general case.** A passing TLC run on 2–3 workers is not a proof. Either argue that a small-scope bound is sufficient or escalate the core single-holder property to an Apalache inductive invariant; do not oversell TLC results as "proven for all N."
- **Liveness is weakly covered.** Trace validation does not verify liveness, and TLA+ liveness checking is more expensive. Chronicle's "a stranded wake is eventually re-emitted by the sweep" is a liveness/recovery claim needing a temporal property and fairness assumptions, separate from the safety invariants — budget for it explicitly.
- **Very recent "unmodified-system" trace validation is unverified here.** OmniLink-style trace validation that needs no instrumentation is very recent (2026) and was summarized in this research from a PDF by a small model; treat its specifics as unverified. Prefer the proven instrument-at-linearization-points approach (Kuppe et al. / CCF) and confirm any OmniLink details against the primary source before relying on it.

## Open questions

- **Full AWS results table.** The per-system spec line counts and bug counts for S3, EBS, and the internal distributed lock manager could not be extracted (the canonical PDFs did not render as text in this run). The DynamoDB 939-lines / 3-bugs / 35-step figures were corroborated from multiple secondary renderings, but the full table should be confirmed against the [CACM HTML](https://cacm.acm.org/research/how-amazon-web-services-uses-formal-methods/) or the assets.amazon.science PDF.
- **Go ergonomics for the tracer.** Whether to port a Java-style `TLATracer` to Go or to emit traces from the existing Go logging/metrics plus a thin Lua hook needs a spike — the paper's instrumentation API is Java, and the exact linearization-point log placements in [`webhook/manager.go`](https://github.com/adityavkk/chronicle/blob/main/webhook/manager.go) and the Lua scripts must be chosen deliberately.
- **One Lua mutation = one atomic spec action?** Whether the Redis Lua scripts' per-stream serialization (single hash-tag slot) lets Chronicle treat each Lua mutation as one atomic spec action — simplifying the grain-of-atomicity mapping — is likely yes but should be confirmed against [`close.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/close.lua), [`append.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/append.lua), [`claim.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/claim.lua), and [`ack.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/ack.lua) semantics.
- **OmniLink applicability.** Whether OmniLink's unmodified-system approach (2026, arXiv ID unconfirmed) could validate Chronicle traces without added instrumentation, and whether it actually intercepts syscalls/network as summarized — needs the primary PDF read before counting on it.
- **Producer-fencing as a second TLA+ target.** Whether to spec the `Producer-Id` / `Epoch` / `Seq` idempotency/zombie-fencing protocol after the subscription fence — it is the other monotonic-fencing-token subsystem and parallels Kafka's leader-epoch fencing, so modeling could be reused.
- **Effort calibration.** CCF's ~2 engineer-months was for a 17-action consensus protocol; Chronicle's single-subscription fence is smaller, so the spec is plausibly faster — validate with the 1-week timeboxed spike before committing the worktree.
