# Event sourcing, exactly-once, fencing tokens, leases, CALM

*Chronicle's safety properties are not bespoke: three of its core invariants are textbook distributed-systems constructs with primary-source backing. The per-subscription monotone `generation` counter is a [Kleppmann fencing token](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html); the idempotent-producer state machine is a faithful re-implementation of [Kafka KIP-98](https://cwiki.apache.org/confluence/display/KAFKA/KIP-98+-+Exactly+Once+Delivery+and+Transactional+Messaging); and append-only/monotone state is the canonical "easy" domain of the [CALM theorem](https://arxiv.org/abs/1901.01930). This note maps each correspondence onto specific Chronicle files, grades the techniques by maturity and effort, and turns the mapping into checkable invariants for the formal-verification worktree.*

## Why this matters for Chronicle

Chronicle's subscription engine fences holders with a `(generation, wake_id)` pair plus leases and a recovery sweep; its producer path does Kafka-style Producer-Id/Epoch/Seq dedup; its streams are append-only with monotone offsets and once-set closure. Each of these maps onto a well-studied formalism whose threat model and proof obligations are already published. Anchoring the design doc on those named correspondences gives reviewers an external oracle to check Chronicle against, and tells us exactly where coordination is unavoidable versus where it is gratuitous. The deeper payoff is a clean separation that the code already half-encodes in [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go): **the lease TTL governs liveness, the monotone generation governs safety**, and clock skew can corrupt a TTL but never a monotone counter.

## Key findings

Each finding below is high-confidence and verified against the cited primary source.

### 1. The generation counter is a textbook fencing token

Kleppmann defines a fencing token as "simply a number that increases every time a client acquires the lock," and requires that the storage server "take an active role in checking tokens, and rejecting any writes on which the token has gone backwards" — the server "remembers that it has already processed a write with a higher token number (34), and so it rejects the request with token 33." ZooKeeper `zxid`/znode version and Chubby sequence numbers are named as real token sources.

Chronicle's [`webhook/scripts/ack.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/ack.lua) / [`webhook/scripts/common.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/common.lua) `fenced` predicate — mirrored in [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) `stepAckOrRelease` as `fenced = tokenGen != gen || reqGen != gen || reqWake != wake` — *is* this server-side rejection, and [`webhook/scripts/claim.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/claim.lua) rotates to a strictly-greater generation on every grant to a new holder, satisfying the monotone-increase requirement. Source: [How to do distributed locking — Martin Kleppmann](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html).

### 2. The fencing argument is empirically load-bearing (the etcd ~18% bug)

This is not academic hygiene. [Jepsen's etcd 3.4.3 analysis](https://jepsen.io/analyses/etcd-3.4.3) found that "if a client waited for another client to release a lock, lost its lease, and then the lock was released, the server would not re-check to make sure the lease was still valid before informing the client that they now held the lock" ([etcd issue #11456](https://github.com/etcd-io/etcd/issues/11456)). With "two-second lease TTLs, five concurrent processes, and process pauses every five seconds," Jepsen "could reliably induce the loss of ~18% of acknowledged updates." The documented remedy is to "use the revision of their lock key as a globally ordered fencing token."

[`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) names this directly in its header: "This is the etcd-lock bug class (Jepsen found etcd double-granted a lock and lost ~18% of acked updates), modeled directly." That bug class is precisely what the model is designed to catch.

### 3. The idempotent producer is KIP-98 line-for-line

[KIP-98](https://cwiki.apache.org/confluence/display/KAFKA/KIP-98+-+Exactly+Once+Delivery+and+Transactional+Messaging): "For a given PID, sequence numbers will start from zero and be monotonically increasing"; "the broker will reject a produce request if its sequence number is not exactly one greater than the last committed message" — a lower seq is a `DuplicateSequenceNumber` (ignored as a harmless retry), a higher seq is an `OutOfOrderSequenceException`.

[`store/producer.go`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go) `ValidateProducer` matches exactly: `seq == 0` for a new producer, `seq == state.LastSeq+1` accept, `seq <= state.LastSeq` returns `ProducerResultDuplicate` (no error), and `seq > lastSeq+1` returns `ErrProducerSeqGap` with `ExpectedSeq`/`ReceivedSeq`. See also [Confluent: Exactly-once Semantics is Possible](https://www.confluent.io/blog/exactly-once-semantics-are-possible-heres-how-apache-kafka-does-it/).

### 4. The producer epoch is itself a writer fence

KIP-98: a new producer instance with the same `TransactionalId` "bumps up the epoch of the PID, so that any previous zombie instance of the producer is fenced off and cannot move forward." [`store/producer.go`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go) `ValidateProducer` implements the same fence: `epoch < state.Epoch` returns `ErrStaleEpoch` (zombie fencing) with `CurrentEpoch`; `epoch > state.Epoch` resets `lastSeq` to 0 and requires `seq == 0` (else `ErrInvalidEpochSeq`) — the same epoch-monotonic fence and seq-reset contract. Source: [KIP-98](https://cwiki.apache.org/confluence/display/KAFKA/KIP-98+-+Exactly+Once+Delivery+and+Transactional+Messaging).

### 5. Append-only/monotone state is the "easy" domain of CALM

The [CALM theorem (Hellerstein & Alvaro, CACM 2020)](https://arxiv.org/abs/1901.01930): "the programs that have consistent, coordination-free distributed implementations are exactly the programs that can be expressed in monotonic logic." Monotonicity means "conclusions made on partial information continue to hold once we have the full information" — once true, future data cannot refute it.

Chronicle's append ([`store/offset.go`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go) `Compare` is a total order; offset only grows), acked-offset monotonicity, and closure-monotonicity (a stream once closed stays closed) are all monotone accumulations, hence coordination-free-consistent in the CALM sense. See the [morning paper summary](https://blog.acolyer.org/2019/03/06/keeping-calm-when-distributed-consistency-is-easy/).

### 6. CALM also tells us exactly where coordination is unavoidable

The same theorem identifies non-monotone operations as those needing coordination: "negation and universal quantification mess with monotonicity," and "deletions" make previously monotone properties (e.g. reachability/GC) non-monotone again — garbage-collection reachability is given as the canonical non-monotone example. Chronicle's GC-cascade is literally reachability-via-refcount, and close/delete are retractions of the "open/exists" fact.

The design already isolates exactly these in per-stream Lua scripts that run atomically within one hash-tag slot — [`store/redis/scripts/delete.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/delete.lua), [`store/redis/scripts/close.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/close.lua), [`store/redis/scripts/decr_ref.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/decr_ref.lua) — i.e. it pays the coordination cost *only* on the non-monotone operations, matching the CALM prescription. Sources: [CALM](https://arxiv.org/abs/1901.01930), [morning paper summary](https://blog.acolyer.org/2019/03/06/keeping-calm-when-distributed-consistency-is-easy/).

### 7. The lattice substrate explains why retried deliveries are safe

The Bloom/BloomL lattice work underpinning CALM gives the deeper mechanism. [Conway, Marczak, Alvaro, Hellerstein, Maier — "Logic and Lattices for Distributed Programming"](https://dsf.berkeley.edu/papers/UCB-lattice-tr.pdf): monotone programs are "deterministic regardless of execution order"; "join-semilattice merge (least upper bound) gives order-, duplicate-, and delay-insensitivity"; and once all replicas have all messages "their states converge to the same value by the lattice's mathematical structure — not through consensus or locking," with a syntactic monotonicity check. This is precisely why Chronicle's catch-up read / SSE delivery is safe to retry and dedupe: a reader's consumed-offset is a lattice that absorbs reordered or duplicated deliveries.

### 8. Leases are the right liveness formalism — and only that

[Gray & Cheriton (SOSP'89)](https://dl.acm.org/doi/10.1145/74851.74870): "A lease is a contract that gives its holder specified rights over property for a limited period of time"; on expiry the grantor acts unilaterally — "the server must delay writes to a file for which the failed client holds a lease until that lease has expired" — so "an unreachable client at most briefly delays write access." Non-Byzantine failures (crash, partition, slow holder) cost performance, not correctness, and the cost shrinks with shorter leases.

Chronicle's worker leases with TTL plus the crash-recovery sweep are exactly this construct. [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) encodes the necessary complement: "the fence, not the lease TTL, is the safety boundary, and clock skew can corrupt a TTL but never a monotonic generation." See the [morning paper summary](https://blog.acolyer.org/2014/10/31/leases-an-efficient-fault-tolerant-mechanism-for-distributed-file-cache-consistency/).

### 9. Event sourcing: state = fold(events), and the read side is its own exactly-once contract

Event sourcing reconstructs aggregate state by "performing a functional fold/reduce over the events," persisting state "as a sequence of state-changing events" on an append-only log, with CQRS materializing views from those events ([microservices.io](https://microservices.io/patterns/data/event-sourcing.html), [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)). Chronicle is exactly the append-only byte-stream substrate this pattern needs: ordered append (Stream-Seq lexicographic writer ordering, mediated by [`protocol/cursor.go`](https://github.com/adityavkk/chronicle/blob/main/protocol/cursor.go), plus monotone byte offsets) makes `fold(prefix)` deterministic, and forking gives copy-on-fork lineage.

The read-side dedup rule completes the picture. [Domain-Centric: Deduplication strategies](https://domaincentric.net/blog/event-sourcing-projection-patterns-deduplication-strategies): "process the event only if the stored sequence number is smaller than the number received"; when the "sequence number is strictly increasing" and processing is ordered, "projections become naturally idempotent under at-least-once delivery semantics," and storing only the offset (not every event id) avoids unbounded dedup-set growth. This is the read-side mirror of KIP-98 producer dedup, sharing [`store/offset.go`](https://github.com/adityavkk/chronicle/blob/main/store/offset.go) `Compare` as its total-order oracle — and it is why the webhook recovery sweep re-emitting stranded pull-wake wakes is safe.

## Recommendation for Chronicle

Anchor the design doc on three named, primary-source-backed correspondences, each rendered as a checkable invariant, then round it out with leases and the event-sourcing read-side contract.

1. **Generation = Kleppmann fencing token.** State in the spec that the monotone generation is the *sole* safety boundary and the lease TTL is liveness-only (clock skew corrupts a TTL but never a monotone counter). [`jepsen/checker/model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) already proves single-holder, so the doc's job is to cite [Kleppmann](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html) plus the [Jepsen etcd ~18%-lost-update result](https://jepsen.io/analyses/etcd-3.4.3) as the threat model, and to ensure a nemesis scenario pauses a holder past TTL and asserts its later ack FENCEs.

2. **Producer = KIP-98 idempotent producer, epoch as writer fence.** Cite [KIP-98](https://cwiki.apache.org/confluence/display/KAFKA/KIP-98+-+Exactly+Once+Delivery+and+Transactional+Messaging) as the normative reference for "exactly-once within `(producer, epoch)` per stream," and extend [`store/redis/differential_test.go`](https://github.com/adityavkk/chronicle/blob/main/store/redis/differential_test.go) so the Go oracle ([`store/producer.go`](https://github.com/adityavkk/chronicle/blob/main/store/producer.go)) and the Lua mirror ([`store/redis/scripts/append.lua`](https://github.com/adityavkk/chronicle/blob/main/store/redis/scripts/append.lua)) agree on the full KIP-98 error taxonomy.

3. **CALM monotonicity ledger.** Classify every protocol operation as monotone (append, offset/acked-offset advance, closure) or non-monotone (close, delete, GC-cascade on refcount-to-zero, epoch reset), citing [Hellerstein & Alvaro](https://arxiv.org/abs/1901.01930) to justify that monotone paths are coordination-free-consistent (hence horizontally scalable) while the per-stream single-slot Lua serialization is correctly reserved for the non-monotone operations.

Round it out with **leases** ([Gray & Cheriton](https://dl.acm.org/doi/10.1145/74851.74870)) for the liveness layer and the **event-sourcing `state = fold(events)` + monotone-position idempotent-projection** framing so the read/consumer side has an equally explicit exactly-once-effective contract. See sibling notes [./03-property-and-model-based-testing.md](./03-property-and-model-based-testing.md) and [./04-linearizability-checking.md](./04-linearizability-checking.md) for the testing machinery these invariants attach to.

## Pitfalls

- **Do not let the lease TTL leak into a safety argument.** Leases bound failure cost but cannot give mutual exclusion under clock skew or process pauses — that is the exact etcd failure. Safety must rest only on the monotone generation. State this explicitly; it is a common reviewer error.
- **Kafka's "exactly-once" is deliberately narrow.** Idempotent-producer dedup is exactly-once only within one `(producer, epoch)` for one stream; cross-stream atomicity needs transactions (a separate KIP-98 mechanism Chronicle does not implement). Claiming unqualified "exactly-once" overstates the guarantee — say "exactly-once within epoch, per stream."
- **CALM monotonicity is about logical retraction, not byte-append.** A `close` or `delete` retracts the "open/exists" fact and is non-monotone even though the byte log only grows. Mis-classifying these as monotone would wrongly conclude they need no coordination. The single-slot Lua serialization for close/delete/refcount is load-bearing and CALM-justified.
- **Idempotent projection requires BOTH ordered delivery AND a strictly-increasing position.** Chronicle gives ordered offsets, but if a consumer ever processes out of offset order (e.g. parallel SSE shards), the "skip if position ≤ checkpoint" rule alone is insufficient and can drop events. The contract must require ordered consumption per stream.
- **The recovery sweep is at-least-once by design.** Exactly-once-effective depends on the consumer deduping by its own monotone consumed-offset. If a consumer treats a re-emitted wake as a fresh event without offset dedup, it double-processes — the doc must make the consumer's dedup responsibility explicit.
- **[`model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) deliberately omits time and treats every FENCED as a legal no-op.** This means the safety model cannot catch a spurious-FENCED *liveness* defect (a live holder wrongly rejected). That class needs a separate liveness check — do not assume the linearizability oracle covers lost progress.

## Open questions

- Is there a published machine-checked (TLA+/Coq/Isabelle) proof of Kafka's idempotent-producer + epoch fencing *as a whole*? Jack Vanlightly's 2024 TLA+ diary series exists, but the entry retrieved this run models only `InitPid` (producer-id acquisition), not the seq/epoch dedup invariants — later entries should be checked before citing a complete formal result.
- Does a peer-reviewed formal (coalgebraic / operational-semantics) model of event sourcing's `state = fold(events)` with CQRS projection consistency exist as a primary source? This run found practitioner sources ([microservices.io](https://microservices.io/patterns/data/event-sourcing.html), [Azure](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing), [Domain-Centric](https://domaincentric.net/blog/event-sourcing-projection-patterns-deduplication-strategies)) but no verified academic formalization; worth a deeper academic search before the doc claims a formal event-store semantics.
- Could Chronicle's monotone invariants be expressed directly as join-semilattices (per the [Bloom/BloomL lattice work](https://dsf.berkeley.edu/papers/UCB-lattice-tr.pdf)) and discharged with a CRDT-style convergence proof, rather than (or alongside) Porcupine linearizability? That would give a coordination-free correctness argument for the read/append paths distinct from the fence's linearizability argument.
- What is the precise generation-rotation rule when a wake is armed but never claimed and the lease expires ([`expire_lease.lua`](https://github.com/adityavkk/chronicle/blob/main/webhook/scripts/expire_lease.lua) clears `wake_id` without rotating gen)? [`model_fence.go`](https://github.com/adityavkk/chronicle/blob/main/jepsen/checker/model_fence.go) notes the model can believe a token current after the server cleared it; the doc should pin down whether any sequence of expiries can let a very old token be re-accepted (likely no, because the next claim rotates) and back it with an explicit invariant and a targeted Jepsen scenario.
- For multi-stream consumers, does Chronicle want Kafka-transaction-style cross-stream atomic commit (read-process-write exactly-once across streams), which KIP-98 provides via transactions and the markers/coordinator, or is per-stream idempotency sufficient for the intended event-store use cases?
