# Chronicle invariant catalog

*The complete, de-duplicated inventory of safety and liveness invariants that the formal-verification and property-based-testing work targets. It merges the four code-extraction passes (pure cores, the Redis/Lua differential substrate, the subscription wake/lease/fence engine, and the existing Jepsen harness) into one authoritative reference, with each invariant linked to its enforcement point in source and tagged with the formal method(s) that can discharge it.*

This catalog is the shared vocabulary for the two hardening worktrees. Each invariant has a stable ID (`INV-<area>-NN`) used by the spec, theorem, and property artifacts to declare what they cover. The companion documents lay out the *how*: [./DESIGN.md](./DESIGN.md) for the overall stack rationale, [./research/03-property-and-model-based-testing.md](./research/03-property-and-model-based-testing.md) for the rapid/differential harness, [./research/02-proof-assistants-lean.md](./research/02-proof-assistants-lean.md) for the Lean pure-core proofs, [./research/01-tla-and-trace-validation.md](./research/01-tla-and-trace-validation.md) for the TLA+ subscription spec and trace bridge, and the [DESIGN.md](./DESIGN.md) / Chronicle protocol docs for the system under verification.

A reader after the headline result should read only the next section. A reader auditing coverage should read the [full inventory](#full-invariant-inventory) and the [coverage gaps](#coverage-gaps).

## Full invariant inventory

Every invariant extracted across the four code areas, merged and de-duplicated. Where one invariant surfaced in multiple passes (e.g. the producer-SM Go/Lua agreement appears as `INV-PROD-08` in the pure-core pass and `INV-DIFF-01` in the Redis pass; the fence/cursor/ownership invariants appear in both the subscription pass and the Jepsen pass), the rows are merged under a single canonical ID and the alternate IDs are listed as aliases so cross-references resolve. Each is **mutually exclusive** with the others â€” no statement is double-counted in the totals.

The `currently tested` column reflects the *existing* test suite, not the planned artifacts: `yes` = a dedicated assertion or live checker exists; `partial` = exercised only by example tables / indirect integration tests; `no` = asserted only in prose or untested. The `formal method(s)` column lists the planned discharge tools.

## Coverage gaps

What is currently unverified, asserted only in prose, or tested only by example. Grouped to mirror the inventory and ending with the scope items the first pass explicitly extends to (SSE/long-poll EOF, JSON-mode flattening + fork sub-offset, the HTTP status/410-Gone surface, and fork-chain recursion termination).

**Pure cores.**

- **Producer SM (INV-PROD-01..07)** is tested only by a 9-row example table plus the *same* table run differentially. No generative coverage of the `int64^3` domain; totality, determinism, and the result/error-pairing invariants are never asserted as universal properties.
- **Offset lexicographic-vs-numeric (INV-OFF-02)** is a confirmed latent bug: `%016d` is a *minimum* width, so at `ReadSeq` or `ByteOffset â‰¥ 10^16` string order diverges from numeric order, breaking Redis ZSET-keyed read ordering at large byte counts. Only tiny values are tested.
- **Offset order laws (INV-OFF-01)**, **round-trip (INV-OFF-03)**, and **Add monotonicity / silent uint64 overflow (INV-OFF-06)** are checked with a handful of hand-picked or single literals; the laws and the overflow wraparound are never asserted.
- **Sentinel avoidance (INV-OFF-05)**: no test asserts minted/parsed offsets cannot collide with `NowOffset`/`ZeroOffset` semantics.
- **Cursor (INV-CUR-01/02)**: the at/ahead monotonicity assertion is a weak string comparison gated on equal length; a real numeric regression could pass. `GenerateCursor` time-monotonicity and pre-epoch (negative interval) behavior are untested.
- **ContentTypeMatches / equalFold / ConfigMatches (INV-CT-01/02, INV-CFG-01)** have no direct unit tests in the named pure-core files; reflexivity/symmetry/transitivity, the deliberate ASCII-only divergence from `strings.EqualFold`, idempotent-create reflexivity, and the JSON-fork `ForkOffsetRequested`-vs-`ForkOffset` / `ForkSubOffset` nil-vs-0 branches are unverified.
- **No Go property-based-testing harness exists at all** for the pure core (no rapid/gopter/testing-quick/Go-fuzz target). Every pure-core invariant is example-tested at best.

**Redis differential.**

- `differential_test.go` covers **only** the 10-row producer table via `Append`. There is **no MemoryStore-vs-Redis model-based equivalence test over random operation sequences** â€” the single highest-value missing artifact for this area.
- **No differential coverage** of `norm_ct`/`ContentTypeMatches`/`normalizeCT` (INV-DIFF-04 / INV-CT-01): three independent normalizations with zero cross-check; multibyte/leading-semicolon/whitespace inputs untested.
- **No differential coverage** of `config_matches` vs `ConfigMatches` (INV-CFG-01), in particular the `forkOffReq`-fallback-to-`forkOff` branch and the nil-vs-0 sub-offset equivalence â€” classic drift points.
- **No frozen-clock differential** of `is_expired` vs `IsExpired` at the exact boundary nanosecond (INV-DIFF-06); existing tests use wall-clock sleeps that cannot pin the strict-inequality boundary.
- **Per-stream linearizability (INV-LIN-01/02)** is asserted only in prose. Porcupine is a dependency and already models the subscription/ownership layer but there is **no Porcupine model for store operations** (append/read/close/getOffset) against a concurrent client harness.
- **No property over the frame lex-order encoding (INV-FRAME-01)** with adversarial payloads (containing `|`, `0x00`, `0xff`, or the byte pattern of a larger offset prefix); only fixed examples are tested.
- **Latent ReadSeq divergence:** `MemoryStore.readOwnMessages` compares only `ByteOffset` while Redis compares the full 33-char offset prefix including `ReadSeq`. They agree only because `ReadSeq` is always 0 today (log rotation unimplemented); this will silently break when `ReadSeq` becomes non-zero. No test guards it.
- **`int64`-vs-double fidelity (INV-REPLY-01):** no test exercises producer epoch/seq near or beyond `2^53`; Lua `tonumber()` comparisons would silently lose precision and no upstream domain restriction is verified to be enforced.
- **Cross-slot non-atomicity of fork create + delete cascade (INV-FORK-01)** is documented as tolerated refcount-leak windows but never formalized â€” no spec states which interleavings/crash points are safe vs which corrupt the refcount invariant.
- **Backend asymmetry in expired-source soft-delete timing (INV-EXP-01):** MemoryStore flips `SoftDeleted` only inside `Read`; Redis flips it in any script hitting `is_expired` with `refCount > 0`. Not caught because no test calls `Append`/`Get` on an expired-but-forked source across both backends.
- **The full Append validation-precedence ladder (INV-DIFF-02)** is never driven end-to-end differentially â€” only individual rungs have integration tests, so a precedence reorder could pass CI.

**Subscription fencing and the Jepsen surface.**

- **INV-WAKE-01 (pending-work â‡’ eventually-wake)** is the prime liveness property and has **no temporal model**: the L1 checker only verifies the end-state after a workload, not that a wake is emitted under arbitrary crash/restart interleavings of the three wake paths. A crash that strands every path simultaneously is not enumerated.
- The full wake/lease/generation state machine is checked only **per-subscription and time-free** (`model_fence.go`). No model interleaves `expire_lease` (server step, no client op), the lease/retry/due workers, the recovery sweep, *and* concurrent claims/acks together â€” exactly the cross-transition interleavings where INV-FENCE-01 could break. This is the named TLA+ gap.
- **INV-RECOVER-02 (the post-emit T4 window)** is recently fixed and its `3*sweepInterval` staleness threshold is an unvalidated magic number: no proof it cannot re-emit to a still-live slow consumer, nor that 3Ã— guarantees eventual re-emit.
- **`model_shard.go` (INV-OWNER-01/02) still carries its SCAFFOLD header** even though `claim_shard.lua`/`check_owner.lua` have shipped; the live driver wiring must be confirmed before INV-OWNER-01/02 can be considered validated against the real Lua.
- The **"owner-epoch fence is an optimization, never a correctness dependency" claim (INV-OWNER-02 / INV-JEP-T3-02) is asserted, not proven**: no test runs the full system with every `owner_fenced`/`check_owner` forced to pass and shows the (gen,wake) fence alone still upholds INV-FENCE-01. No single checker models the **composition** of the two fence layers â€” a bug in the interaction would pass both isolated checkers.
- The interaction of the **two fence layers plus the third axis (claim granularity G shards)** is not jointly modeled.
- **INV-LR-01 idempotence-under-concurrent-idle** relies on `RestoreLease` being phase-conditioned, but the race window between the manager reading subs and calling `RestoreLease` is not modeled.
- **Tier B durability (INV-DUR-01)** is unit-tested only at the pure-function level; there is no end-to-end test/model of a real failover that drops un-WAITed tail and confirms the eager reconcile + fence self-heal (the RPO/self-heal claim of `Manager.Promote`).
- The **`sweepWindow` rolling-cursor batching** (`sweepBatch` cap) changes recovery latency to up to `ceil(K/sweepBatch)` ticks; there is no liveness proof that every subscription is still eventually swept (the wrap-around math is untested â€” an off-by-one could starve a subscription).
- **L1's three-lemma justification chain (INV-JEP-L1-02)** is asserted in a comment, never machine-checked together. **FENCED-as-no-op (INV-FENCE-04)** and **wake-id non-distinctness (INV-FENCE-02)** are load-bearing modeling decisions justified only by prose.
- **L2 / L4 / L5 (bounded recovery, ownership re-convergence, no starvation) are all PENDING-CLOUD** with only in-process trigger tests or unbuilt drivers. The distributed membership/HRW/slot-reconcile convergence â€” the heart of the horizontal-scale design â€” has no exhaustive consistency check; finite cloud runs sample it at best.
- All **liveness verdicts are asserted against a wall-clock bound, not against fairness**; the harness cannot distinguish "recovered just in time" from "would have recovered under fairness."
- **Three independent reimplementations** of the same logic exist (Go pure core, `common.lua`, Jepsen checker mirror: `offsetGreater`/`fenced`/`dsSlotOf`/`crc16`). The committed differential covers only the producer/offset core Go-vs-Lua; the webhook control-plane fence predicates have **no committed differential**, so a divergence among the three copies is invisible.

**SSE / long-poll EOF semantics.** *(First-pass scope, currently entirely outside the verification surface.)* There is no invariant or test covering when a streaming reader (SSE / long-poll) sees end-of-stream. The intended contract â€” a closed stream read up to its tail terminates the response (EOF), an open stream at tail blocks/long-polls per the cursor protocol, and a `done:true` boundary is conveyed exactly once â€” is unstated as a checkable property. The interaction with cursor monotonicity (INV-CUR-01) and at-least-once delivery (INV-JEP-L1-01) at the connection-EOF boundary is unmodeled: e.g. does a re-connect after EOF resume at exactly the post-tail offset, and does a stream closed mid-poll deliver the final frame before EOF rather than dropping it? Needs a stated EOF/termination invariant and a metamorphic property (reconnect-resumes-exactly, close-flushes-then-EOFs).

**JSON-mode flattening + fork sub-offset.** *(First-pass scope.)* JSON-mode append flattens a JSON array into per-element messages, folding `Add(len(msg))` over the elements (touching INV-OFF-06), and a fork into a JSON stream carries a `ForkSubOffset` selecting an element *within* a byte-offset message. The per-element monotonicity of the flattened append, the `ForkSubOffset` nil-vs-0 equivalence and the `ForkOffsetRequested`-vs-`ForkOffset` fallback (INV-CFG-01 / INV-DIFF-05), and the read-stitching at a sub-offset boundary (INV-READ-01, where the cap uses inclusive `LessThanOrEqual` â€” a known off-by-one risk) are all untested in the named files and are the highest-risk untested logic in the fork path. Needs a generative property over JSON appends + fork-sub-offset creation, cross-checked between backends, plus the `LessThanOrEqual` boundary-frame inclusion pinned.

**HTTP status / 410-Gone surface.** *(First-pass scope.)* The mapping from store/protocol outcomes to HTTP status codes is unverified as a property: the validation boundary is split across layers (e.g. `IsValidIntegerString` admits unbounded width but the handler's `strconv.ParseInt(...,64)` overflows â†’ 400, INV-PARSE-03), and the soft-deleted / expired / hard-reaped states must surface a consistent status (notably 410 Gone for a stream that existed and is now gone vs 404 for one that never existed, and the closed-stream / `ErrStreamClosed` surface). No invariant states the total function from `{NotFound, SoftDeleted, Expired-reaped, Closed, ContentTypeMismatch, SeqGap, StaleEpoch, ...}` to status codes, nor that producer-response headers (`Producer-Expected-Seq`/`Producer-Received-Seq`, INV-PROD-06) are echoed exactly. Needs a handler-level table/property pinning the status surface, including the 410-vs-404 distinction.

**Fork-chain recursion termination.** *(First-pass scope.)* Fork-chain reads recurse through ancestors (`readInherited`/`readForkChain`, recursively through *soft-deleted* ancestors; INV-READ-01, INV-EXP-01), and the refcount cascade (`deleteWithCascade`, INV-FORK-01) walks `forkedFrom` links. Neither has a stated termination invariant: nothing proves the fork lineage is acyclic (a cycle in `ForkedFrom` would non-terminate the read stitch or the delete cascade) nor bounds the recursion depth. Needs an invariant that the fork graph is a finite DAG (no cycles, bounded depth) and that read-stitch and delete-cascade terminate on every reachable store shape â€” an Alloy relational model (acyclicity of `ForkedFrom`) and a PBT generator over random fork trees that asserts termination and depth bounds.
